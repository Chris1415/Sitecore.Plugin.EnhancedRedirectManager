## 1. Architecture Overview

PRD-001 is a **feature-scoped delta** on the live PRD-000 Redirect Manager app. It does **not** change the architectural style, deployment model, security posture, or extension-point surface — those carry forward unchanged. It lifts the en-only constraint (ADR-0010, fulfilled-by-deferral) at every Sitecore-data boundary that was already parametric on `language`.

**The architectural change set is narrow:**

1. **One new SDK surface** — language enumeration via `xmc.sites.listLanguages` (tenant-wide) with site-scoped fallback derived directly from the existing `Sites.Site.languages: Array<string>` field on the already-fetched site object. There is no new HTTP call when site-scoped path applies.
2. **Three new Authoring GraphQL verbs on the existing `xmc.authoring.graphql` surface** — `addItemVersion`, `deleteItemVersion`, `updateItem` (extended to write `__Display name` with language scope). All ride the same SDK envelope contract verified in PRD-000.
3. **One new pure module** — language picker state hook backed by `localStorage` (ADR-0017 locks key shape + lifetime).
4. **One JSON schema break** — `redirect-manager/v1` → `redirect-manager/v2` (ADR-0015 — no back-compat, clean rejection of v1).
5. **One Context Panel reader** — `pageInfo.language` lifted from `pages.context` (already subscribed in PRD-000) to drive the Pages-aware banner copy. **No matcher changes** — that's PRD-002.

**What stays identical from PRD-000:**

- Mode A client-side iframe app (ADR-0002). No server-side code, no Auth0, no Mode B.
- Three extension-point routes (`/context-panel`, `/dashboard-widget`, `/full-page`) — ADR-0011 unchanged.
- `lib/sdk/*` boundary is the only place that touches the Authoring wire shape (architecture § 4.1 of PRD-000).
- DOUBLE `.data.data` unwrap on `xmc.authoring.graphql` mutations with `body` INSIDE `params` (verified 2026-05-11; PRD-000 production code is canonical).
- `sitecoreContextId` policy = `.preview` for every `xmc.*` call (architecture § 5.7 of PRD-000).
- Blok primitives + Nova preset + structural focus-visible guard.
- Vercel static deploy; no infra changes; no new CSP rules; no new secrets.

**What this delta enables:** every operator-visible feature of PRD-001 (US-1 through US-13) without a second data store, without a server, without changing the head-app contract, and without modifying any Sitecore template. The cost is concentrated in three places: the picker state module, the `addItemVersion` / `deleteItemVersion` GraphQL bodies, and the import/export schema v2 split.

**What this constrains:** PRD-001 carries forward every PRD-000 architectural constraint. The "no second data store" boundary (ADR-0003) still binds — picker state is per-browser via `localStorage`, not synchronized across devices (ADR-0017 accepts this). The "no server-side code" boundary (ADR-0002) still binds — copy-from create-version is two sequential client-side mutations with a client-side rollback (ADR-0016), not a server transaction.

**Critical SDK finding (§ 5 + § 11):** the PRD assumed verb names `createItemVersion` / `deleteItemVersion`. The Authoring schema actually exposes **`addItemVersion`** (not `createItemVersion`) and `deleteItemVersion` is confirmed by the `sitecore:sitecoreai-graphql-schemas` skill. **The architecture pins `addItemVersion` as the correct verb.** PRD-001 R-1 / OQ-1 close on this name change — the decision rule still applies if the real-tenant probe rejects the assumed input/payload shapes during /implement Tranche 1 capture pass.

**Create-version (copy-from) flow** — two sequential client-side mutations, client-side rollback (ADR-0016):

```
Operator: "Copy from de" on a map with no fr version
  → app calls addRedirectMapVersion(itemId, language: 'fr')
       → Authoring mutation: addItemVersion → returns new fr version
  → app reads source de version's UrlMapping (already in memory from prior read)
  → app calls updateRedirectMap(itemId, language: 'fr', fields: { UrlMapping: <de's mappings> })
       → Authoring mutation: updateItem(input: { itemId, language: 'fr', fields })
  → on success: toast "Copied de mappings to fr"; operator lands in edit flow
  → on populate failure: app calls deleteRedirectMapVersion(itemId, language: 'fr')  ← rollback
       → on rollback success: toast "Could not copy mappings; new version removed"
       → on rollback failure: toast "Partially populated version detected" + cleanup affordance
```

## 4. Data Model

### 4.1 Wire shape extension — `__Display name` aliased accessor

```typescript
// Extended WireItem shape in lib/sdk/redirects-read.ts
interface WireItem {
  itemId?: string;
  name?: string;
  template?: { name?: string } | null;
  IncludeVirtualFolder?: WireFieldValue;
  PreserveQueryString?: WireFieldValue;
  RedirectType?: WireFieldValue;
  PreserveLanguage?: WireFieldValue;
  UrlMapping?: WireFieldValue;
  __Updated?: WireFieldValue;
  Display_Name?: WireFieldValue;  // NEW — aliased from field(name: "__Display name") because
                                  // the leading-underscore identifier is invalid as a GraphQL alias
}
```

**GraphQL alias rule.** The Sitecore standard field is `__Display name`. GraphQL aliases cannot start with `__` (reserved for introspection types). The query aliases it as `Display_Name`:

```graphql
Display_Name: field(name: "__Display name") { value }
```

This matches the existing alias pattern in `redirects-read.ts` (`IncludeVirtualFolder: field(name: "IncludeVirtualFolder") { value }`). Verify the **read shape returns `value: string`** during /implement Tranche 1 capture pass — empty display names may return `value: ""` or `value: null` depending on tenant configuration; the decoder handles both as "no display name".

### 4.3 Field-versioning matrix — assumed shape (OQ-2)

Per PRD § 13 R-2, the matrix is a hard prerequisite. The architecture pins the **assumed shape** from Sitecore conventions; /implement Tranche 1 must capture the real-tenant matrix via `itemTemplate(where: { templateId })` + `templateField.versioning` introspection (Authoring schema § 1.5 in the `sitecoreai-graphql-schemas` skill).

| Field | Assumed versioning | Rationale |
|---|---|---|
| `UrlMapping` | **Versioned** | Translated URLs differ per language by definition. PRD-000 already treats this implicitly (every read passes `language`). |
| `RedirectType` | **Shared** | Redirect semantics (301/302) are language-independent. Same enum across all versions. |
| `IncludeVirtualFolder` | **Shared** | Virtual-folder behavior is a site-wide concern, not per-language. |
| `PreserveQueryString` | **Shared** | Query-string preservation is a redirect-rule property, not per-language. |
| `PreserveLanguage` | **Shared** | The flag literally encodes language behavior at redirect-time; per-language toggle would create contradictions. |
| `__Display name` | **Versioned** | Standard Sitecore — `__Display name` is the canonical per-language label. |
| Item name | **Shared** | Sitecore item names are always shared. Renames affect all language versions. |
| `__Updated` | **Versioned** | Sitecore system field — updated per version on every write. |

**Capture recipe (T01x at /implement):**

```graphql
query FieldVersioning($templateId: ID!) {
  itemTemplate(where: { templateId: $templateId }) {
    name
    fields {
      nodes {
        name
        versioning   # VERSIONED | UNVERSIONED | SHARED
      }
    }
  }
}
```

If the capture surprises (any shared field is actually versioned, or vice versa), Schema v2's `shared` vs `languages.<code>` block split adjusts inline per ADR-0015. FR-21 UI hint copy follows.

### 4.4 Picker state (client-only, `localStorage`)

Per ADR-0017 — verbatim:

```typescript
// localStorage key: redirect-manager.language.<tenantId>.<siteId>
type PickerState = {
  tenantId: string;
  siteId: string;
  language: string;        // ISO code, e.g. "en", "de", "fr-CH"
  lastUsedAt: string;      // ISO-8601, for future stale-entry cleanup
};

// Default resolution chain (lib/picker/language-picker-state.ts):
// 1. Persisted PickerState.language (when valid for current site)
// 2. site.language (Sites.Site.language — first-class default; see § 5.1)
// 3. "en"
// 4. site.languages[0] (first enumerated)
```

**Migration policy.** ADR-0017 documents the v1 key shape (`redirect-manager.language.<tenantId>.<siteId>`). Future versioning introduces `redirect-manager.language.v2.<...>`. PRD-001 ships v1; v1 keys are read-as-is, no migration needed because PRD-000 wrote nothing to `localStorage` (en-only had no picker).

```jsonc
          "mappings": [{ "source": "/old", "target": "/new" }]
        },
        "de": {
          "displayName": "SEO-Weiterleitungen",
          "lastUpdated": "2026-05-12T18:00:00Z",
          "mappings": [{ "source": "/alt", "target": "/neu" }]
        }
      }
    }
  ]
}
```

**v1 rejection prevalidator.** Before Zod parsing, a thin function reads `JSON.parse(...)?.schema` and matches `redirect-manager/v1` exactly — if matched, return the clear error string immediately. Avoids running the full v2 Zod schema and emitting confusing v1-shape errors. Any other unknown schema string passes through to Zod which rejects with its standard "Unsupported schema version" message.

## 8. Security Considerations

**No new attack surfaces in PRD-001.** Every PRD-000 mitigation (architecture § 8 of PRD-000) carries forward unchanged. The PRD-001 delta is reviewed below per attack class.

### 8.1 Auth model

Unchanged from ADR-0002. Mode A custom app; portal-brokered session; no tokens in app code. The new `xmc.sites.listLanguages` call and `addItemVersion` / `deleteItemVersion` mutations all ride the same operator-session bridge. PRD-001 does **not** require new API access scopes — Authoring + Sites scopes from PRD-000 cover both new surfaces.

### 8.2 New input — JSON import schema v2

The v2 schema is structurally similar to v1 (top-level metadata + array of maps + per-map fields). New input vectors:

- **Top-level `siteLanguages: string[]`** — bounded by string length per array entry; ISO language codes are short. **Mitigation:** Zod schema enforces `z.string().max(10)` per entry + `z.array(...).max(50)` overall (50 languages per site is well above any realistic value). No prototype-pollution surface (strings only).
- **Per-map `languages: Record<string, { displayName, mappings, ... }>`** — operator-supplied keys (language codes). **Mitigation:** Zod schema validates keys against `z.string().regex(/^[a-z]{2}(-[A-Z]{2,4})?$/)` (ISO 639-1 + optional regional). Rejects any key matching `__proto__` / `constructor` / `prototype` (prototype-pollution defense — Zod's `z.record(...)` enumerates known keys; we use `z.object(...).catchall(z.never())` if stricter). **Confirm Zod-version-specific protection at /implement.**
- **Per-language `displayName: string`** — operator-supplied free text. **Mitigation:** Zod enforces `z.string().max(255)`; React default escaping handles XSS at render time (same as PRD-000 NFR-Sec3). No `dangerouslySetInnerHTML` introduced.
- **Item cap** still 1000 items per file (NFR-Sec2 carry); applies pre-language-fanout (each map with N language versions still counts as 1 against the cap).

### 8.3 XSS from operator-supplied per-language strings

`__Display name` is operator-typed text rendered into the map list and detail header. **Same React-default-escaping mitigation as PRD-000.** No new vector; the field is rendered via Blok `text` primitives that auto-escape. Structural guard from PRD-000 (no `dangerouslySetInnerHTML` outside `components/ui/*`) extends to PRD-001 code via the existing grep-based build-fail rule.

### 8.4 New `localStorage` write

ADR-0017 records the key shape. **Threat analysis:**

- **Stored data:** language ISO code (`"en"`, `"de"`, etc.) + tenant ID + site ID + ISO timestamp. **No tokens, no auth material, no operator-identifying data beyond what's already in the operator's URL.** NFR-Sec1 (no tokens in storage) holds.
- **Cross-tab impact:** ADR-0017 accepts last-write-wins. No security implication — picker state is operator preference, not access control.
- **Malicious-localStorage scenario:** an attacker with write access to the operator's `localStorage` could set the picker to a language the operator doesn't expect, leading to "I edited the wrong language" confusion. **Mitigation:** the picker's value is always validated against the enumerated language set on read (ADR-0017 invalid-state recovery) — an unknown language code falls back to default-resolution-chain and surfaces a toast. The threat reduces to "operator gets a one-time toast"; not exploitable for data exfiltration or privilege escalation.

### 8.5 Iframe sandbox (NFR-C1)

Unchanged. No new `window.parent` / `document.cookie` / raw `postMessage` usage. Picker hook uses `window.localStorage` + `window.addEventListener('storage', ...)` — both are sandbox-permitted within the iframe's own origin.

### 8.6 CSP and iframe headers

Unchanged. No new external endpoints called; no new CSP `connect-src` entries required.

**Confirmation:** PRD-001 introduces zero new security review items beyond standard input-validation-on-the-new-Zod-schema. The change is genuinely additive on top of PRD-000's posture.

| **R-10** | Phase overrun into PRD-002 — Context Panel multilingual matching tightly couples to Full Page picker state. | Architectural boundary: **picker state is Full Page + Dashboard ONLY** (cross-surface within the operator's own UI). | **R-11** *(new)* | **`addItemVersion` schema variant** — Sitecore versions are per-language *and* numbered (multi-version-per-language). PRD § 5 + FR-1 carry. | **LOW.** |

### Tradeoffs explicitly accepted (PRD-001 delta)

- **`localStorage` over `sessionStorage` or a server-side store** (ADR-0017). Cross-tab last-write-wins is the cost; cross-session persistence is the benefit. Operator working from two machines maintains independent picker state — accepted, not a regression.
- **No multi-version-per-language UI in PRD-001.** Sitecore items can have multiple numbered versions per language. PRD-001 exposes "the current language version" as a unit. Multi-version exposure is a future PRD.
- **Hard break on import schema (`v1` → `v2`) with no migration.** ADR-0015 records. The operator population for PRD-000 is currently zero (confirmed); any v1 bundles obtained out-of-band must be re-exported. The cost of writing a v1-compat path was judged higher than the cost of the clean break.
- **Sequential (not transactional) copy-from create.** ADR-0016. Client-side rollback is best-effort; "partially populated" recovery requires operator action when rollback itself fails. Mode A boundary (ADR-0002) precludes server-side transactions; accepted.

### New ADR candidates (4 — under density gate)

1. **ADR-0019** — **`addItemVersion` is the canonical Authoring verb for new-language-version creation.** Documents the verb-name correction (PRD-001 PRD doc said `createItemVersion`; schema documents `addItemVersion`). Records the SDK envelope contract (`xmc.authoring.graphql` `client.mutate` + body INSIDE params + DOUBLE `.data.data` unwrap) for the new wrapper module `lib/sdk/redirects-version.ts`. Closes PRD-001 OQ-1's verb-name half. **Rationale:** the verb name is load-bearing across multiple downstream artifacts (FR-5, US-6, US-7, ADR-0016) and pinning it in a dedicated ADR prevents drift.

2. **ADR-0020** — **Language enumeration prefers site-scoped `Sites.Site.languages` over tenant-wide `xmc.sites.listLanguages`; tenant-wide is fallback only.** Documents the site-scoped-vs-tenant-wide decision with the fallback policy (when `Sites.Site.languages` is null or empty, fall through to `xmc.sites.listLanguages` and present tenant-wide list with UI hint). **Rationale:** PRD-001 § 5 FR-1 + R-3 + R-13 raise the question; the architecture closes it with a specific contract grounded in the `.d.ts` shape of `Sites.Site.languages`. ADR records the contract so future PRDs don't re-litigate.

3. **ADR-0021** — **Copy-from partial-failure rollback uses `deleteItemVersion` of the just-created version; if rollback fails, surface "Partially populated version detected" warning with operator-driven cleanup affordance.** Expands ADR-0016's "partial-failure recovery" paragraph into a dedicated decision record covering the exact recovery state machine: (a) addItemVersion ok + updateItem ok → success; (b) addItemVersion ok + updateItem fail → call deleteItemVersion(new version) → success → "Could not copy mappings; new version removed" toast; (c) addItemVersion ok + updateItem fail + deleteItemVersion fail → "Partially populated version detected" + manual cleanup link. **Rationale:** the state machine is non-trivial (3 outcome paths) and the failure-mode UX is operator-facing; pinning the contract avoids drift between the wrapper and the toast/modal copy. The Mode A boundary (no server-side transaction) makes this design necessary.

4. **ADR-0022** — **PRD-001 v1 `localStorage` picker-state key shape is the canonical v1; future-PRD migration uses a `redirect-manager.language.v2.<...>` namespace.** Records the explicit v1 key contract (`redirect-manager.language.<tenantId>.<siteId>`) and the migration policy: future shape changes write to a new namespace, read both keys during a transition window, prune v1 after a deprecation period. **Rationale:** ADR-0017 mentions key-version pinning in a one-paragraph aside; promoting it to a dedicated ADR makes the migration contract explicit so a future PRD that, e.g., adds `userId` scoping can be planned without re-deriving the migration rules.
