## 1. Architecture Overview

Redirect Manager is a **client-side iframe Marketplace app** (Mode A — `sitecore:marketplace-sdk-client` § 2 + ADR-0002). It runs entirely in the browser inside three Cloud Portal extension-point iframes, with **zero server-side code** and **no external infrastructure beyond Sitecore itself**. Every read and write rides the operator's authenticated Cloud Portal session via the typed `postMessage` bridge — `client.query(...)` for GETs and `client.mutate(...)` for POSTs.

The app exposes **three extension-point routes**, each with its own page module under `app/`:

| Extension point | Route URL | SDK enum (`AllowedExtensionPoints`) | Primary surface |
|---|---|---|---|
| Pages Context Panel | `/context-panel` | `xmc:pages:contextpanel` | Per-page redirect awareness (Pages editor sidebar) |
| Dashboard Block | `/dashboard-widget` | `xmc:dashboardblocks` | Per-site at-a-glance counters |
| Full Page (XM Cloud Full screen) | `/full-page` | `xmc:fullscreen` | Power-user CRUD + JSON import/export |

There is **no backend**. There is **no second data store**. Sitecore Authoring GraphQL is the single canonical source for redirect rules (ADR-0003); every component in the app reads from and writes to that endpoint exclusively, scoped to the operator's session via `sitecoreContextId`. The app is stateless across page loads — the only persisted state lives in Sitecore.

**Architectural style:** Next.js App Router SPA per extension-point route, hydrated client-side (`'use client'` everywhere the SDK is used). RSC for static shell only — no server-side data fetching. **No middleware**, no API routes, no `experimental_createXMCClient`. The deployment artifact is a static-rendered Next.js bundle hosted at any HTTPS-reachable URL (Vercel by default, but not required — ADR-0002 consequence).

**What this enables:** trivial install (just register the app in Cloud Portal — no infra, no secrets, no Auth0 application). Permission boundary collapses to "exactly the operator's Cloud Portal session" — no privilege escalation surface. CI/CD is a single `npm run build` + static deploy.

**What this constrains:** zero machine-to-machine flows. No scheduled jobs. No per-tenant background work. **All future PRDs must respect this boundary or supersede ADR-0002.** PRD-001's Upstash-backed analytics will need a separate design pass; PRD-002's sync-back is fundamentally an authoring-mutation operator UX, so it stays Mode A too.

## 2. System Components

The system is a **single Next.js application** with **three route modules**, **one shared SDK boundary**, **one shared parser**, and **one shared schema**. There is intentionally no service decomposition — at MVP scale, splitting introduces overhead without benefit.

### 2.2 Extension points

Three Cloud Portal extension points; one Next.js route per point. Identifiers and route URLs are locked here so Cloud Portal registration values match what the app actually serves.

| # | Extension point identifier | Route URL | Cloud Portal label | Iframe viewport (per `sitecore:blok-marketplace-integration`) |
|---|---|---|---|---|
| 1 | `xmc:pages:contextpanel` | `/context-panel` | "Page context panel — left-hand panel" | 320–400px wide × variable height (scrollable) |
| 2 | `xmc:dashboardblocks` | `/dashboard-widget` | "Dashboard widgets — SitecoreAI" | 300–800px × 200–400px |
| 3 | `xmc:fullscreen` | `/full-page` | "Full screen — SitecoreAI" | 1024–1920px × 600–1080px |

**Cloud Portal registration table** (paste verbatim into App Studio → Extension points):

```
Extension point          | Route URL
-------------------------|----------------------
Page context panel       | /context-panel
Dashboard widgets #1     | /dashboard-widget
Full screen              | /full-page
```

**Decision rationale:**

- **Why `xmc:fullscreen` (not `standalone`) for Full Page.** PRD-000 is a per-tenant tool, not an org-level umbrella. `standalone` would expose `application.context.resourceAccess[]` as multi-install fan-out (`sitecore:marketplace-sdk-xmc` § 2c) — irrelevant for a tenant-scoped CRUD app and adds a "pick which install" UX prompt that confuses operators. `xmc:fullscreen` binds to one tenant, which is exactly the model.
- **Why `xmc:dashboardblocks` (single widget) for Dashboard.** PRD-000 specifies one widget (3 stat tiles). The skill notes Dashboard widgets *can* expose multiple routes; we don't need that. One route, one widget — simplest.
- **Why root `/` returns 404 (not redirect).** The `MarketplaceProvider`-wraps-everything trap (`sitecore:marketplace-sdk-extension-routes` § 5c-bis) means a redirect at `/` would hang outside the iframe (the SDK handshake never resolves with no parent window). 404 is honest: the root route is not part of the app's surface. Cloud Portal binds by extension-point route URL, never by `/`. **This is a new architectural decision — captured as ADR candidate (§ 11).**

### 2.3 SDK boundary

There is exactly **one** SDK init (the `MarketplaceProvider` singleton). Every component reads from `useMarketplaceClient()` / `useAppContext()`. No component imports `@sitecore-marketplace-sdk/*` directly except the provider and the typed `lib/sdk/*` wrappers. This contract is enforced by a structural test (per the pattern in `last-edit-trail/site/__tests__/structural.test.ts`).

**Read flow (Context Panel example):**

```
Pages editor → iframe → Context Panel page mounts
  → useMarketplaceClient() (already inited in Provider)
  → client.query('pages.context', { subscribe: true })  // PagesContext stream
       └→ pageInfo.path = "/foo/bar", siteInfo.name = "MySite"
  → lib/sdk/sites → resolve siteName → siteRoot path
  → lib/sdk/redirects-read → Authoring GraphQL: list all RedirectMap children
       under /sitecore/content/{COLLECTION}/{MySite}/Settings/Redirects/
       (read fields: name, RedirectType, UrlMapping, flags, __Updated)
  → lib/url-mapping/parse → parse each item's UrlMapping into rows
  → lib/match/context-panel-matcher → filter rows where source==="/foo/bar" OR target==="/foo/bar"
  → render grouped by parent RedirectMap with Blok primitives
```

**Write flow (Full Page edit example):**

```
Operator edits a mapping row → form save
  → lib/url-mapping/serialize → re-encode row list to "%2ffoo=%2fbar&..." string
  → lib/sdk/redirects-write → Authoring GraphQL mutation:
       updateItem(itemId, fields: [{ name: "UrlMapping", value: "..." }])
       (mutation shape CAPTURE PENDING — § 5.4)
  → response: { data: { updateItem: { ...} } }   // single .data unwrap (mutate)
  → on success: emit Blok success toast, refresh list
  → on failure: friendly banner + expandable verbatim GraphQL error
```

## 4. Data Model

PRD-000 § 10 is the source of truth for the canonical Sitecore template fields and the `redirect-manager/v1` JSON schema; this section expands on the **runtime data flow** and the **shape contracts the engineering must respect**.

### 4.1 Two types live in the system

| Layer | Type | Source | Used by |
|---|---|---|---|
| Wire | Authoring GraphQL response — `{ item: { name, fields: [{ name, jsonValue }] } }` | Real-tenant capture (PRD § 9) | `lib/sdk/redirects-read.ts` |
| Domain | `RedirectMapItem` — `{ id: string, name: string, redirectType: RedirectType, preserveQueryString: boolean, preserveLanguage: boolean, includeVirtualFolder: boolean, updatedAt: string, mappings: Array<{ source: string, target: string }> }` | Decoded from wire shape | Everything above `lib/sdk/` |

**The `lib/sdk/redirects-read.ts` boundary is the only place that touches the wire shape.** Components, matchers, parsers, and the import/export schema all operate on the domain `RedirectMapItem`. This isolates the wire shape's awkwardness (fields-as-array, `jsonValue.value` envelope) from the rest of the codebase.

### 4.2 The `UrlMapping` round-trip

The `UrlMapping` string is the **only field with non-trivial parsing**. ADR-0008 locks the encoding contract:

- Parse: split on `&` → split each segment on the **first** `=` → `decodeURIComponent` source and target.
- Serialize: `encodeURIComponent` source and target → join with `=` → join all rows with `&`. Uppercase hex on serialize; case-insensitive on parse.
- Order: preserved on round-trip — operator-defined order is authoritative.

`lib/url-mapping/parse.ts` and `serialize.ts` are pure modules, property-tested for round-trip stability. **They have zero SDK dependency** — they can be tested without any client mocking.

### 4.3 GUID-keyed import diff

`lib/import-export/diff.ts` consumes:

- Incoming JSON items (parsed via `redirect-manager/v1` schema)
- Target-site domain `RedirectMapItem[]` (loaded via `lib/sdk/redirects-read.ts`)

Output: `Array<{ incoming: JsonItem, classification: 'new' | 'conflicting', existing?: RedirectMapItem, fieldDiff?: FieldDiff }>` — ADR-0009 keys on `id` (the Sitecore item GUID).

**The `createItem` GUID-control caveat (ADR-0009 consequences):** if the Authoring `createItem` mutation does not accept a caller-supplied `id`, "create" actions on cross-environment imports will mint a new GUID, breaking idempotent re-import. **This is a CAPTURE PENDING risk — § 10 OQ-9 / OQ-A.**

### 4.4 `RedirectType` enum

Real-tenant capture confirms `"ServerTransfer"`. The 301/302/307 enum names are TBV (PRD § 9 + OQ-8). The dropdown options must be **derived from real-tenant introspection** (the `__type(name: "RedirectType")` GraphQL query against the schema), not hard-coded — otherwise we ship a UI that lets operators select values the schema rejects, surfacing as a write error after the fact.

**Engineering recipe for OQ-8 resolution (CAPTURE PENDING):**

```graphql
query {
  __type(name: "RedirectType") {   # exact type name TBV — may be `Enum_RedirectType`, etc.
    enumValues { name }
  }
}
```

If introspection is locked down on Authoring (some tenants disable it), the fallback is a hand-validated list captured from the Sitecore Content Editor template definition, recorded as a constant in `lib/redirects/redirect-type-enum.ts` and revisited at PRD-001.

## 5. API Design

This section enumerates **every external SDK call** the app makes. Every call is annotated with:

- **Verb:** `client.query` vs `client.mutate` (queries vs mutations have different unwrap depths — `marketplace-sdk-client` § 8b).

- **Capture status:** captured (with fixture path) vs CAPTURE PENDING (with what the architect-stage real-tenant validation must produce).

### 5.7 `sitecoreContextId` policy — `.preview` for ALL calls in MVP

`xmc.*` calls require `sitecoreContextId` in `params.query`. **Policy:** every `xmc.*` call uses `resourceAccess[0].context.preview`. The single source of `sitecoreContextId` is a typed helper:

```typescript
// lib/sdk/require-context-id.ts
function requireContextId(appCtx: ApplicationContext): string {
  const id = appCtx?.resourceAccess?.[0]?.context?.preview;
  if (!id) {
    throw new Error('Sitecore context unavailable — app not bound to a resource');

  return id;
}
```

### 6.4 The `MarketplaceProvider` root-trap mitigation

Option (b) (extension-point-specific layouts) becomes worth it only when there is a meaningful difference in provider configuration per extension point — there isn't, in PRD-000. ## 8. Security Considerations

### 8.1 Auth model (ADR-0002)

Every Sitecore call rides the operator's authenticated Cloud Portal session via the typed postMessage bridge. **The app holds no tokens.** No `localStorage` or `sessionStorage` use for auth material (NFR-Sec1). The handshake is brokered by the parent window — if the operator's portal session expires, SDK calls return `TOKEN_ERROR` (`marketplace-sdk-client` § 8f) and the friendly error UX (FR-13) surfaces it.

**Permission boundary** = exactly the operator's session. No privilege escalation surface. The MVP target tenant is verified to grant Authoring write on `Settings/Redirects` items (PRD Q-CR-19); other tenants may differ — the friendly error UX surfaces 403s (PRD R2).

### 8.2 JSON import attack surface

**Threat:** malicious JSON file uploaded by an operator could attempt prototype pollution, deeply-nested DoS, or schema bypass.

**Mitigations:**
- **Schema validation first** (NFR-Sec2). `lib/import-export/schema.ts` is a Zod schema for `redirect-manager/v1`; the file is rejected before any preview is shown if the schema fails. Validation runs **before** any access to nested fields — no field access on untrusted input.
- **No `eval`-class APIs.** `JSON.parse` only.
- **Item count cap** at parse time: reject files with > 1000 items (10× the typical site cap from NFR-S1) with a clear error. Caps prototype-pollution-via-massive-array vectors and protects the operator from accidentally importing the wrong file.
- **Future-version rejection** with a clear error (PRD R9) — `redirect-manager/v2` import into a `v1`-only app fails fast.

### 8.3 XSS from operator-supplied strings

**Threat:** operator types a `<script>` tag as a Redirect Map item name or mapping target; the string is rendered into the UI.

**Mitigation:** all rendering uses React's default escaping (NFR-Sec3). The unsafe-HTML React prop is prohibited anywhere in the app; the project's structural test must grep for it (and the raw `innerHTML` DOM property) and fail the build on any hit. Verifiable via `grep -r 'dangerously' app/ components/ lib/` returning zero matches.

### 8.4 Iframe sandbox (NFR-C1)

Cloud Portal sets `sandbox` on the iframe; the app must respect it:

- **No `window.top.location` writes** (no parent navigation).
- **No `document.cookie` reads/writes** at top-level — auth cookies are scoped to the parent.
- All cross-frame communication via the SDK's typed channels — never raw `postMessage` to `window.parent`.

### 8.5 No secrets in the bundle

There is nothing to hide. PRD-000 has no `client_secret`, no API keys, no third-party tokens. Bundle inspection by an operator reveals only public source code. **No `NEXT_PUBLIC_*` env vars beyond what the scaffold ships** (e.g. `NEXT_PUBLIC_APP_URL`).

### 8.6 CSP and iframe headers

The static host (Vercel) needs:
- `Content-Security-Policy` allowing the parent origin (`portal.sitecorecloud.io`) to embed the iframe.
- Cookies — the app sets none, so SameSite/Secure rules don't bite at this layer.
- HTTPS with valid cert — mandatory per `marketplace-sdk-lifecycle` § 6.

The scaffold output ships sane defaults; verify at `/test` against the host-frame smoke harness.

### Tradeoffs explicitly accepted

- **One single Next.js app, not three.** A multi-app monorepo with one app per extension point would isolate bundles, but at MVP scale the bundle savings are negligible (`next/dynamic` for the heavy import preview gives most of the benefit) and three apps means three deploy URLs, three CI pipelines, three Cloud Portal app registrations. **Single app wins.**
- **No service-worker / offline support.** Operators are inside the Cloud Portal — they have connectivity by definition. Adding offline support would imply caching authoring data, which immediately conflicts with ADR-0003 (single canonical source).
- **No analytics fields in MVP export.** PRD-000 explicitly omits `UsageCount`. The export schema can grow additively in PRD-002 (`v2`); current decision is to keep `v1` flat and let the version bump signal the field addition.
