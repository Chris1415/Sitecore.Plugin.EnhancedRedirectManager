## 1. Executive Summary

**Redirect Manager** is a Sitecore Marketplace **client-side** app that gives content authors, site managers, and Sitecore implementers a purpose-built UI for redirect operations across an XM Cloud tenant. It replaces the awkward Content-Editor-based workflow for managing items under `/sitecore/content/{COLLECTION}/{SITE}/Settings/Redirects/*` and surfaces redirects in the contexts where editors work.

The app exposes **three Cloud Portal extension points**, all reading and writing a single canonical data source — Sitecore Authoring GraphQL — and styled with Blok:

- **Context Panel** (Pages editor) — for the page being edited, list redirects that affect it (where this page is the **source OR target** of a redirect by exact-string match), grouped by parent Redirect Map. Inline create / edit / delete.
- **Dashboard Widget** (per site) — at-a-glance count tiles: number of Redirect Map items, number of individual mappings, last-updated timestamp.
- **Full Page** — the power-user surface: pick a site collection, pick a site, browse all Redirect Map items, perform full CRUD on items and their mappings, import/export JSON.

PRD-000 ships the **MVP** — a self-contained Sitecore CRUD app with **no external dependencies beyond Sitecore itself**. The MVP operates **only in the default content language `en`**; multilingual / per-language CRUD is deferred to a follow-on PRD. Usage analytics (hit counts, broken-vs-healthy detection) are deferred to PRD-001. Sync-back of consolidated counters to a Sitecore template field is deferred further to PRD-002. Concurrent-edit detection is deferred to a later PRD.

The app is built with `sitecore:setup-marketplace-client-side` (Mode A only — ClientSDK + the `xmc` package, Next.js App Router, Vitest, Blok). It does not require server-side OAuth, custom Auth0 flows, or any backend the operator must provision.

## 2. Problem Statement

Redirect management today is painful for everyone who interacts with it:

- **Content authors and marketers** working in the Pages editor have **no in-context visibility** into redirects affecting the page they are editing. To check whether a page is being redirected away from, or whether other URLs redirect into it, they must leave Pages, open Content Editor, navigate to the site's `Settings/Redirects` folder, and read the raw item field values. Most do not, so misconfigured redirects ship to production.
- **Site managers** have **no per-site overview** of redirects on their site. They cannot answer "how many redirects do we have?" without manually counting items.
- **Sitecore implementers and admins** have **no portable mechanism** to move redirect sets between environments (dev → staging → prod). Today this means hand-replicating Redirect Map items or writing one-off PowerShell — error-prone and time-consuming.
- **The Content Editor UX for the Redirect Map item is rough** — a flat URL-encoded mapping string with no validation, no inline preview, and no batch operations.

The data exists. The editing capability exists. The routing infrastructure exists. What is missing is a **purpose-built surface** that meets each user where they work.

## 3. Goals and Success Metrics

### Goals

- **G1 — Eliminate the Pages → Content Editor jump for redirect awareness.** Marketers should see and resolve redirects affecting the current page from inside Pages.
- **G2 — Provide a per-site at-a-glance redirect overview.** Site managers should see redirect totals without leaving their dashboard.
- **G3 — Provide a single management surface for all redirect CRUD.** Admins should not need Content Editor for routine redirect work.
- **G4 — Enable portable redirect sets across environments.** JSON import/export should let admins move rule sets between sites or environments without scripting.

### Success Metrics (measurable acceptance for MVP shipped state)

- **m1 — Install:** All three extension points install via Cloud Portal Test App registration in approximately **5 minutes** on a real XM Cloud tenant. *(Approximate target — registration flow is unverified at PRD-000 authoring; will be confirmed during architecture.)*
- **m2 — Host-frame visual smoke:** All three extension routes pass the canonical 5-axis pixel comparison against the chosen UI variant's POC clickdummy (per `sitecore:marketplace-sdk-host-frame-testing`).
- **m3 — CRUD round-trip:** Create → edit → delete a Redirect Map item with at least three mappings via the Full Page UI on a real tenant — each operation returning in **<10 seconds** end-to-end (Authoring GraphQL round-trip plus UI feedback).
- **m4 — Import / export round-trip:** Export the redirect set from site A, import the resulting JSON into site B (different site or environment), with **zero data loss on rules** and conflict prompts firing correctly.
- **m5 — Live walkthrough:** A ≥**5-minute** editor-driven exploration on a real tenant produces **zero unrecoverable errors** (per `live_walkthrough` smoke definition in the run manifest).

*Note: analytics live-read metric (formerly m5) is deferred to PRD-001 along with all telemetry features.*

## 4. Target Users

**Primary — Content authors and marketers in the Pages editor.** The Context Panel is for them. They already have the page open; they need to see which redirects affect it and fix them inline without leaving Pages. They are not technical; they understand "if someone visits this URL, they'll be sent here instead", they don't read regex.

**Primary — Site managers and content owners.** The Dashboard Widget is their daily glance: how many redirects this site has, when they were last touched. They occasionally use the Full Page surface for spot-checks but do not own bulk operations.

**Primary — Sitecore implementers, dev leads, and admins.** The Full Page surface is their workshop: site-by-site management, import/export-driven environment portability. They are technical; they understand redirect rules and can read Authoring GraphQL errors when they bubble up.

**Not the target:**
- End-site visitors (the head application handles redirect *resolution* at request time; this app does not run in that path).
- Build tooling (this is a UI app, not a CLI / pipeline tool).

## 5. Product Scope

### In Scope (PRD-000 / MVP)

- **Three Marketplace extension points** — Context Panel, Dashboard Widget (per site), Full Page — registered against Cloud Portal extension points via `sitecore:marketplace-sdk-extension-routes`.
- **Site collection + site picker** in the Full Page, populated from `marketplace-sdk-xmc` sites SDK call.
- **Redirect Map item CRUD** via Authoring GraphQL — create, read, update, delete items under `/sitecore/content/{COLLECTION}/{SITE}/Settings/Redirects/*`, **operating on the `en` language version only**.
- **Mapping CRUD** within a Redirect Map item — add, edit, delete, reorder rows in the `UrlMapping` field. (The `UrlMapping` field is a single string with URL-encoded source=target pairs joined by `&`; the app parses, edits, and re-serializes this format. See § 10.)
- **Field editing on the Redirect Map item:** `RedirectType` (301 / 302 / 307 / Server Transfer — exact enum values verified at architecture), `PreserveQueryString`, `PreserveLanguage`, `IncludeVirtualFolder`.
- **JSON export** of a site's redirect rules — includes each Redirect Map's **Sitecore item GUID**, name, type, flags, and parsed mappings. No analytics fields included.
- **JSON import** with mandatory **per-item conflict review** — for every conflicting Redirect Map item (matched by **Sitecore item GUID**), the operator picks **create / overwrite / skip** via a per-item action picker with a collapsible diff preview. "Apply same action to all conflicts" is supported. Items with GUIDs not present in the target site are classified as "new" and default to "create". Per-item success/fail summary at end.
- **Context Panel listing** — exact source AND/OR exact target string equality only for MVP. UI banner explicitly surfaces "regex matches not yet covered — direct-string matches only". Listing is **grouped by parent Redirect Map**: each Redirect Map matched by at least one mapping shows its name, `RedirectType` badge, flag indicators, and the specific matched mapping rows beneath it.
- **Inline validation** on all writes (block save when source or target is empty; block save when `RedirectType` is unset).
- **Empty Redirect Map items allowed** — operators may create an item with zero mappings as a scaffolding step, then add mappings later.
- **Friendly error UX** — write failures show a friendly summary banner ("Couldn't save this Redirect Map") with an expandable "Show technical details" section containing the verbatim GraphQL error for advanced users.
- **Cancel/discard behavior** — when the operator closes or cancels an inline form with unsaved changes, a confirm prompt asks before discarding. When the form is pristine (no changes), close silently.
- **Blok-styled UI** across all three extension points, respecting Marketplace iframe viewport constraints. Dark and light theme as Blok defaults dictate.
- **Accessibility** — WCAG 2.1 AA, full keyboard operability, screen-reader friendly labels, focus-visible everywhere.
- **Modern evergreen browser support** — Chrome / Edge / Firefox / Safari (latest 2 versions each).
- **Vitest test stack** with typed mock client patterns from `marketplace-sdk-client`. RED-first per rule `30-tdd`.
- **Authoring GraphQL fixtures** sourced from real-tenant captures during the architecture phase (per `sitecore:sitecoreai-graphql-schemas` + rule `40-sdk-contracts`). Stakeholder validates produced GraphQL against a real tenant during architecture; if validation fails, the architecture iterates the mutation shape until it passes — no scope reduction on validation failure.

## 6. User Stories

### Context Panel (Pages editor)

- **US-1 (Marketer)** — *As a marketer in the Pages editor, I want to see all redirects that affect the current page, grouped by their parent Redirect Map, so I can understand the rule context, not just isolated rows.*
  - **AC:** When I open the Context Panel on a page at path `/foo/bar`, the panel lists every Redirect Map in the current site that contains at least one mapping where the **source** equals `/foo/bar` OR the **target** equals `/foo/bar`.
  - **AC:** Each Redirect Map is rendered as a group: header shows the map's name, `RedirectType` badge (301 / 302 / 307 / Server Transfer), and indicator chips for `PreserveQueryString`, `PreserveLanguage`, `IncludeVirtualFolder` (chips visible only when the flag is true).
  - **AC:** Beneath each map header, the matched mapping rows are listed (source → target).
  - **AC:** A persistent UI banner clearly states "Direct-string matches only — regex pattern matches are not yet covered".
  - **AC:** When no matching redirects exist, the panel shows a Blok empty state: "No redirects affect this page".

- **US-2 (Marketer)** — *As a marketer, I want to add a redirect for the current page directly from the Context Panel, so I do not have to leave Pages.*
  - **AC:** An "Add redirect for this page" affordance is visible. Clicking it opens a Blok modal sized for the Context Panel viewport.
  - **AC:** The modal contains a searchable list of existing Redirect Maps in the current site (showing name + last-updated). The first option in the list is "+ Create new Redirect Map".
  - **AC:** Picking an existing Redirect Map opens an inline form to add one mapping; the source is pre-populated to the current page path; target is required. `RedirectType` and flags are inherited from the chosen map (no Type prompt in this path).
  - **AC:** Picking "Create new Redirect Map" opens a fuller form: new map name, `RedirectType` (dropdown initially empty — operator must choose), flag checkboxes, plus the first mapping (source pre-populated, target required).
  - **AC:** Save fails with inline validation when source or target is empty, or when (in the create-new-map path) `RedirectType` is unset.
  - **AC:** Success closes the modal and refreshes the panel listing.

- **US-3 (Marketer)** — *As a marketer, I want to edit or delete a listed redirect row inline, so I can quickly fix mistakes.*
  - **AC:** Each listed mapping row has inline edit and delete actions.
  - **AC:** Edit opens a small inline form pre-filled with the current source/target; save persists via Authoring GraphQL.
  - **AC:** Delete asks for confirmation, then removes the mapping. If the parent map ends up with zero mappings as a result, the empty map remains intact (empty Redirect Maps are allowed and may be intentional scaffolding — see US-7).

### Dashboard Widget (per site)

- **US-4 (Site manager)** — *As a site manager, I want to see redirect totals for my site at a glance, so I can spot abnormal growth or shrinkage.*
  - **AC:** Tiles show: total Redirect Map items, total individual mappings (sum of parsed `UrlMapping` rows across all items), and the most recent `__Updated` timestamp across items in the site.
  - **AC:** When the site has zero Redirect Maps, the widget shows a Blok empty state: "No redirects configured for this site".
  - **AC:** Widget loads in ≤2 seconds (per NFR-P3).
  - **AC:** A footnote line clarifies: "Redirect counts only — usage analytics ship in a follow-on release."

### Full Page

- **US-5 (Admin)** — *As an admin, I want to switch between site collections and sites, so I can manage any site in the tenant from one surface.*
  - **AC:** A site-collection dropdown and a site dropdown, both populated from the SDK's sites call.
  - **AC:** When a collection is picked, the site dropdown shows only sites in that collection.
  - **AC:** When a site is picked, the redirect list pane loads its Redirect Maps.
  - **AC:** First-load default state: nothing selected, both panes show "Pick a collection to begin" Blok empty states.

- **US-6 (Admin)** — *As an admin, I want to browse all Redirect Maps for a site and their mappings, so I have full visibility.*
  - **AC:** Each Redirect Map item is shown as a collapsible row revealing its mappings on expand.
  - **AC:** For sites with up to 30 items / 500 mappings, the list renders in ≤2 seconds after site selection (NFR-P2). Light virtualization is acceptable.
  - **AC:** Per-item header shows: item name, `RedirectType` badge, mapping count (parsed from `UrlMapping`), last-updated timestamp.

- **US-7 (Admin)** — *As an admin, I want to create, edit, and delete Redirect Map items, so I can manage the redirect structure.*
  - **AC:** "Create new Redirect Map" button opens a form for: item name, `RedirectType` (dropdown 301 / 302 / 307 / Server Transfer — **initial state is empty; operator must explicitly choose**), `PreserveQueryString`, `PreserveLanguage`, `IncludeVirtualFolder`, and optionally one or more mappings.
  - **AC:** An item with zero mappings is allowed (empty draft).
  - **AC:** Edit opens the same form pre-filled with the existing values.
  - **AC:** Delete requires confirmation: "Delete `<item name>` and all its mappings? This cannot be undone."
  - **AC:** All operations succeed or fail in ≤3 seconds (p95) per NFR-P4. On success the list refreshes; on failure the user sees a friendly error banner with an expandable "Show technical details" section containing the verbatim GraphQL error and a retry button.

- **US-8 (Admin)** — *As an admin, I want to manage individual mappings (rows) within a Redirect Map item.*
  - **AC:** Within an expanded item, an inline editor shows the parsed mapping rows.
  - **AC:** Per-row actions: edit, delete, reorder (drag handle). Order is preserved on save (the `UrlMapping` field is re-serialized in operator-defined order).
  - **AC:** "Add mapping" button appends a new empty row; save fails with inline validation when source or target is empty.

- **US-9 (Admin)** — *As an admin, I want to export redirects from a site to JSON, so I can move them between environments or back them up.*
  - **AC:** "Export" button on the Full Page emits a JSON file download named `redirects-<site>-<timestamp>.json`.
  - **AC:** The JSON contains all Redirect Map items for the site (in the `en` language only). Each item record includes: **Sitecore item GUID**, item name, `RedirectType`, `PreserveQueryString`, `PreserveLanguage`, `IncludeVirtualFolder`, and the parsed list of source→target mappings in order.
  - **AC:** No analytics fields are included (analytics is environment-specific).
  - **AC:** Export of a typical site (≤30 items, ≤500 mappings) completes in ≤5 seconds.

- **US-10 (Admin)** — *As an admin, I want to import a JSON redirect file into a site, with explicit conflict handling, so I can promote rule sets between environments without surprises.*
  - **AC:** "Import" button on the Full Page accepts a `.json` file via browser file upload.
  - **AC:** App parses the JSON and validates the schema against `redirect-manager/v1`. Schema errors abort with a clear message before any preview is shown.
  - **AC:** App computes the per-item diff against the target site by **Sitecore item GUID**: each incoming item is classified as **new** (no item with this GUID exists in the target) or **conflicting** (an item with this GUID exists with at least one differing field).
  - **AC:** A preview screen lists all incoming items grouped by classification. New items default to "create"; conflicting items require an explicit per-item action (**create / overwrite / skip**) — no default action.
  - **AC:** Each conflicting item shows a collapsible diff between current and incoming values for `RedirectType`, flags, item name, and mappings.
  - **AC:** "Apply same action to all conflicts" bulk control is available.
  - **AC:** "Confirm import" is disabled until every conflict has an explicit action.
  - **AC:** On confirm, the app executes mutations per-item; partial failures are reported in the final summary (per-item success/fail). No transactional rollback across the batch.
  - **AC:** Import of 100 mappings end-to-end (parse + preview + review + apply) completes in ≤30 seconds (NFR-P5).

## 9. Technical Considerations

### SDK and stack

- **Scaffold:** `sitecore:setup-marketplace-client-side` (Mode A only). ClientSDK + the `xmc` package. Next.js App Router. Vitest. Blok.
- **No server-side OAuth proxy.** Every Sitecore call rides the operator's Cloud Portal session.
- **App Router defaults are fine.** RSC for static shell, CSR for interactive surfaces. No restriction on RSC. (Q-CR-22.)
- **Three extension routes** registered post-scaffold via `sitecore:marketplace-sdk-extension-routes` (one for Context Panel, one for Dashboard Widget, one for Full Page). Each route is a separate page module under `app/`.

### Authoring GraphQL contract — read shape (captured from real tenant)

A real-tenant query against the Preview endpoint (which the stakeholder confirmed has the same field shape as Authoring on reads) returns the following structure for a Redirect Map item:

**Query:**
```graphql
query {
  item(path: "{E39157F3-A81F-4692-B05D-178D48C836DE}", language: "en") {
    name
    fields { name jsonValue }
  }
}
```

**Response:**
```json
{
  "data": {
    "item": {
      "name": "My Redirect Map",
      "fields": [
        { "name": "IncludeVirtualFolder", "jsonValue": { "value": true } },
        { "name": "PreserveQueryString",  "jsonValue": { "value": false } },
        { "name": "RedirectType",         "jsonValue": { "value": "ServerTransfer" } },
        { "name": "PreserveLanguage",     "jsonValue": { "value": false } },
        { "name": "UrlMapping",           "jsonValue": { "value": "%2ftest=%2FnewTest&%2fhello=%2Fworld" } },
        { "name": "__Sortorder",          "jsonValue": { "value": "" } },
        { "name": "__Created",            "jsonValue": { "value": "2026-05-09T18:33:50Z" } },
        { "name": "__Updated",            "jsonValue": { "value": "2026-05-09T18:38:02Z" } },
        { "name": "__Semantics",          "jsonValue": [] }
      ]
    }
  }
}
```

**Key field shapes:**

- `IncludeVirtualFolder`, `PreserveQueryString`, `PreserveLanguage` — `jsonValue.value` is a boolean.
- `RedirectType` — `jsonValue.value` is a string enum. Confirmed value: `"ServerTransfer"`. Other values (the 301 / 302 / 307 names) are TBV at architecture (OQ-8).
- `UrlMapping` — `jsonValue.value` is a single URL-encoded string. Format: `<encoded-source>=<encoded-target>` pairs joined by `&`. Sources and targets are URL-encoded (`%2f` lowercase or `%2F` uppercase both observed for `/`). The example value `%2ftest=%2FnewTest&%2fhello=%2Fworld` decodes to `[{ "/test" → "/newTest" }, { "/hello" → "/world" }]`. **The app must round-trip parse/serialize this format losslessly.** See § 10 and ADR candidates.
- `__Updated` — ISO-8601 timestamp; suitable for the Dashboard "last-updated" tile.
- `__Sortorder`, `__Semantics` — present but unused by the app in MVP.

**Authoring GraphQL — write shape (TBV).** The stakeholder noted the Authoring endpoint write surface "may be slightly different" from the read shape. The architecture phase **must** verify the actual mutation verbs and argument shapes against the real tenant before the task breakdown is generated. Captured fixtures from real-tenant write attempts are mandatory before RED tests are written (rule `30-tdd` + `40-sdk-contracts`).

### `UrlMapping` parsing contract

- **Parse:** split on `&`; for each segment, split on the first `=`; URL-decode source and target; collect into an ordered list of `{ source, target }` objects.
- **Serialize:** for each `{ source, target }` row, URL-encode source and target separately (using the same encoding policy — confirmed at architecture, default to standard `encodeURIComponent`); join with `=`; join all rows with `&`.
- **Order:** the order of `&`-separated segments is preserved on round-trip. Operator-defined order from US-8 is the authoritative source.
- **Edge cases:** empty source or empty target rows are blocked at write time (FR-11). Existing items in the wild that contain malformed segments must surface a non-fatal warning and skip that segment on display rather than crashing.

This format is locked in **ADR-H** (see ADR candidates below).

### Sites SDK call

- **`xmc.sites.list`** (or its current-version equivalent — wrapper name verified at architecture stage from `node_modules/@sitecore-marketplace-sdk/xmc/*.d.ts`).
- Returns site collections and their sites for the current tenant.
- Two new Sitecore SDK surfaces in MVP: `xmc.authoring.graphql` and `xmc.sites`. Below the 3-surface split heuristic — no further phasing recommended on that basis.

### Context Panel — page-context source

- The app receives the current page context (including the page path and the site identifier) from the Marketplace SDK page-context channel. Architecture-stage deliverable: invoke `client.query()` with the page-context query, capture the actual response payload, identify the canonical fields for **page path** and **site identifier**, and document them in the architecture doc (Q-CR-10).
- "Include virtual folder" semantics on the Redirect Map item potentially affect path normalization. The app's matcher and the head-app's resolver must agree on normalization rules. Captured as Open Question OQ-3; resolution at architecture stage.

## 10. Data Model Overview

### Sitecore items (canonical, read/write)

**Redirect Map item template.** Children of `/sitecore/content/{COLLECTION}/{SITE}/Settings/Redirects/`. Fields used by this app — **field names captured from real tenant**:

| Field | Type | Notes |
|---|---|---|
| `RedirectType` | Droplist (string enum) | Confirmed value: `ServerTransfer`. 301/302/307 enum values verified at architecture (OQ-8). |
| `UrlMapping` | String | URL-encoded `source=target` pairs joined by `&`. App parses/serializes losslessly. See § 9 and ADR-H. |
| `PreserveQueryString` | Checkbox (boolean) | Shared field |
| `PreserveLanguage` | Checkbox (boolean) | Shared field |
| `IncludeVirtualFolder` | Checkbox (boolean) | Shared field; affects head-app path normalization |
| `__Updated` | Datetime (read-only) | ISO-8601; consumed by Dashboard "last-updated" tile and per-item header |
| `__Created` | Datetime (read-only) | Not surfaced in MVP UI |

**Item identity:** the Sitecore item GUID is the canonical identifier for cross-environment matching (ADR-I). Item name is human-readable but not the matching key.

The exact template ID is confirmed during architecture from real-tenant introspection (OQ-5).

**Sitecore template changes deferred:** the Redirect Map template change (`UsageCount`, `UsageLastSyncedAt`) is deferred to PRD-002.

### JSON import/export shape (`redirect-manager/v1`)

```json
{
  "schema": "redirect-manager/v1",
  "exported_at": "<ISO-8601>",
  "site": "<site-name>",
  "collection": "<collection-name>",
  "items": [
    {
      "id": "{E39157F3-A81F-4692-B05D-178D48C836DE}",
      "name": "My Redirect Map",
      "redirect_type": "ServerTransfer",
      "preserve_query_string": false,
      "preserve_language": false,
      "include_virtual_folder": true,
      "mappings": [
        { "source": "/test", "target": "/newTest" },
        { "source": "/hello", "target": "/world" }
      ]
    }
  ]
}
```

- `id` is the Sitecore item GUID; it is the **matching key** during import (ADR-I).
- `mappings` is an ordered array; order is preserved on import.
- No language array or per-language `versions` array (MVP is `en`-only).
- No analytics fields (analytics is environment-specific).
- The schema is versioned (`redirect-manager/v1`); the importer rejects unknown major versions with a clear error.

## 11. UX Considerations

### Three extension points, three viewport profiles

- **Context Panel:** narrow vertical strip in the Pages editor sidebar. Layout: vertical stack. Group-per-Redirect-Map list; "Add redirect for this page" affordance; modal-search for picking which map.
- **Dashboard Widget:** bounded card with a heading and 3 stat tiles. No interactive deep-dives — clicking a tile may link out to the Full Page.
- **Full Page:** full iframe width. Layout: two-pane (site/collection picker on the left, content pane on the right). On narrow widths, fall back to a tabbed layout.

### Information architecture

- Context Panel: list-first (grouped by parent Redirect Map); affordances secondary.
- Dashboard Widget: tiles-first; no list.
- Full Page: picker-first; the picker drives every other view.

### State coverage (must be designed)

For each extension point, designers must produce: **default, loading, empty, error, focus, success-toast** states. Empty states use Blok empty-state primitives with copy specific to the situation (no collections, no sites, no redirects, no analytics in MVP).

### UI variants at /architect

`scope_dial.ui_variants = 3`. The UI Designer agent produces three divergent variants per `sitecore:ui-designer` defaults — different typography, color systems, spacing scales, layouts. Each variant is a self-contained HTML POC clickdummy under `project-planning/ui-design/variants/`. Stakeholder picks one before task breakdown. The selected variant becomes the visual ground truth for the host-frame smoke at `/test`.

### Conflict-resolution UX (import)

- One row per incoming item, classified as **new** (no GUID match in target) or **conflicting** (GUID match with at least one differing field).
- New items: default action `create`, displayed with a small "+" icon.
- Conflicting items: action picker (radio) — `create` / `overwrite` / `skip`. No default.
- Each conflicting row has a "Show diff" toggle revealing field-by-field changes (name, `RedirectType`, flags, mappings).
- "Apply same action to all conflicts" lets the operator bulk-set.
- Confirm button is disabled until every conflict has an action.
- Final summary screen lists per-item outcomes (succeeded / failed with reason).

### Cancel / discard behavior

- Closing or cancelling an inline form with **unsaved changes** triggers a confirm prompt: "Discard your changes?"
- Closing a **pristine** form (no changes) closes silently.

### Error UX

- Write failure → friendly Blok error banner with short, human copy.
- "Show technical details" toggle reveals the verbatim GraphQL error for advanced users.
- Retry button repeats the last operation.

### Marketplace iframe constraints

- No top-level navigation.
- All cross-frame communication via the Marketplace SDK's typed channels.
- Theme tokens flow through the SDK; the app honors light/dark mode set by Cloud Portal.

## 13. Risks and Mitigations

- **R1 — Authoring GraphQL mutation shape unverified at PRD time.** The read shape is captured from a real tenant; the write/mutation shape is not. **Mitigation:** mandatory architecture-stage validation — stakeholder runs candidate mutations against a real tenant; iterate on the shape until it works. Fixtures are captured before tasks are written. If validation reveals a fundamentally different interface than expected, architecture iterates on the design rather than dropping scope (Q-CR-18).

- **R2 — Authoring write permissions varied tenants.** The MVP target environment is verified to grant Authors write on `Settings/Redirects` (Q-CR-19). Other tenants may differ. **Mitigation:** when the GraphQL response indicates a permission failure on a non-target tenant, the friendly error UX (FR-13) surfaces it clearly. Re-evaluate adding a runtime permission check in PRD-001+ if the Marketplace public-app submission requires it.

- **R3 — `UrlMapping` parse/serialize round-trip correctness.** The encoding format (`%2ftest=%2FnewTest&%2fhello=%2Fworld`) is mostly straightforward but subtle: case differences in URL-encoded characters (`%2f` vs `%2F`), edge cases (empty rows, malformed segments, `=` or `&` literal in source/target), and operator-defined order all matter. **Mitigation:** comprehensive Vitest coverage for parse → serialize → parse round-trip on real-tenant captured `UrlMapping` values, plus property-based tests for round-trip stability across permutations. Captured in ADR-H.

- **R4 — Bulk import correctness.** Per-item success/fail with no transactional rollback can leave the target site mid-state if some items succeed and others fail. **Mitigation:** detailed per-item summary with "retry failed items only" affordance. "Export before import" reminder in the import preview UI for cheap rollback. No silent partial-success masking.

- **R5 — Context Panel match accuracy.** MVP exact-match misses every regex-pattern redirect. Operators may distrust the panel if they assume regex coverage. **Mitigation:** the persistent UI banner is non-dismissible and visually prominent. The follow-on PRD with regex matching is captured as the highest-priority deferred work. README and `docs/architecture.md` document the limitation explicitly.

- **R6 — Virtual folder / path normalization mismatch.** If the marketplace app's matcher disagrees with the head-app's resolver on path normalization (especially "Include virtual folder" semantics), the Context Panel shows incorrect "affecting this page" lists. **Mitigation:** architecture defines the exact normalization in an ADR matching the head-app's resolver implementation; integration tests cover the canonical normalization cases.

- **R7 — `RedirectType` enum values incomplete.** Only `ServerTransfer` is observed. The 301/302/307 enum names need real-tenant verification. **Mitigation:** architecture-stage tenant introspection enumerates all valid values; the dropdown options are derived from the verified set, not hard-coded from PRD speculation.

- **R8 — Iframe viewport for Full Page.** The Full Page extension iframe width may not accommodate a comfortable two-pane layout. **Mitigation:** UI design phase tests against the actual extension iframe dimensions (Cloud Portal); fallback tabbed layout designed in parallel.

- **R9 — JSON import schema drift across releases.** Future shape changes might break older exports. **Mitigation:** the schema is versioned (`"schema": "redirect-manager/v1"`); the importer rejects unknown major versions with a clear error. Versioning policy documented in `docs/architecture.md`.

- **R10 — No concurrent-edit detection in MVP.** Two operators editing the same Redirect Map simultaneously can clobber each other's writes (last-writer-wins). **Mitigation:** explicitly out of scope for MVP per Q-CR-20. Documented as a known limitation in README and `docs/architecture.md`. Concurrent-edit detection is a future PRD if a real incident surfaces.
