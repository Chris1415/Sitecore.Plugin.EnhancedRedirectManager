## 1. Executive Summary

Wire the placeholder "Publish all" hero CTA on the Full Page workspace to a real Sitecore publish job, renaming it to **"Publish Site"**, and add a per-map **"Publish"** action in the Redirect Maps rail. Both actions hit the **SitecoreAI Publishing v1 REST API** (`POST /authoring/publishing/v1/jobs`) — operator-pre-approved as the integration target. Tranche 1 first probes `@sitecore-marketplace-sdk/xmc` for a native publishing wrapper; if absent (operator's initial inspection suggests it is), a new server-side Next.js API route holding OAuth client-credentials proxies the call. UX is fire-and-forget: confirmation dialog for the tenant-wide Publish Site action, no confirmation for per-map, toast feedback on both. No job-status polling, no progress UI — deferred.

This PRD is light-rigor scope: single tranche of implementation work with a Tranche 1 capture-and-decide step. It is the FIRST PRD in Redirect Manager that may introduce a server-side route (ADR-0002 explicitly anticipated this inflection).

## 2. Problem Statement

PRD-002 shipped the Workspace Hero with four hero CTAs — Refresh, View activity, Validate health, and "Publish all". The first two are wired to real behavior; the latter two are decorative placeholders that surface a toast. Operators who edit redirect maps in the Redirect Manager today must leave the app and use SitecoreAI's native publishing UI to ship those changes — context switch, multi-step navigation, and a separate permission boundary. Per-map publish is even harder: there is no per-item publish entry point in the Redirect Manager, forcing operators to locate the Redirect Map item in the SitecoreAI content tree and publish it from there.

The publishing surface itself is well-defined (SitecoreAI Publishing v1) and the operator has pre-confirmed it as the integration target. The work is wiring, not designing — keep scope tight.

## 3. Goals and Success Metrics

**Goal:** zero-leave operator publish flow for both site-wide and per-map shipping from the Redirect Manager Full Page.

**Success metrics:**
- M-P1 — From "Publish Site" click (post-confirmation) to "queued" toast: **≤ 3s P95** over a healthy network and a healthy SitecoreAI Publishing endpoint.
- M-P2 — From per-map "Publish" click to "queued" toast: **≤ 3s P95**.
- M-P3 — **Zero ambiguous failure states**: every non-2xx response from the Publishing API surfaces as a visible error toast with HTTP status code and the API's `detail` / `title` field.
- M-P4 — Real-tenant smoke `m_publish` passes on the solo tenant: both Publish Site and per-map Publish produce jobs visible in the SitecoreAI publishing job list with `source: "Redirect Manager"`.

## 4. Target Users

Same operator persona as PRD-000 / PRD-002 — the SitecoreAI **redirect content owner** who maintains URL redirect rules for one or more sites. They authenticate to Cloud Portal, open the Redirect Manager Marketplace app, edit Redirect Map items, and need to ship their changes. They have publish authority for the sites they manage; permission scoping is the operator-credential's responsibility (out of scope for this PRD).

## 5. Product Scope

### In Scope

- **Rename "Publish all" → "Publish Site"** on `site/components/full-page/WorkspaceHero.tsx` (line 212 in current main).
- **Wire "Publish Site"** to start a SitecoreAI publish job with: `options.xmc.site.mode = "Republish"` + the site's configured locales (all of them), `source: "Redirect Manager"`.
- **Confirmation dialog** before Publish Site fires: shows site display name, locale count, mode (Republish), source. Confirm / Cancel buttons. Blok styling consistent with existing PRD-002 dialogs.
- **Per-map "Publish" action** on every Redirect Map row in the rail. Single click, no confirmation, fire-and-forget.
- **Per-map publish payload**: `options.items = [{ id: <map-guid>, type: "Item", locale: "*" }]`, `options.xmc.items.mode = "Republish"`, `publishChildren: false`, `publishRelatedItems: false`. Item-only — no children, no related (per operator's explicit answer: "Only the Redirect Map Item. Nothing more needed").
- **Toast UX** for both actions: three states — requested (intermediate), queued (success, includes truncated job id), failed (error, includes status + detail). Re-uses the existing Blok toast / feedback component from PRD-002.
- **Tranche 1 capture step**: inspect `node_modules/@sitecore-marketplace-sdk/xmc/dist/*.d.ts` for any publishing-related surface (`publish`, `publishing`, `jobs`, etc.). Decide branch:
  - **Branch A — SDK wrapper found**: use ClientSDK Mode A like every other call today. ADR-0002 remains intact.
  - **Branch B — no SDK wrapper** (operator's expectation): introduce a new Next.js API route at `/api/publish` that holds OAuth client-credentials and proxies the REST call. New ADR will partially supersede ADR-0002.
- **Site locales resolution** at click time via the existing `xmc.sites` SDK surface (already used for site picking in PRD-000/002).
- **Error handling**: explicit toast for every non-2xx; no silent failures.
- **Auto-lite docs at ship**: 1-paragraph README delta + CHANGELOG entry.

### Out of Scope

- Job-status polling, progress card UI, in-app job history
- Cancel-mid-publish action (API supports `canCancel`, deferred)
- Bulk multi-select publish (publish N maps at once)
- Publish actions on the Context Panel or Dashboard Widget surfaces
- HealthBadge real probe + "Validate health" CTA wire-up (paired placeholder from PRD-002, separate PRD)
- Per-mapping publish (no separate mapping child items exist in the Sitecore data model — mappings live inside the SHARED `UrlMapping` field on the parent map item, so publishing the map item publishes the data)
- Mode toggle UI (Smart / Republish / Incremental) — Republish hardcoded per operator answer
- Schedule-publish-for-later
- Per-locale partial publish (e.g. publish only `de-DE`)
- Refactor of decorative "Validate health" CTA (assumption: it remains decorative until a separate PRD)
- Visual redesign of the Full Page hero (PRD-002 is canonical visual base; this PRD touches only the CTA label and adds wiring)

## 6. User Stories

### US-P1 — Site-wide publish from the Full Page workspace

**As** the operator,
**when** I have edited one or more Redirect Maps and need to ship those changes across the entire site,
**I want** to click a single "Publish Site" button in the workspace hero, confirm my intent, and have SitecoreAI start a full republish of the site,
**so that** I don't have to leave the Redirect Manager and navigate the SitecoreAI tree to publish manually.

**Acceptance criteria:**
- AC-P1.1 — The Full Page WorkspaceHero shows a CTA labeled exactly "**Publish Site**" (renamed from "Publish all" — the old label MUST NOT appear anywhere in the rendered DOM).
- AC-P1.2 — Clicking the CTA opens a modal dialog with title "Republish site" and body content stating: (a) the site display name, (b) the locale count being published, (c) the mode is "Republish (full)", (d) the source is "Redirect Manager".
- AC-P1.3 — The dialog has a primary button "Republish site" and a secondary button "Cancel". Cancel closes the dialog without action.
- AC-P1.4 — On confirmation, the app POSTs to `/authoring/publishing/v1/jobs` (or the SDK equivalent if Tranche 1 finds one) with body:
  ```json
  {
    "name": "Site publish — <site display name> — <ISO timestamp>",
    "source": "Redirect Manager",
    "description": "Triggered from Redirect Manager Full Page workspace hero",
    "options": {
      "xmc": {
        "site": { "mode": "Republish" },
        "locales": ["<each configured locale of the current site>"]
      }
    }
  }
  ```
- AC-P1.5 — On a 201 response, a success toast is shown: `"Site publish queued — job <id-short>"` where `<id-short>` is the first 8 characters of the returned `id`.
- AC-P1.6 — On any non-2xx response, an error toast is shown: `"Publish failed — <status>: <detail or title>"`.
- AC-P1.7 — Between click-confirm and response, the confirm button shows a loading / disabled state.
- AC-P1.8 — The dialog meets the same a11y standards as PRD-002 modals (focus trap, ESC closes, screen-reader-labeled).
- AC-P1.9 — The dialog and CTA render correctly in dark, light, and system themes (per global Redirect Manager theme policy).

### US-P2 — Single-map publish from the rail

**As** the operator,
**when** I have just edited a specific Redirect Map and want to ship only that map's changes,
**I want** to click a "Publish" action directly on that map's row in the rail and have SitecoreAI publish just that item,
**so that** I don't trigger a full site republish unnecessarily and don't have to locate the item in the SitecoreAI tree.

**Acceptance criteria:**
- AC-P2.1 — Every Redirect Map row in the rail (`RedirectMapList.tsx`) displays a "Publish" action. Final placement is /architect's call; pick from the three options enumerated in § 11 UX Considerations (icon button / overflow menu item / row-end action). Note: edit/delete live on the detail pane (`RedirectMapDetail.tsx`), not on rail rows today — there is no existing per-row action pattern to inherit.
- AC-P2.2 — Clicking "Publish" does NOT show a confirmation dialog (per operator UX choice).
- AC-P2.3 — The click triggers a POST to the publishing endpoint with body:
  ```json
  {
    "name": "Map publish — <map display name> — <ISO timestamp>",
    "source": "Redirect Manager",
    "description": "Triggered from Redirect Manager per-map publish action",
    "options": {
      "items": [{ "id": "<map-guid>", "type": "Item", "locale": "*" }],
      "xmc": {
        "items": {
          "mode": "Republish",
          "publishChildren": false,
          "publishRelatedItems": false
        }
      }
    }
  }
  ```
- AC-P2.4 — On 201: success toast `"<map name> publish queued — job <id-short>"`.
- AC-P2.5 — On non-2xx: error toast `"<map name> publish failed — <status>: <detail or title>"`.
- AC-P2.6 — The Publish action visually indicates a transient "publishing…" state between click and response (e.g. button spinner, icon swap).
- AC-P2.7 — The Publish action's color / contrast / hover state honors dark, light, and system themes.

## 7. Functional Requirements

- **FR-P1** — `WorkspaceHero.tsx` CTA rename: the existing label `"Publish all"` MUST be replaced with `"Publish Site"`. The accompanying `DecorativeCta` toast placeholder is removed and replaced with the real handler.
- **FR-P2** — A confirmation dialog component is added (or an existing Blok confirmation pattern is re-used) for the Publish Site flow. Dialog content per AC-P1.2 and AC-P1.3.
- **FR-P3** — A per-map "Publish" action is added to each Redirect Map row in the rail. Style and placement at /architect's recommendation.
- **FR-P4** — A publish-call service module encapsulates the body construction, the request, and the response-to-toast mapping. Both Publish Site and per-map Publish call into the same module with different parameters. (Goal: one place to maintain the contract.)
- **FR-P5** — Conditional on Tranche 1 outcome:
  - **Branch A (SDK wrapper found)**: the service module uses ClientSDK Mode A. No server-side route is added.
  - **Branch B (REST proxy fallback)**: a new Next.js API route at `app/api/publish/route.ts` accepts a typed request body, calls the SitecoreAI Publishing endpoint with the OAuth client-credentials token, and returns the response (or error) to the client. The client-side service module calls this route instead of SitecoreAI directly.
- **FR-P6** — Site locales for Publish Site are sourced from the existing `xmc.sites` SDK surface — specifically the `languages` array on the currently selected site. **Default body shape: enumerated locales** (per AC-P1.4). Only if Tranche 1 (D-T1.2) confirms the API also accepts `["*"]` should the body be simplified to that shorthand — otherwise enumerated wins. If `xmc.sites` does not enumerate locales for the site, fall back to a Tranche 1 capture step to resolve.
- **FR-P7** — The publish payload `source` field MUST be the string `"Redirect Manager"`. (Surfaces this app's jobs in SitecoreAI's publishing job list.)
- **FR-P8** — Toast feedback re-uses the existing Blok toast component. Three states: requested (transient ≤ ~1s), queued (sticky success), failed (sticky error). No new toast styles introduced.
- **FR-P9** — All errors surface as toasts; no silent failures, no console-only errors.
- **FR-P10** — All publish jobs include a `description` field in the body that identifies the trigger (`"Triggered from Redirect Manager Full Page workspace hero"` or `"Triggered from Redirect Manager per-map publish action"`).

## 8. Non-Functional Requirements

- **NFR-P1 — Latency**: click-to-toast ≤ 3s P95 over a healthy network and healthy SitecoreAI endpoint. (No client-side throttling; latency is dominated by the SitecoreAI API.)
- **NFR-P2 — Secret hygiene (Branch B only)**: if the server-side route lands, the OAuth client-credentials secret MUST be loaded from server-only environment variables and MUST NOT be exposed to the client bundle. The route response to the client contains only `{ jobId, status, ... safe fields }`.
- **NFR-P3 — Accessibility**: the confirmation dialog and per-map Publish action meet the same a11y standards as PRD-002 (keyboard navigation, ESC-to-close, focus management, screen-reader labels).
- **NFR-P4 — Reduced motion**: any dialog transitions or button-state animations honor `prefers-reduced-motion: reduce`. Pattern carries over from PRD-002 motion policy.
- **NFR-P5 — Theme parity**: dark / light / system themes render the new CTA, dialog, and per-map button with correct contrast (≥ AA on the relevant Blok tokens). Per the global Redirect Manager theme policy.
- **NFR-P6 — Idempotency** (best-effort): a double-click on the per-map Publish action SHOULD NOT submit two jobs in flight. The button is disabled while a request is in flight. (No server-side dedup; this is a client-side guard.)

## 9. Technical Considerations

### 9.1 Publishing API contract

- **Endpoint**: `POST /authoring/publishing/v1/jobs`
- **Auth**: Bearer (OAuth 2.0 client-credentials)
- **Required body fields**: `name`, `options`, `source` (per the operator-supplied API spec)
- **Site-publish body**: see AC-P1.4
- **Item-publish body**: see AC-P2.3
- **Success response**: 201 with `{ id, name, options, permissions, source, description, statistics, system }`. For PRD-003 fire-and-forget UX, only `{ id, system.status }` is consumed.
- **Error responses**: 400 ProblemDetails with `{ type, title, status, detail, instance }`. The toast surfaces `status` + `detail` (or `title` if `detail` is empty).

### 9.2 Auth scopes (per operator-supplied spec)

- **API-call scope** (one of): `xmcpub.jobs.a:w` (admin write) OR `xmcpub.jobs.t:w` (tenant write) OR `xmcpub.jobs.t:wl` (limited write, requires Sitecore Client Advanced Publishing / Sitecore Client Publishing role or admin flag).
- **Client application scope** (for Branch B server route): `xmcloud.cm:admin`.
- Setup steps documented per `sitecore:sitecoreai-auth` skill.

### 9.3 Architectural inflection — server-side route (Branch B)

If Tranche 1 confirms no SDK wrapper:
- **First server-side route in Redirect Manager.** PRD-000 / PRD-002 shipped as pure Client-Side Marketplace iframe (per ADR-0002). The new route runs inside the same Next.js app, server-side only.
- **CSP frame-ancestors unchanged.** The route is not part of the iframe surface — it's a server endpoint the client calls. No Cloud Portal registration change.
- **ADR-0002 partial supersede.** A new ADR records the addition of the server-route pattern and the rationale (publishing not exposed by the SDK). ADR-0002's "Harder" section explicitly anticipated this: *"If a future PRD needs to call a Sitecore REST API the SDK does not wrap, the OAuth proxy or a different scaffold must be revisited at that point."*
- **Auth wiring**: per `sitecore:sitecoreai-auth` — token endpoint, client_id / client_secret env vars, token caching strategy. The `sitecoreai-publishing` skill covers the publishing endpoint shape itself.

### 9.4 Environment variables (Branch B only — record in `.env.example`)

| Variable | Purpose |
|----------|---------|
| `SITECORE_OAUTH_CLIENT_ID` | Service-client ID with required publishing + admin scopes |
| `SITECORE_OAUTH_CLIENT_SECRET` | Corresponding secret (server-only) |
| `SITECORE_OAUTH_TOKEN_URL` | Per `sitecore:sitecoreai-auth` |
| `SITECORE_PUBLISHING_BASE_URL` | Base URL for the Publishing v1 API |
| `SITECORE_TENANT_ID` | Existing PRD-000 var if present; re-used. Required for tenant-scoped token / job routing. |

Document in `.env.example` only; never commit `.env.local`.

### 9.5 SDK-surface count

- 0–1 new SDK surfaces:
  - **If Branch A**: 1 new — `xmc.publishing.*` or similar SDK call.
  - **If Branch B**: 0 new SDK surfaces (REST-only). But adds a new architectural layer (Next.js API route + OAuth client).
- Existing surfaces re-used: `xmc.sites` (for locale resolution).
- Total new surfaces ≤ 1 → SDK-surface split heuristic does not trigger. Single PRD.

## 10. Data Model Overview

- **No Sitecore template / schema changes.**
- **No new fields on `Redirect Map` template.**
- **Client-side state**: transient — in-flight publish requests are tracked per-action to disable the button between click and response. No persisted state added.
- **Publish job response** (consumed slice): `{ id: string, system: { status: "Queued" | "Running" | "Completed" | "Failed" | "Canceled" | "Canceling" } }`. PRD-003 reads `id` for the toast and `system.status` only to confirm the job was accepted. It does NOT poll for state changes.

## 13. Risks and Mitigations

- **R-P1 — SDK has no publishing wrapper.** Likelihood: high (operator's initial inspection found none). Mitigation: pre-approved REST proxy fallback (Branch B). Tranche 1 confirms; no detour required if hit. Adds a server-side route — see R-P3.
- **R-P2 — OAuth client-credentials require operator-side admin setup** in Cloud Portal: register service client with required scopes + roles. Mitigation: document the credential creation steps in CHANGELOG + README at `/ship`. Reference `sitecore:sitecoreai-auth` for the protocol.
- **R-P3 — First server-side route in this app** (Branch B only) is an architectural inflection. Mitigation: route runs inside the same Next.js app (no infra change), CSP frame-ancestors unchanged, server-only env vars. New ADR documents the supersede of ADR-0002's "no server-side OAuth proxy" stance. Future PRDs touching publishing or other REST surfaces inherit this pattern cleanly.
- **R-P4 — `locales: ["*"]` may be rejected by the API.** Mitigation: Tranche 1 probes this against the live solo tenant. If rejected, enumerate locales from `xmc.sites.<site>.languages`. Either way, the user-facing behavior is unchanged (all configured locales get published).
- **R-P5 — Publish failures from permissions** (operator's credentials lack `xmcpub.jobs.*:w`). Mitigation: clear error toast surfacing HTTP status + detail. README documents required scope assignments. Not an app-layer problem to solve.
- **R-P6 — Fire-and-forget UX hides the actual outcome.** If the publish job ultimately fails after a successful 201, the operator never sees it from the Redirect Manager UI. Mitigation: explicitly out of scope per scope-dial; job-status polling deferred to future PRD. Operator is aware (this is the simple-PRD trade-off the operator chose).
- **R-P7 — Double-click submits two jobs.** Mitigation: button disabled between click and response (NFR-P6).
- **R-P8 — Operator misclicks "Publish Site" on the tenant-wide button.** Mitigation: confirmation dialog (US-P1) prevents accidental tenant-wide publishes. Per-map publish has no confirmation, but its blast radius is one item.
