## 🛑 Status: CANCELLED (2026-05-13)

**PRD-001 was cancelled during Tranche 1 of `/implement` — before any feature code was written.** Real-tenant probes against `CHAH DevEx Journey / PROD` revealed that the entire multilingual thesis rests on a Sitecore template configuration that does not hold.

### Cancellation rationale

The PRD-001 thesis (from /rubber-ducky D-Framing): *"Sitecore content-versioning model — Redirect Map items have per-language versions; mappings in `de` may differ from `en`."*

Tranche 1 capture probe **A4 (field-versioning matrix)** introspected the Redirect Map template's field definitions on the live tenant. Result:

| Field | Assumed | Actual |
|---|---|---|
| `UrlMapping` (the redirect rules) | VERSIONED | **SHARED** |
| `RedirectType` | SHARED | SHARED |
| `IncludeVirtualFolder` | SHARED | SHARED |
| `PreserveQueryString` | SHARED | SHARED |
| `PreserveLanguage` | SHARED | SHARED |
| `__Display name` | VERSIONED | UNVERSIONED |

**`UrlMapping` SHARED means the redirect rules are stored once per item, with no language axis.** Creating a `de-DE` language version of a Redirect Map item is mechanically possible (probe A1 confirmed `addItemVersion` works), but every language version of the item returns the *same* `UrlMapping` content. Probe A5 confirmed this empirically — `My Redirect Map 2` had both `en` and `de-DE` versions with byte-identical `UrlMapping` values.

`__Display name` is UNVERSIONED — different per language, but shared across versions within a language. The only field with a real per-language axis is the display *label*.

### What PRD-001 would have shipped if continued

A picker, a localized display-name editor, a "has-`de`-version" indicator dot — over 42 tasks. The picker would change the active language, the indicator would show "version exists," and the rules behind that version would be byte-identical to the next language over. **The product would be a lie:** operators would believe they were localizing redirects when Sitecore was storing one global value. PRD-001 was explicitly designed to avoid "collapsed UX over honest UX" (brain-dump D11); shipping cosmetic multilingual that misrepresents the data model breaks that principle.

### Why this is OK

Tranche 1 was designed exactly to catch this class of surprise before substantial implementation. From task breakdown T001:

> *"If the capture surprises (verb absent, field-matrix differs), the wrappers and downstream tasks adjust before Tranche 2 starts."*

PRD § 13 R-2 anticipated field-matrix surprise — though it assumed the surprise would be one-or-two fields differing, not the foundational field being SHARED. **A 30-minute real-tenant probe session caught a fundamental architectural assumption failure before any feature code shipped.** Cost: a few days of planning artifacts (PRD, architecture, 9 ADRs, 3 UI variants, 1 POC). Cost avoided: a 42-task implementation of a misleading feature.

### What's preserved as value

- **SDK learnings** (verb names, envelope shapes, response shapes for `addItemVersion`, `updateItem` per-language, `Item.versions(allLanguages: true)`) — captured in `docs/decisions.md` § *"PRD-001 Tranche 1 captures (real-tenant probes, 2026-05-13)"* and in memory as `reference_sitecore_redirect_map_template_field_versioning.md`.
- **The field-versioning matrix** — durable knowledge about the actual stock Redirect Map template; future multilingual PRDs reference this directly rather than re-probing.
- **The probe surface pattern** — proven in-app probe wizard that future T001-style discovery passes can reuse.
- **ADR-0023** (the cancellation decision) — documents *why* multilingual is deferred indefinitely until a Sitecore-side template change is on the table.

### What's deferred

The full multilingual story (real per-language redirect rules) requires a coordinated change with no clean phase split:

- Sitecore Authoring: modify the Redirect Map template so `UrlMapping` becomes VERSIONED (or add a new field `UrlMappingLocalized` that is, and migrate). Authoring-tool work, operator-supervised.
- Head-app resolver: read per-language `UrlMapping` content at runtime. External deliverable.
- App UI: language picker, per-language CRUD, version mechanics. The work PRD-001 was originally going to do — but on top of a template that actually supports it.

When the operator chooses to take on the template change (a substantial Sitecore-side commitment), a future PRD will bundle all three changes together.

- **Analytics** — original PRD-003 in the phasing-reframed plan; bigger scope (Upstash + head-app instrumentation + Dashboard enrichment) but high signal value.

## 1. Executive Summary

Redirect Manager PRD-001 adds **multilingual CRUD** to the Sitecore Marketplace app shipped in PRD-000. Operators can now author, edit, and manage redirect mappings in any language version of a Redirect Map item — not just `en`. Scope is **app-internal**: Full Page (language picker, per-language CRUD, create-version flow, per-language display name editing), Dashboard Widget (language-scoped tiles), and import/export (multi-language JSON bundle, schema `redirect-manager/v2`). The Context Panel keeps its current `en`-only matching behavior with a Pages-aware banner update; full Context Panel multilingual matching requires a parallel head-app resolver contract and is deferred to **PRD-002**.

PRD-001 lifts the en-only constraint deferred in ADR-0010 and reframes ADR-0004's three-PRD phasing into a four-PRD plan (ADR-0014; ADR-0004 marked Superseded). One new SDK surface family is introduced (`xmc.sites.languages`-style language enumeration); the Authoring GraphQL family from PRD-000 is extended with `createItemVersion` / `deleteItemVersion` verbs (assumed shapes — verify at `/architect`). If `deleteItemVersion` is absent from the Authoring schema, delete-version-only is **cut from PRD-001** (whole-item delete remains); no fallback that silently corrupts the version model.

## 2. Problem Statement

PRD-000 shipped en-only by design — a deliberate MVP boundary (ADR-0010) that let the app install with zero external dependencies and a single SDK surface family. The language seam was intentionally preserved: every Authoring read and write is already parametric on `language` (literal `"en"` today).

Today's app is honest about the limitation via a non-dismissible banner, but Sitecore is intrinsically multilingual. Virtually every Sitecore customer has multiple languages defined (operator-confirmed during discovery: *"every customer literally has that potentially"*). Without multilingual CRUD, the app is unusable for the majority of real Sitecore tenants — even single-language tenants' site collections may contain other sites in other languages and operators have no way to manage their redirects from this app. The en-only banner is acceptable as an MVP signal; it becomes a credibility problem as the product matures.

PRD-001 closes that gap for the app-internal surface (Full Page + Dashboard + import/export). PRD-002 will close it for Context Panel matching, which requires coordinated head-app changes outside this app's deployment scope.

## 3. Goals and Success Metrics

### Goals

1. Operators can switch the Full Page surface between any language version supported by the current site (with tenant-wide fallback) and author redirects in that language.
2. Operators can create a new language version for an existing Redirect Map — either empty or pre-filled from another language's mappings — without leaving the app.
3. Operators can edit the per-language `__Display name` of a Redirect Map; the map list shows the current-language display name with a fallback to item name.
4. Operators can safely delete either a single language version OR the entire Redirect Map across all languages, with explicit confirmation distinguishing the two.
5. Operators can export a site's entire redirect inventory (all language versions) to a single JSON file, and re-import it to reconstitute all language versions.
6. Operators viewing a non-`en` page in Pages see a transparent banner in Context Panel explaining that multilingual matching ships in a follow-on PRD.

### Success Metrics (smoke gates)

| ID | Criterion | Gate category |
|----|-----------|---------------|
| **m1** | Language picker enumerates **site-scoped** languages when the SDK exposes them; falls back to tenant-wide when it does not. Verified on `CHAH DevEx Journey / PROD`. | `live_walkthrough` |
| **m2** | Per-language CRUD round-trip (read → edit one mapping → save → re-read) completes in <10s for `en`, `de`, and one additional language. | `crud_round_trip` |
| **m3** | Create-version flow works end-to-end on real tenant for **both paths**: (a) empty start, (b) copy from chosen source language. New version visible in Sitecore Authoring after one round trip. | `crud_round_trip` |
| **m4** | Multi-language import/export round-trip: export a site with maps in ≥2 languages → bundle JSON contains every language version → import into a clean state → all language versions reconstituted with mappings intact. | `import_export_round_trip` |
| **m5** | Live walkthrough ≥5 min covering language switching across surfaces, create-version (both paths), delete (both scopes), display name editing, multilingual import/export — zero unrecoverable errors. | `live_walkthrough` |

All five gates must pass on the real target tenant before PRD-001 ships. `host_frame_smoke` and `registration` gates carry over from PRD-000 with no semantic change (POC clickdummies re-run because UI surface changed).

## 4. Target Users

Same personas as PRD-000; the change is what they *can* do.

- **Marketers** authoring redirects in a non-`en` working language (often the primary language for their region).
- **Site administrators** managing multilingual Sitecore sites with localized URL structures (e.g. `/de/produkte` ↔ `/en/products`).
- **Migration operators** importing redirect inventories from legacy systems that already cover multiple languages — no pre-conversion to `en` required.

## 5. Product Scope

### In Scope

- **Language picker** in Full Page surface — site-scoped languages when available, tenant-wide fallback otherwise (ADR-0014 records the fallback policy).
- **Picker state** persisted across reloads via `localStorage`; shared across Full Page and Dashboard Widget surfaces within the same browser context (ADR-0017).
- **Language-aware map list** showing current-language `__Display name`, falling back to item name when display name is empty for that language.
- **Per-language CRUD** on existing language versions (read, edit mappings, edit flags subject to A-Field-Matrix, save).
- **Create-version flow** (ADR-0016) — two paths, no name input at creation:
  - "Create empty [language] version" → creates version → operator lands in standard edit flow.
  - "Copy from [source language]" → source picker filtered to languages with existing versions on this map → pre-fills new version with source mappings → operator lands in standard edit flow.
- **Two-state UX** for missing-or-empty versions:
  - "No version exists" (version count = 0) → modal CTA "Create [language] version".
  - "Empty version" (version count ≥ 1, `UrlMapping` empty) → inline "Add first mapping" affordance, no modal.
- **Per-language `__Display name` editing** in map detail header via a discoverable edit affordance (pencil icon or button). When the current-language display name is empty, the detail header shows placeholder text `_(no display name for [language] — add one)_` to make the edit affordance visible.
- **Delete scope safety** (ADR-0018) — every delete on a multilingual map opens a confirmation modal with an explicit radio choice: "Delete current language version only" vs "Delete entire map (all languages)".
- **Dashboard language scoping** — tile inventory and language-aware aggregation resolved via 3 clickdummy variants at `/architect`. Plausible directions captured in § 11 UX Considerations.
- **Import/export schema bump to `redirect-manager/v2`** (ADR-0015) — single JSON bundle, per-map `languages` map of language code → version data. Schema `redirect-manager/v2` is the only supported version.
- **v1 import rejection** — clear error message when operator uploads a v1 JSON: *"Schema v1 is no longer supported. Re-export from a current Redirect Manager instance."* No silent migration.
- **Context Panel banner update** (no behavior change in PRD-001) — Pages-aware enrichment reads `pageInfo.language`; renders the matched language explicitly when Pages-host language ≠ `en`. Static fallback when `en` or context unavailable.

### Out of Scope

Deferred by design (ADR-0014 phasing).

- **Context Panel multilingual matching** — full Pages-host-language matching with head-app resolver contract → **PRD-002**.
- **Head-app resolver contract change** to honor per-language redirects at runtime → **PRD-002** (parallel head-app deliverable).
- **Usage analytics** (Upstash via Vercel Marketplace + head-app instrumentation + Dashboard enrichment used/unused, broken/healthy, top-5 lists + per-mapping analytics column) → **PRD-003**.
- **Sync-back of counters** to Sitecore template (`UsageCount`, `UsageLastSyncedAt`) + "Sync analytics to Sitecore" button → **PRD-004**.
- **Regex-aware Context Panel matching** → future PRD.
- **Bulk multi-language operations** (e.g. "create all language versions for this map", "delete language X across all maps") → future PRD.
- **Concurrent-edit detection** across language versions → future PRD.
- **Audit log** (who changed which language version when) → future PRD.
- **App UI localization** (translating the Redirect Manager UI itself; content localization is in scope, UI localization is not) → future PRD.
- **Public Marketplace App submission** — still shipping as a custom app.

## 6. User Stories

> All stories assume the user is authenticated in Cloud Portal with at least the `Authors` role on the target Sitecore tenant. Acceptance criteria are written as observable behaviors, not implementation details.

**US-1 — Marketer / site admin: switch language on Full Page.**
> *As a* marketer, *I want to* switch the active language in the Full Page surface, *so that* I can view and edit redirects in my primary working language.

Acceptance:

- AC-1.1 Full Page surface exposes a language picker; the picker enumerates languages from `listLanguages` (site-scoped if available, tenant-wide fallback).
- AC-1.2 Picker default = the operator's last-used language (persisted via `localStorage`); on first ever use, defaults to the site's default language; if site default is unavailable, defaults to `en`; if `en` is not in the enumerated language set for this site, defaults to the first enumerated language.
- AC-1.3 Changing the picker re-fetches the current map's data scoped to the chosen language; transient loading state is visible during the swap.
- AC-1.4 When the operator navigates to a different Redirect Map in the same session, the language stays on the picker's current selection.
- AC-1.5 When the operator closes and reopens the Full Page in the same browser, the picker restores the last-used language.
- AC-1.6 If the picker cannot enumerate any languages (SDK failure), it shows an error state with a "Retry" affordance; the rest of Full Page remains functional in fallback mode pinned to `en`.

**US-2 — Marketer / site admin: see which maps have content in the current language.**
> *As a* marketer, *I want to* see at a glance which Redirect Maps have content in my current language and which do not, *so that* I can spot maps that still need a localized version.

Acceptance:

- AC-2.1 The map list row shows the **current-language** `__Display name`; if the current language has no display name, the row falls back to the item name.
- AC-2.2 Each map row surfaces a compact indicator distinguishing "this map has a version in the current language" from "this map has no version in the current language". Exact visual form (chip, dot, icon) resolved via UI clickdummies at `/architect` (OQ-5). A richer "languages with versions" multi-indicator is out of scope for PRD-001; if a clickdummy variant naturally surfaces it, that's a bonus, not a requirement.
- AC-2.3 Maps without a version in the current language remain visible and selectable in the list; selecting one navigates to the detail pane and triggers the no-version flow (US-4).

**US-3 — Marketer: edit a redirect in the current language.**
> *As a* marketer, *I want to* edit a redirect's source/target/type/flags in my current language, *so that* my changes affect only that language's redirects without modifying other language versions.

Acceptance:

- AC-3.1 Inline-edit on a mapping row commits the edit to **the current language version only** (writes pass `language: <current>`).
- AC-3.2 When the operator edits a field that is shared across language versions, the UI surfaces an inline hint before save: *"Shared across languages — changes apply to all language versions of this map."* The shared/versioned classification per field is determined by the field-matrix capture at `/architect` (OQ-2, blocking).
- AC-3.3 When the operator edits a field that is versioned per language (definitely includes `UrlMapping`), the change is scoped to the current language version only.
- AC-3.4 Successful save shows a toast with the language explicitly named: *"Saved to [language] version."*
- AC-3.5 When any field-edit is dirty (typed but unsaved), the language picker becomes read-only with a tooltip *"Save or discard your changes before switching languages."* The picker is re-enabled after the operator saves or explicitly discards.

**US-4 — Marketer / site admin: open a map that has no version in the current language.**
> *As a* marketer, *I want to* be told clearly when a map has no content for my current language, *so that* I can decide whether to create a version or switch to a language that does have content.

Acceptance:

- AC-4.1 Detail pane shows a clear empty state: *"This map has no `[language]` version yet. Create one to start adding mappings."*
- AC-4.2 Empty state offers two affordances: **(a)** "Create empty [language] version", **(b)** "Copy from another language".
- AC-4.3 The empty state lists which languages **do** have versions so the operator can switch if they wanted to view content, not create.

**US-5 — Marketer / site admin: open a map whose current-language version exists but is empty.**
> *As a* marketer, *I want to* distinguish a Redirect Map that has a real (but empty) language version from one that has no version at all, *so that* the system reflects what Sitecore actually shows.

Acceptance:

- AC-5.1 Detail pane shows the existing version's metadata (created date, last-modified) but displays an inline "Add first mapping" affordance instead of a populated list.
- AC-5.2 No modal interrupts — operator types the first mapping directly.
- AC-5.3 The "empty version" state never offers the "Create version" CTA (the version already exists).

**US-6 — Marketer / site admin: create a new language version from scratch.**
> *As a* marketer, *I want to* create an empty new language version for a Redirect Map, *so that* I can start authoring localized redirects without copying anyone else's data.

Acceptance:

- AC-6.1 Modal CTA "Create empty [language] version" creates the version via the Authoring `createItemVersion` mutation (or equivalent — A-Mutation-Shapes capture).
- AC-6.2 No name input is required; the new version inherits the item name (shared) and the operator lands in the standard edit flow.
- AC-6.3 The new version's `__Display name` is empty by default; map list and detail header fall back to item name.
- AC-6.4 Operator can edit `__Display name` later via the per-language edit affordance (US-9).
- AC-6.5 Failure to create version surfaces a Sonner toast with a friendly summary message and a collapsible "Technical details" section containing the verbatim GraphQL error response. Toast remains until dismissed.

**US-7 — Marketer / site admin: create a new language version by copying from another language.**
> *As a* marketer, *I want to* pre-fill a new language version's mappings from an existing language, *so that* I can translate URLs without re-entering the entire mapping list.

Acceptance:

- AC-7.1 "Copy from another language" opens a source-language picker.
- AC-7.2 Source picker shows **only languages that have existing versions on this map with non-empty `UrlMapping` content**. Languages without versions are filtered out (not shown as disabled). Languages with versions but empty `UrlMapping` are filtered out as well (copying from empty would produce an empty new version, which the operator should reach via the "empty start" path instead).
- AC-7.3 The new version is created with **mappings copied verbatim** from the chosen source. Versioned fields copy; shared fields are not duplicated (they already apply across languages).
- AC-7.4 UI hint communicates the copy scope: *"Mappings will be copied from [source language]. Flags and redirect type are shared across all languages and apply automatically."* Final wording adjusts after OQ-2 field-matrix capture if any flag turns out to be versioned.
- AC-7.5 On success, operator lands in the standard edit flow with the new version's mappings pre-populated and editable.
- AC-7.6 Failure to copy-create surfaces a Sonner toast with a friendly summary and a collapsible "Technical details" section containing the verbatim GraphQL error. If the create succeeded but the populate failed, the new (empty) version is **rolled back** by calling `deleteItemVersion` on the newly created version; the operator is left in the original state. If rollback itself fails, the operator sees a "Partially populated version detected" warning and is offered a manual cleanup affordance.

**US-8 — Marketer / site admin: delete a map with explicit scope choice.**
> *As a* marketer, *I want to* delete either just the current language version or the entire Redirect Map (all languages), *so that* I never accidentally remove translations I did not mean to touch.

Acceptance:

- AC-8.1 "Delete" action on a multilingual map (≥1 language version exists) opens a confirmation modal with an explicit radio choice: **(a)** "Delete current language version only — other languages remain"; **(b)** "Delete entire map and all language versions".
- AC-8.2 **No default radio is pre-selected.** Operator must explicitly choose before the "Delete" button enables. This rule applies uniformly — including the single-version edge case (see AC-8.6).
- AC-8.3 Modal shows the map name and the number of language versions that exist (e.g. *"This map has 3 language versions: en, de, fr"*).
- AC-8.4 Choosing (a) calls the version-delete Authoring mutation `deleteItemVersion` (or operator-confirmed substitute). If the mutation surface does not exist (resolved at `/architect`), delete-version-only is cut from PRD-001 and only option (b) is offered; modal copy adjusts to: *"Single-version-delete is unavailable in this Authoring schema. Delete the entire map across all languages?"* See R-1 mitigation for the decision rule.
- AC-8.5 Choosing (b) calls the whole-item `deleteItem` mutation (existing from PRD-000).
- AC-8.6 If the map has only one language version, the modal still appears with both options visible. Both radios are unselected (per AC-8.2). An informational note below the radios states: *"This map has only one language version; both options remove all redirect data. Choose explicitly to confirm intent."* This preserves the strict no-default safety rule while informing the operator about the equivalence.

**US-9 — Marketer / site admin: edit the per-language display name.**
> *As a* marketer, *I want to* set a localized display name for a Redirect Map per language, *so that* the map list shows recognizable labels in each language without changing the underlying item name.

Acceptance:

- AC-9.1 Detail header shows the current-language `__Display name` with a discoverable edit affordance (pencil icon or button next to the title).
- AC-9.2 When the current-language `__Display name` is empty, the header shows placeholder text `_(no display name for [language] — add one)_` to make the affordance visible.
- AC-9.3 Editing commits to **the current language version only**; other languages are unaffected.
- AC-9.4 Save behavior: pressing Enter or clicking the explicit "Save" button commits; pressing Escape or clicking "Cancel" discards. Blur does **not** commit (avoids accidental save). Successful save shows a toast.
- AC-9.5 Editing the display name does not modify the **item name** (shared across languages).

**US-10 — Site admin: export multilingual redirect inventory.**
> *As a* site administrator, *I want to* export a site's entire redirect inventory including all language versions into a single JSON file, *so that* I can back up, transfer, or version-control the multilingual configuration.

Acceptance:

- AC-10.1 Full Page export action produces a single JSON file conforming to schema `redirect-manager/v2`.
- AC-10.2 Each map entry includes a `languages` object keyed by language code (`en`, `de`, …); each language entry includes `__Display name`, `UrlMapping` (parsed and structured), and `__Updated`.
- AC-10.3 Shared fields (`RedirectType`, the 3 flags — pending A-Field-Matrix) appear once per map, **not** duplicated per language.
- AC-10.4 Item name and item GUID appear once per map (always shared in Sitecore).
- AC-10.5 The 3 export actions from PRD-000 (open in new tab / copy to clipboard / download) remain available.

**US-11 — Site admin: import multilingual redirect inventory.**
> *As a* site administrator, *I want to* import a multilingual JSON bundle and have all language versions reconstituted in Sitecore, *so that* I can promote redirect configurations across environments without per-language manual work.

Acceptance:

- AC-11.1 Import wizard accepts a v2 JSON bundle and validates against the Zod schema before any mutation runs.
- AC-11.2 Preview step shows per-map diffs by GUID (existing logic from PRD-000) **plus a language axis**: each map row shows which languages exist on the tenant vs which languages are in the bundle. Exact preview UI resolved via clickdummies at `/architect` (OQ-F from brain-dump).
- AC-11.3 Per-row action picker remains 3 actions (create / overwrite / skip) — applied at **map level**, fanning out across all language versions in the bundle. (Per-language-per-map action sub-axis is OQ-F-deferred.)
- AC-11.4 Apply step creates / overwrites missing language versions via `createItemVersion`-equivalent (or `updateItem` with new language scope).
- AC-11.5 Summary step lists per-map outcome with per-language detail (e.g. "Map X: en updated, de created, fr unchanged").
- AC-11.6 v1 JSON imports are **rejected** with a clear error: *"Schema v1 is no longer supported. Re-export from a current Redirect Manager instance."* No silent migration.
- AC-11.7 Newly minted GUIDs for created maps surface the same warning as PRD-000 (ADR-0009).

**US-12 — Site manager: Dashboard tiles reflect multilingual content.**
> *As a* site manager, *I want to* see redirect totals scoped to my chosen language (or aggregated across all languages), *so that* I can monitor the health of each locale or the whole tenant.

Acceptance:

- AC-12.1 Dashboard Widget inherits the same persisted picker state as Full Page (via shared `localStorage` key).
- AC-12.2 Tile inventory and scoping behavior resolved via 3 UI clickdummy variants at `/architect`. At minimum, tiles must reflect the current picker selection or expose an explicit "all languages" mode.
- AC-12.3 Tiles maintain PRD-000's "footnote" pattern: *"Redirect counts only — usage analytics ship in a follow-on release (PRD-003)."*

**US-13 — Marketer: Context Panel reflects en-only matching transparently.**
> *As a* marketer, *I want to* understand exactly why the Context Panel shows the matches it does on non-`en` pages, *so that* I am not confused when relevant `de` redirects fail to appear.

Acceptance:

- AC-13.1 Context Panel matching behavior **does not change** in PRD-001 (still `en`-only exact-string matching).
- AC-13.2 The non-dismissible banner reads pages-host language via `pageInfo.language`:
  - When language is `en` or unavailable: *"Exact-match, en-only. Regex + multilingual matching ship in follow-ons."*
  - When language is non-`en` (e.g. `de`): *"Currently viewing a `de` page — matches shown are `en`-only. Multilingual matching ships in PRD-002."*
- AC-13.3 Banner remains non-dismissible (existing PRD-000 rule).

## 9. Technical Considerations

- **SDK surfaces (new):** `xmc.sites.languages` (or equivalent — A-Site-Lang-Surface) for site-scoped enumeration. Tenant-wide fallback via `xmc.authoring.graphql` query against `__Languages` on the site root item or system items, if the dedicated wrapper is absent.
- **SDK surfaces (extended):** `xmc.authoring.graphql` — new verbs `createItemVersion`, `deleteItemVersion` (or substitutes — A-Mutation-Shapes). Existing reads / writes parameterize on `language` (already-present seam).
- **Sitecore-side preconditions:** None. PRD-001 does not require any Sitecore template changes (those land in PRD-004 sync-back). Existing Redirect Map template and Redirect Map Grouping template work as-is.
- **Head-app contract:** Unchanged in PRD-001. Head-app continues to resolve `en` redirects only. PRD-002 changes this contract.
- **Marketplace SDK envelope quirks** from PRD-000 carry forward (body inside `params`, double `.data.data` unwrap) — captured in `lib/sdk/redirects-write.ts` and `docs/decisions.md`. Apply same pattern to new mutation verbs.
- **Real-tenant fixture capture** (ADR-0013 workflow) — progressive, not a hard gate. Plan: capture at least one map with multiple language versions, one map with `en` only, one map with `de` only (no `en`), one map with versioned-but-empty content. Operator-tenant `CHAH DevEx Journey / PROD` availability resolved via OQ-7.
- **`__Display name` accessor:** Standard Sitecore field, accessed via the named accessor pattern `field(name: "__Display name")` per the established memory note `reference_sitecore_authoring_write_envelopes`. Field-ID-based access is not used (less brittle to template changes). Verify the read path returns the expected shape during `/architect` fixture capture.
- **State management:** Picker state persisted via plain `localStorage` (no library), keyed e.g. `redirect-manager.language.<tenantId>.<siteId>`. ADR-0017 records the exact key shape and migration handling.
- **Field-versioning matrix:** Assumed shape — `UrlMapping` versioned; `RedirectType` shared; 3 flags shared; `__Display name` versioned; item name shared. A-Field-Matrix capture confirms or revises during `/architect`. The matrix directly drives FR-3, FR-4, FR-7, FR-21, and the UI hint copy.
- **Performance — cross-language Dashboard aggregation:** If Dashboard tiles need cross-language totals, batch reads via parallel `xmc.authoring.graphql` calls or a single aggregated query (depending on what Authoring exposes). Avoid per-language sequential blocking.
- **Schema bump impact:** `redirect-manager/v2` is a clean break. All Zod schemas in `site/lib/schema` are renamed/replaced; no parallel-v1 path. Import wizard's diff logic (`buildItemDiffs`) acquires a language dimension. Export serializer rewrites top-to-bottom.
- **Smoke-gate plan:** 5 gates — see § 3 Success Metrics. Plus `registration` + `host_frame_smoke` from PRD-000 (POC clickdummies re-run because UI surface changed).
- **Backward-shipping consideration:** PRD-001 is an additive change on top of installed PRD-000. Operators who already installed PRD-000 should upgrade without losing en-only mappings — the Redirect Map items already exist in Sitecore in whatever languages they have; no on-disk migration required. Picker defaults to `en` (last-used or fallback) so first-launch UX is familiar.

## 10. Data Model Overview

### Language picker state (client-only, localStorage)

```ts
type PickerState = {
  tenantId: string;
  siteId: string;
  language: string; // e.g. "en", "de", "fr-CH"
  lastUsedAt: string; // ISO-8601
};
// localStorage key: redirect-manager.language.<tenantId>.<siteId>
```

### Multi-language JSON bundle (schema `redirect-manager/v2`)

```jsonc
{
  "schema": "redirect-manager/v2",
  "exportedAt": "2026-05-13T09:20:23Z",
  "site": {
    "name": "MyMarketingSite",
    "collectionName": "MyCollection",
    "siteId": "..."
  },
  "siteLanguages": ["en", "de", "fr"],
  "maps": [
    {
      "guid": "{ABCD-...}",
      "itemName": "SEO Redirects",
      "shared": {
        "redirectType": "Redirect301",
        "regex": false,
        "includeVirtualFolder": true,
        "stripQueryString": false
      },
      "languages": {
        "en": {
          "displayName": "SEO Redirects",
          "lastUpdated": "2026-05-12T12:00:00Z",
          "mappings": [
            { "source": "/old", "target": "/new" }
          ]
        },
        "de": {
          "displayName": "SEO-Weiterleitungen",
          "lastUpdated": "2026-05-12T18:00:00Z",
          "mappings": [
            { "source": "/alt", "target": "/neu" }
          ]
        }
      }
    }
  ]
}
```

> Final field-versioning split (which fields go under `shared` vs `languages.<code>`) depends on OQ-2 field-matrix capture at `/architect`. **Bounded worst case:** if all four currently-shared fields (`redirectType`, `regex`, `includeVirtualFolder`, `stripQueryString`) turn out to be versioned, they move under `languages.<code>` and the `shared` block contains only `itemName` + `guid` (which are always shared in Sitecore). ADR-0015 records the final shape.
> All timestamps use ISO-8601 extended format (`YYYY-MM-DDTHH:MM:SSZ`). Sitecore-internal compact format (`yyyyMMddTHHmmssZ`) from PRD-000's `parseSitecoreCompactDate` is normalized at the export-serializer boundary.

### Authoring GraphQL contracts (assumed)

```graphql

    item { ... }
  }
}

  deleteItemVersion(input: { itemId: $itemId, language: $language }) {
```

**Hard-fail commitment:** If `deleteItemVersion` is absent from the Authoring schema (or has a shape that cannot achieve true version removal), delete-version-only is cut from PRD-001 per R-1 decision rule. **No silent fallback** that leaves a version stub — the silent-fallback path was considered and rejected (it corrupts the no-version-vs-empty-version model from US-4/US-5).

### Rollback story

PRD-001 is additive on top of PRD-000's installed app. If a critical bug ships:

- **App-level rollback:** Revert the PRD-001 PR; PRD-000 install continues to function. No Sitecore data changes need rollback (operator's Redirect Map items are never destructively modified by PRD-001 reads; writes are scoped to the operator's explicit actions).
- **Data-level rollback:** A user who imports a v2 bundle and then realizes a mistake can re-import the previous v2 bundle with `overwrite` action; no schema-level rollback exists because there is no v1 back-compat to fall back to.
- **Cloud Portal Test App:** Same App ID as PRD-000; no re-registration required for in-place updates.
