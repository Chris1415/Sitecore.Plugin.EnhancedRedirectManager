## 1. Executive Summary

Redirect Manager PRD-002 ships a **V4-aligned redesign with reality validation** of three Sitecore Marketplace extension-point routes (Full Page, Context Panel, Dashboard Widget) using the V4 "Blok Elevated" marketing POC at `pocs/poc-marketing-v4-blok-elevated/` as the canonical design intent. The redesign adopts V4's **visual language AND V4's UX evolutions** where V4 shows a different interaction pattern than current code (e.g. Context Panel inline quick-add form replaces the existing modal; hero-scale `<h1>` count header replaces the quiet header). The **functional contract with Sitecore** (SDK reads/writes, CRUD semantics, security model, en-only operation, RedirectType enum, mapping schema) is preserved 1:1 — no new SDK surfaces, no new GraphQL mutations, no new backend.

The redesign is validated against current `site/` source code as canonical reality: where PRD-000 prose said one thing but the shipped code does another (notably, `RedirectType` shipped with **3 values** — `Redirect301`/`Redirect302`/`ServerTransfer` — not the 4 that PRD-000 § 5 listed), PRD-002 reflects the code. Where V4 shows content that has no real data source today (Dashboard Widget hero stat / sparkline / top-destinations list / "Recently shipped" tile; Full Page stat strip / "Last publish by Anna" line / "all healthy" badge), PRD-002 ships **hardcoded mocks** under a per-surface "Preview data" banner so operators cannot mistake them for live values.

The IntroPage at `/` (added by intro commit `76a6507`) stays out of scope per operator decision — PRD-002 is scoped to the 3 daily-driver extension-point routes; an IntroPage redesign is a future opportunity. A follow-on "data plumbing" PRD will wire real data into the V4 visual chassis built here, removing the "Preview data" banners element-by-element as live sources land.

## 2. Problem Statement

PRD-000 ships a working, operator-grade Redirect Manager — correct, useful, but visually quiet to the point that the product under-sells the engineering care behind it. In a portfolio where polish drives adoption, a marketing-grade visual identity is a real differentiator. The 4-variant marketing exploration (2026-05-13) tested four directions (Aurora, Neo Brutalism, Editorial, Blok Elevated). V4 Blok Elevated won because it achieves premium feel **while staying entirely inside the Blok design system** — same Geist Sans/Mono stack, same semantic tokens, same component anatomy, but higher composition ambition: hero typography on the Blok type scale, gradient-text headlines composed from token expressions, frosted-glass surfaces over drifting `--primary` / `--info` plumes, primary-tinted hover glows, kinetic letter reveals, count-ups on stat numbers.

PRD-002 takes V4 from speculative exploration to actual shipped product. The work is threefold: (1) faithfully translate V4 visual decisions across every PRD-000 surface in the three extension-point routes, (2) adopt V4's UX evolutions where they materially improve the product (inline quick-add on Context Panel, hero count header on Context Panel, workspace hero banner on Full Page), (3) reconcile V4's freehand content with reality — V4 was a marketing exploration that took liberties (fabricated "Active"/"Draft" status pills; speculative "Edge caches refreshed across 7 languages" copy that contradicts ADR-0010 + ADR-0023; speculative hero stats; mixed-language `/de/...` mock paths). Reconciliation classifies every V4 element across four categories: (a) already real in code → re-skin only, (b) trivial re-skin → element exists, chrome upgrades, (c) hardcoded mock → content doesn't exist in code today; ship under "Preview data" banner, (d) NEW V4 UX change → interaction pattern differs from current code; V4 pattern ships, functional contract preserved.

## 3. Goals and Success Metrics

Two viable options: (i) the form always **adds to an existing map** — pick the most-recently-updated matching map by default, with a small "+ New map" affordance to fall through to the existing `AddRedirectModal` for full flag control; (ii) the form **creates a new map** by default when no matching map exists. V4 POC ships with no map-picker on the inline form (implies option (ii) or implies single-map default). PRD-002 marks this as OQ-9.
- AC-R5.5 Functional contract: submission executes one of two SDK calls — `updateRedirectMap` (when adding to an existing map; same shape used today in `ContextPanel.handleAddToExistingMap`) or `createRedirectMap` (when creating a new map; same shape used today in `ContextPanel.handleCreateNewMap`). No new mutation. No new SDK surface. The duplicate-source guard from `handleAddToExistingMap` (refetch fresh map → reject duplicate source before write) carries forward unchanged.
- AC-R5.6 Success path: toast confirmation, form clears (source resets to current page route; RedirectType resets to default), panel refetches matched groups + canvas reloads.
- AC-R5.7 Error path: toast error, form preserves entered values.
- AC-R5.8 The full-flag "Create new map" flow (with `preserveQueryString` / `preserveLanguage` / `includeVirtualFolder` checkboxes) — whether retained as a fallback path via the existing `AddRedirectModal` or migrated into an inline-expanded form — is an open `/architect` decision per AC-R2.7.

## 9. Technical Considerations

- **No SDK changes.** PRD-002 is pure client-side rendering / styling + one interaction-pattern change (inline quick-add). The Marketplace SDK envelope, all `xmc.authoring.graphql` calls, all wrappers in `site/lib/sdk/*` are untouched. No re-capture of SDK shapes required.
- **No Cloud Portal Test App re-registration.** Same App ID, same 3 extension points, same scopes. PRD-000 registration carries forward unchanged.
- **CSS architecture.** New V4-elevation CSS lives alongside existing PRD-000 styles. Approach: new `site/styles/elevated.css` (or similar) with V4-specific patterns; tokens come from `site/app/globals.css`. Avoid scattering V4 styles across component files when shared utility classes can centralize them. **Design-token CSS variables for V4-specific tunable values** (plume drift duration, glass alpha, hover-glow tint percentage, hero clamp range) should be defined in **one** place — likely the top of `site/styles/elevated.css` — to avoid drift across files (see R-13).
- **Motion implementation.** Plumes via CSS `radial-gradient` on absolutely-positioned pseudo-elements with `@keyframes` translating + opacity. Kinetic letter reveals via JS-split spans + staggered CSS transitions. Count-ups via `requestAnimationFrame` with easing helpers (V4 POC `data-count` attribute pattern documents the intent). All scoped to specific selectors so the mixed motion budget (D3) is enforceable.
- **Frosted-glass implementation.** `backdrop-filter: blur(<glass-blur>) saturate(<glass-sat>)` + `background: color-mix(in oklch, var(--card) <glass-alpha>%, transparent)`. Specific blur radius / saturation / alpha values live in the design-contract variable set (see § 11 Design contract values). Fallback: solid `--card` with subtle border emphasis. `@supports not (backdrop-filter)` rule covers older browsers.
- **Gradient-clip text.** `background: linear-gradient(135deg, var(--primary), color-mix(in oklch, var(--primary) 55%, var(--info)))` + `background-clip: text` + `color: transparent`. Theme-aware: switches automatically with light/dark since both ends are tokens.
- **Mock-data architecture.** Single constants file: `site/lib/mocks/preview-data.ts` (path tentative; architect decides exact location). Exports typed mock values for every mocked element across Full Page + Dashboard Widget. Components import from this one file. A surface-level boolean flag (e.g. `previewDataActive: { fullPage: true, dashboardWidget: true }`) gates banner rendering + `data-preview-mock` attribute emission. Follow-on "data plumbing" PRD swaps the constants for real SDK reads + flips the flag for the migrated surface in one place. See FR-R2.
- **"Preview data" banner component.** New component (e.g. `site/components/ui/preview-data-banner.tsx`) that renders the banner. Mounted at the top of any surface where the surface-level flag is true. Structural test pairs banner-presence with `data-preview-mock="true"` attribute presence on the surface.
- **Reduced-motion guard.** New structural test asserts every `@keyframes` block in `site/` source has a corresponding `@media (prefers-reduced-motion: reduce)` rule that disables the animation. Implementation: grep-based assertion on built CSS.
- **No-hex guard.** New structural test asserts no `#[0-9a-fA-F]{3,8}` hex literals appear in `.ts` / `.tsx` / `.css` files outside `site/app/globals.css`. Grep-based. Allow-list contains exactly one path.
- **No-language-count guard.** New structural test asserts no string of the form `\b\d+\s+languages?\b` or `across\s+\d+\s+languages?\b` appears in any source file outside test fixtures. Grep-based.
- **No-Active/Draft-pill guard.** New structural test asserts no JSX string literal `"Active"` or `"Draft"` in a status-pill / badge context, and no `status-pill--active` / `status-pill--draft` / `lr-row__dot--draft` CSS class anywhere in the source tree.
- **Marketing copy storage.** Hero copy + Dashboard headline copy + Context Panel hero header copy live inline in component files for now (one or two strings each). If copy grows, refactor to a single content file. Not a load-bearing decision at PRD level.
- **Webfont loading.** Hero-scale type (48-96px clamp) on Geist Sans uses the existing PRD-000 `next/font/google` strategy. Architect verifies no FOUT at hero scale; proposes adjustments (preload, `display: optional`) if needed. See NFR-R6.
- **Footer z-index audit.** HahnSoloFooter at `position: fixed; bottom: 0; right: 0; z-index: 50; pointer-events: none` (verified in `site/components/hahn-solo-footer.tsx`). New plumes + frosted-glass layers introduce new stacking contexts. `/architect` verifies the footer remains visible on every redesigned route. Plumes must use a stacking context below the footer's `z-index: 50`.

Test assertions that check Tailwind class strings (e.g. `toHaveClass("bg-card")`) may need updating to match new V4 utility classes. **The AddRedirectModal-vs-inline-quick-add change (US-R5) is the largest test-impact item** — existing modal-flow Vitest tests (`AddRedirectModal.test.tsx` if it exists, or modal-related assertions in `ContextPanel.test.tsx`) need to be re-pointed at the inline form OR retained if the modal is kept as a fallback. ### Design contract values (V4-specific tunables)

ACs are observable; the specific easings, durations, blur radii, alpha values that achieve them live here as a design contract. **All values are defined once as CSS variables in `site/styles/elevated.css`** (or equivalent) to prevent drift across files (mitigation for R-13). Architect refines exact values during `/architect`; the V4 POC reference values are listed below as anchors.

- Plume drift loop: ~28s (V4 POC reference); reduced-motion: paused/static.
- Frosted-glass blur radius: ~18px (V4 POC); saturation: ~160%; surface alpha: ~80%.
- Frosted-glass fallback (no `backdrop-filter` support): solid `--card` with `color-mix(in oklch, var(--background) 95%, var(--primary) 5%)` accent.
- Hero hover lift transform: `translateY(-2px) scale(1.005)`.
- Hero hover glow: `box-shadow: 0 12px 32px color-mix(in oklch, var(--primary) 14%, transparent)`.
- Easing curve: `cubic-bezier(0.16, 1, 0.3, 1)` for hover transitions.
- Kinetic letter reveal: ~28ms per-letter stagger; ~800ms total; reduced-motion: instant render.
- Count-up duration: ~1400ms (V4 POC `data-duration="1400"`); reduced-motion: lands at final value immediately.
- Hero gradient-clip text: `linear-gradient(135deg, var(--primary), color-mix(in oklch, var(--primary) 55%, var(--info)))` + `background-clip: text` + `color: transparent`.
- Hero clamp range: 48-96px (Geist Sans 700).
- Mixed motion budget enforcement: see FR-R9.
