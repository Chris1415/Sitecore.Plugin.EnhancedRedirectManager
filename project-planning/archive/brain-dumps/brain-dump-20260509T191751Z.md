## 1. Idea Summary

**Redirect Manager** is a Sitecore Marketplace **client-side** app that gives content authors, marketers, and implementers a first-class UI for redirect operations across an XM Cloud tenant. It exposes **three extension points** at three levels of context:

- **Context Panel (Pages editor)** — for the page currently being edited, list every redirect that affects it (where this page is the source *or* the target). Inline CRUD on those redirects.
- **Dashboard Widget (per site)** — at-a-glance analytics: number of Redirect Maps, number of individual redirects, used vs unused, broken vs healthy.
- **Full Page** — the power-user surface: pick a site collection → pick a site → browse all Redirect Map items (each with N source→target mappings), full CRUD on item and mapping level, JSON import/export across environments.

Redirect rules stay in Sitecore (canonical source) accessed via the Authoring GraphQL endpoint through `marketplace-sdk-xmc`. Usage telemetry lives **outside** Sitecore — Upstash Redis provisioned via the Vercel Marketplace — fed by the head application's redirect handler. The marketplace app reads from Upstash for analytics. **Synchronizing a consolidated counter back onto the Sitecore Redirect Map item is explicitly deferred to a follow-on PRD** so the MVP does not require any Sitecore template changes.

Built on `sitecore:setup-marketplace-client-side` (Mode A only, ClientSDK + xmc package, Next.js App Router, Blok design system, Vitest stack).

## 2. Problem Statement

Redirect management today is painful for everyone:

- **Marketers / content authors** working in Pages have no in-context visibility into redirects affecting the page they're editing. To check if a page is redirected away or has redirects pointing into it, they leave Pages, open Content Editor, navigate to the site's `Settings/Redirects` folder, and read raw item fields. Most don't, so misconfigured redirects ship to production.
- **Site managers** have no per-site overview. They can't answer "how many redirects do we have?", "how many are actively used?", or "how many lead to dead targets?" without manual audits.
- **Implementers / admins** have no portable way to move redirect sets between environments (dev → staging → prod). Today this means hand-replicating Redirect Map items or writing one-off PowerShell.
- **No telemetry.** There's no signal whether a redirect is alive, dead, or hot. Stale redirects accumulate; broken redirects ship undetected.
- **Content Editor UX for the Redirect Map item is rough** — a flat regex pair list with no validation, preview, or batch operations.

The data, the editing capability, and the routing infrastructure all exist. They just need a purpose-built surface that meets each user where they work.

## 3. Target Users

**Primary — Content authors and marketers in Pages.** The Context Panel is for them. They already have the page open; they need to see redirects affecting *this* page and fix them inline without leaving Pages.

**Primary — Site managers and content owners.** The Dashboard Widget is their daily glance: how many redirects on my site, how many actually used, how many broken.

**Primary — Sitecore implementers, dev leads, and admins.** The Full Page extension is their workshop: bulk operations, JSON-driven environment portability, multilingual CRUD.

**Not the target:** End-site visitors (the head app handles their redirect resolution; this app does not run in that path) and Authoring GraphQL itself (we use it as a consumer, we do not extend it).

## 4. Proposed Solution

A single Marketplace app published into three extension points, sharing one authoring stack and one analytics stack.

**Extension point 1 — Context Panel (Pages editor)**
- Lists redirects affecting the currently edited page. MVP matches **exact source** OR **exact target** equality with the current page path. Regex-pattern matching is **deferred to a later phase** (explicitly noted in UI as "showing direct-string matches; regex matches not yet covered").
- Per matched redirect: status code badge (301 / 302 / 307 / Server Transfer), the source→target pair, "preserve QS" / "preserve language" indicators, inline edit and delete, link to "open in Full Page".
- "Add redirect for this page" affordance — pre-populates source = current page path.

**Extension point 2 — Dashboard Widget (per site)**
- Stat tiles: # of Redirect Map items, # of individual mappings, # used (hit at least once in window) vs # unused, # broken (404-reported by head app) vs # healthy, last-updated timestamp.
- Compact list of "top 5 most-used redirects" and "top 5 broken".
- Source for "used / broken" is Upstash; tile counts cross-reference Sitecore for totals.

**Extension point 3 — Full Page**
- Site collection picker → site picker (populated via `marketplace-sdk-xmc` sites functionality).
- For the chosen site: list of Redirect Map items (groups), each expandable to show its mappings.
- CRUD on items: create, rename, delete a Redirect Map; edit Type (301/302/307/Server Transfer), Preserve QS, Preserve Language, Include virtual folder.
- CRUD on mappings within an item: add, edit, delete, reorder rows.
- **Multilingual CRUD** — language and version are first-class. Operator picks language; Authoring GraphQL queries/mutations carry the language and version arguments correctly.
- **Import:** upload JSON; the app parses, shows a diff/preview, and prompts the operator on every conflict (overwrite / skip / merge per item). On approval, missing items are **created**; existing items are touched per the operator's choice. Per-item success/fail summary at end.
- **Export:** JSON dump of the selected site's redirect rules (no analytics — telemetry is environment-specific and intentionally not in the export).
- Per-mapping analytics column: hits, last hit, healthy/broken, last 404 (read directly from Upstash, no Sitecore round-trip).

**Deferred to follow-on PRD (post-MVP, after real-tenant verification):**
- "Sync analytics to Sitecore" button — writes a consolidated counter onto the Redirect Map item via Authoring GraphQL.
- Sitecore template change adding `UsageCount` (int) + `UsageLastSyncedAt` (datetime) fields to the Redirect Map template.

**Cross-cutting**
- Blok design system, Marketplace iframe constraints respected.
- Authoring GraphQL writes use the operator's Cloud Portal session.
- Read-only Upstash REST token shipped to the iframe; head app holds full token via env var.
- One shared Upstash database across all environments; multitenant via key prefix `{tenantId}:{site}:{mapId}:{index}`.

## 5. Value Proposition

- **Eliminate the Pages → Content Editor jump** for redirect awareness. Marketers see and fix redirects from the same surface they already work in.
- **Site-level visibility** that doesn't exist today: redirect health, redirect usage, broken-redirect count.
- **Portable redirect sets** via JSON import/export, with conflict prompts so operators don't blindly overwrite production. Solves environment promotion without bespoke scripting.
- **Cheap, accurate usage telemetry** — head-app `INCR` on Upstash is sub-millisecond and free of Authoring write pressure. Sitecore stays the source of truth for rules; Upstash carries the firehose.
- **Multilingual support out of the gate** — language/version aware CRUD is rare in the redirect-tool space and matters for international content teams.
- **Dogfood value** — exercises the agentic pipeline on a Marketplace app with a non-trivial backend dependency (Vercel Marketplace + Upstash) and three extension points fed by the same data layer.

## 6. Core Features

1. **Three Marketplace extension points** — Context Panel, Dashboard Widget (per site), Full Page — sharing data layers and component primitives.
2. **Site collection / site picker** in the Full Page, populated via `marketplace-sdk-xmc` sites functionality.
3. **Redirect Map item CRUD** via Authoring GraphQL — create, read, update, delete items under `/sitecore/content/{COLLECTION}/{SITE}/Settings/Redirects/*`.
4. **Mapping CRUD** within a Redirect Map item — manage the source→target row list.
5. **Multilingual + versioned CRUD** — language/version arguments correctly threaded through Authoring GraphQL queries and mutations.
6. **JSON import** with mandatory conflict preview and per-conflict operator decision; auto-creates missing items.
7. **JSON export** of redirect rules only — analytics are intentionally environment-specific and excluded.
8. **Context Panel "redirects affecting this page"** — exact source/target string match in MVP; regex matching deferred.
9. **Per-site analytics dashboard** — totals from Sitecore, usage and broken/healthy from Upstash.
10. **Hybrid analytics store** — Upstash Redis as canonical for telemetry; no Sitecore-side analytics field in MVP.
11. **Blok-styled UI** across all three extension points, respecting iframe viewport constraints.

**Deferred features (own follow-on PRD, after MVP real-tenant verification):**
- "Sync analytics to Sitecore" action.
- New Redirect Map template fields (`UsageCount`, `UsageLastSyncedAt`).

## 7. Decisions

All locked during the brainstorming session. Recorded as decisions, not assumptions.

- **D1 — Product slug:** `redirect-manager` (no `.hahn-solo` suffix; aimed at distribution).
- **D2 — Scaffold:** `sitecore:setup-marketplace-client-side` (Mode A only, ClientSDK + xmc, Vitest stack, Next.js App Router, Blok). No server-side OAuth proxy; all Sitecore writes ride the operator's Cloud Portal session through Authoring GraphQL.
- **D3 — Extension points:** Context Panel, Dashboard Widget (per site), Full Page. Three routes, registered to the corresponding Cloud Portal extension points via `sitecore:marketplace-sdk-extension-routes` post-scaffold.
- **D4 — Source of truth for redirect rules:** Sitecore Authoring GraphQL on Redirect Map items at `/sitecore/content/{COLLECTION}/{SITE}/Settings/Redirects/*`.
- **D5 — Analytics store:** **Upstash Redis** via Vercel Marketplace integration. Lightweight, atomic `INCR`, sub-ms, env-vars auto-injected into linked Vercel projects.
- **D6 — Single shared Upstash database** across all Sitecore environments (dev / staging / prod) and tenants. Multitenant via key prefix using the tenant identifier from the Marketplace app context. Head app uses an env variable to identify its environment when writing.
- **D7 — Iframe access pattern (Q6):** direct browser → Upstash REST API using a **read-only** token. The Upstash database is dedicated to redirect analytics — no other data — so whole-DB read scope is acceptable.
- **D8 — Counter sync trigger (Q7):** **manual button** in Full Page UI ("Sync analytics to Sitecore"). Operator-driven, predictable, no surprise authoring writes. **Deferred to a follow-on PRD** per D18 — keeps Sitecore template changes out of the MVP install.
- **D9 — Broken detection (Q3):** No live HTTP probing from this app. Head app captures 404s when resolving a redirect at request time and reports them into Upstash. Dashboard reads `count_404` and `last_404_at` from Upstash. Live probing is **out of scope**.
- **D10 — Import behavior (Q8):** Imports **create missing** Redirect Map items. On conflict, the user is prompted on the fly — preview-and-decide per item, not blanket overwrite.
- **D11 — Export behavior (Q8):** Export contains the redirect **rules only** (Type, Mappings, Preserve QS, Preserve Language, Include virtual folder). Analytics are **environment-specific** and explicitly excluded from export.
- **D12 — Context Panel matching (Q9):** MVP matches when the current page path is **exactly equal** to a redirect's source string OR target string. **Regex matching is deferred** to a later phase. UI must surface this limitation explicitly so users don't assume regex coverage.
- **D13 — Group ≡ Redirect Map (Q4):** A "redirect group" in our wording is **one Redirect Map item** (child of `…/Settings/Redirects`). Each row in the item's Mapping field is one "individual redirect". No extra folder layer.
- **D14 — Multilingual CRUD (Q13):** **In scope.** Redirect Map items have language + version. CRUD operations must use Authoring GraphQL's language/version parameters correctly. Operator picks language in the UI.
- **D15 — Site / collection discovery (Q14):** Use `marketplace-sdk-xmc` sites functionality (SDK-first per `40-sdk-contracts`). Authoring GraphQL traversal only as fallback if a wrapper does not exist.
- **D16 — Vercel hosting topology (Q11):** Same Vercel team, two linked projects (head app + marketplace app). One Upstash Marketplace integration linked to both projects so envs are shared.
- **D17 — Head-app responsibility:** Out of scope for this product. Head app increments the Upstash counter on hit and reports 404s; this is implemented separately by Christian and is **not** part of the marketplace app deliverable.
- **D18 — MVP excludes Sitecore-side analytics sync.** The "Sync analytics to Sitecore" button and the corresponding Redirect Map template fields (`UsageCount`, `UsageLastSyncedAt`) are **deferred to a follow-on PRD** that ships only after the MVP has been verified on a real tenant. **Why:** keeps the MVP install free of Sitecore template changes (the heaviest non-code precondition), tightens the verification loop, and aligns with the "phase per SDK surface, ship-and-verify between phases" working style. **MVP analytics consumption:** dashboard and per-mapping columns read directly from Upstash; no Sitecore-side mirror is required for MVP value.

## 8. Assumptions

Items that remained unresolved after probing — must be confirmed at PRD time.

- **A1 — Authentication.** The operator's active Cloud Portal session grants Authoring GraphQL read AND write permissions on items under `/sitecore/content/.../Settings/Redirects/*`. No separate auth flow built into this app.
- **A2 — Template field for sync-back.** Not relevant to MVP per D18. When the follow-on sync-back PRD lands, a new field on the Redirect Map template is required to store the consolidated counter — proposed names `UsageCount` (integer) and `UsageLastSyncedAt` (datetime). Final names + permissions to be re-confirmed at that PRD's discovery stage.
- **A3 — Upstash retention.** No TTL on counter or 404 keys; data persists indefinitely until manually purged. Acceptable for low-cardinality redirect sets.
- **A4 — Bulk import error model.** Per-item success/fail; partial commits acceptable; final summary lists what succeeded and what failed. No transactional rollback across the whole import.
- **A5 — Empty states.** Standard Blok empty-state primitives for "no collections", "no sites", "no redirects yet", "no analytics yet".
- **A6 — File transport.** Import = browser file upload (`<input type=file>`). Export = browser download trigger. No presigned URLs, no Storage SDK.
- **A7 — Tenant identifier.** The Marketplace app context provides a stable identifier (e.g. `tenantId` or `contextId`) usable as the Upstash key prefix. The same identifier is recoverable by the head app via env var or runtime config.
- **A8 — Iframe viewport.** Full Page extension has enough width to render a two-pane layout (site picker + content); Context Panel is narrow enough to require a vertical-stack layout.
- **A9 — Multilingual scope.** All redirect items are stored under one root per site (no per-language item parents); language is purely a versioning axis on the same item.
- **A10 — Virtual folder & path normalization.** "Include virtual folder" affects how the head app resolves a path against a redirect — the marketplace app's matching logic must apply the *same* normalization (or explicitly opt out and surface that to the user). Treated as an implementation detail to be specified at architecture stage.

## 9. High-Level Technical Considerations

- **SDK mode.** Pure Mode A. ClientSDK initialized in the iframe; Authoring GraphQL invoked via `xmc.authoring.graphql` (mutation-capable surface per `marketplace-sdk-xmc`). No `experimental_createXMCClient`, no Auth0 PKCE.
- **GraphQL surface.** Read items via Authoring GraphQL `item(path|where)` queries; write via `createItem`, `updateItem`, `deleteItem` mutations. All calls carry `language` and `version` per D14. Source for shapes: introspected schema captured via `sitecore:sitecoreai-graphql-schemas`. Validate produced GraphQL during architecture stage and again at QA.
- **Site / collection discovery.** SDK-first via `xmc.sites.list` (or equivalent — verify the exact wrapper at PRD time). Falls back to `/sitecore/content/*` traversal only if no wrapper exists. Cite the `.d.ts` shape inline at architecture stage per `40-sdk-contracts`.
- **Upstash schema (informal).**
  - Hit counter: `usage:{tenantId}:{site}:{mapId}:{mappingIndex}` → integer (`INCR`).
  - 404 counter: `health:{tenantId}:{site}:{mapId}:{mappingIndex}:404` → integer; `health:{tenantId}:{site}:{mapId}:{mappingIndex}:last404` → ISO timestamp.
  - Per-site rollups maintained on each `INCR` for cheap dashboard reads.
- **Token model.** Two Upstash tokens minted in the Upstash console:
  - **Read-only** token → exposed in the iframe bundle. Whole-DB read scope (Upstash REST tokens have no per-key ACL) — mitigated by dedicating one Upstash database to redirect analytics, no co-tenancy.
  - **Read-write** token → server-side env var on the head app for `INCR` and 404 reports. Never shipped to the browser.
- **Vercel topology.** Same Vercel team. One Upstash Marketplace integration linked to both projects (head app + marketplace app). `KV_REST_API_URL` and `KV_REST_API_TOKEN` auto-injected; the read-only token override for the marketplace app is a manually managed env var.
- **Multitenant key prefix.** Tenant identifier sourced from the Marketplace app context (see A7) and from an explicit env var on the head app side. Choice of identifier (`tenantId` vs `contextId`) to be locked at architecture time.
- **Context Panel matching algorithm.** MVP: load all Redirect Maps for the current page's site, flatten to mappings, compare current page path equality with each mapping's source AND target. Surface "regex matches not yet covered" notice. Phase 2: regex-aware matching with a JS regex engine, with safety limits (timeout, complexity cap).
- **Import workflow.** JSON parse → diff against existing items in the target site → preview UI shows per-item action (create / overwrite / skip / merge) → operator confirms each → batched mutations → summary report. No transactions across the batch.
- **Export workflow.** Read all Redirect Maps for the chosen site (with chosen language), serialize to JSON omitting all analytics fields, trigger browser download.
- **Blok / Marketplace SDK constraints.** `sitecore:blok-marketplace-integration` for iframe-safe theming; `sitecore:marketplace-sdk-host-frame-testing` for host-frame visual smoke against the POC clickdummy at `/test`.
- **Testing stack.** Vitest + the typed mock client pattern from `marketplace-sdk-client`. RED-first per `30-tdd`. Authoring GraphQL fixtures sourced from real-tenant captures, never paraphrased prose.

## 10. Risks

- **R1 — Authoring GraphQL schema drift / wrapper coverage.** Schema introspection is fresh; the exact shape of mutations on the Redirect Map template needs verification. **Mitigation:** validate every produced GraphQL mutation against a real tenant during architecture and again during QA. User explicitly offered to validate.
- **R2 — Read-only token exposure.** A whole-DB read-only token in the browser is acceptable only because the Upstash database is dedicated to redirect analytics. Risk if anyone later co-tenants other data into that DB. **Mitigation:** ADR locks single-purpose Upstash DB; document in README; check at code-review.
- **R3 — Template change requirement (deferred to follow-on PRD per D18).** Adding `UsageCount` / `UsageLastSyncedAt` fields to the Redirect Map template is a Sitecore platform change, not just an app deployment. **No longer an MVP risk.** Will resurface as a primary risk in the follow-on sync-back PRD; mitigation at that point will be: install-guide precondition, manual button only writes when fields exist, clear error if missing.
- **R4 — Bulk import correctness.** Partial commits with per-item success/fail can leave the target site in mid-state. **Mitigation:** make the conflict preview comprehensive; show the operator exactly what will change; provide an export-before-import "snapshot" affordance for cheap rollback.
- **R5 — Multilingual CRUD complexity.** Language/version handling in Authoring GraphQL is easy to get subtly wrong (e.g. updating one language unintentionally affects the shared field). **Mitigation:** pin per-field "shared vs versioned" semantics during architecture; explicit Vitest coverage for language-switch flows.
- **R6 — Context Panel match accuracy.** MVP exact-match misses every regex redirect, which may be the majority of real-world rules. Operators may distrust the panel. **Mitigation:** explicit UI banner surfacing the limitation; make Phase 2 regex matching the highest follow-on priority.
- **R7 — Virtual folder normalization mismatch.** If the marketplace app's matching logic disagrees with the head app's path normalization, the Context Panel shows incorrect "affecting this page" lists. **Mitigation:** document the exact normalization in an ADR; align with the head app's resolver implementation; add integration tests.
- **R8 — Vercel Marketplace integration vs token control.** The auto-injected `KV_REST_API_TOKEN` is full-access. Manually minting a read-only token in the Upstash console (and keeping it in sync if rotated) is a small operational burden. **Mitigation:** documented in install guide; rotation playbook in README.
- **R9 — Head-app dependency cadence.** Telemetry only exists if the head app's INCR + 404 reporting is implemented. The marketplace app must degrade gracefully when Upstash is empty (show "no data yet" rather than zeros that look like "unused"). **Mitigation:** explicit empty-state design; "needs head app instrumentation" callout in README.
- **R10 — Iframe viewport for Full Page.** Full Page extension iframe width may not accommodate a two-pane layout comfortably. **Mitigation:** UI design phase tests across the actual extension iframe dimensions; tabbed layout as fallback.

- **OQ1 — Field naming and shape on the Redirect Map template** for the synced counter. Deferred to the follow-on sync-back PRD per D18.

- **OQ3 — Exact SDK wrapper for site discovery.** `xmc.sites.list`? Verify name and shape from `.d.ts` at architecture stage.

- **OQ5 — Conflict-resolution UI for import.** What does the per-item preview actually look like — diff-style? Side-by-side? Three-way?
- **OQ6 — Retention policy for Upstash counters.** Indefinite is fine for now; should there be a manual "purge analytics for this site" admin action?
- **OQ7 — Phase 2 regex matching design.** Engine choice, timeout/complexity safety, UX for showing "regex match" vs "exact match" results.
- **OQ8 — Authoring GraphQL bulk batching.** Are there per-mutation limits we'll hit on import, and do we need request batching or rate limiting?
- **OQ9 — "Top 5 most-used / broken" computation.** Cheap to compute on a few dozen mappings; needs a strategy when a site has thousands.
- **OQ10 — Sync-back PRD scope and ordering.** Once MVP is verified on a real tenant, scope the follow-on PRD: template change, sync button, optional inclusion of synced counter in JSON export, migration story for items predating the new fields.
