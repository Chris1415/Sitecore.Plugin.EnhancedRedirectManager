## 1. Idea Summary

Redesign the Redirect Manager Sitecore Marketplace app's visual identity end-to-end using the V4 "Blok Elevated" marketing POC as the canonical visual reference. Scope is **pure visual / surface re-skin** — functional behavior stays 1:1 with PRD-000. All three extension points (Full Page, Context Panel, Dashboard Widget) plus the `/` IntroPage are redesigned together. V4 elements showing data that doesn't exist yet (hero stats, sparklines, top-hit lists) are rendered with **hardcoded mock values** behind a single per-surface "Preview" banner; the wiring-up of real data sources is tracked as a follow-on PRD ("data plumbing").

PRD-002 (proposed numbering — PRD-001 is the cancelled multilingual record) ships a visually premium, marketing-grade Redirect Manager that stays entirely inside the existing Blok Nova token system. Zero invented hex. Same Geist Sans/Mono stack. Same component anatomy. Higher composition ambition + motion + hero typography + gradient-from-tokens.

## 2. Problem Statement

PRD-000 shipped a working, operator-grade Redirect Manager — utility-first, Blok-default, no marketing flair. It is **correct, useful, and visually quiet** to the point that it under-sells the product's care and craft. In a portfolio of Sitecore Marketplace apps where polish drives adoption (operators choose tools that feel built-with-care), the current PRD-000 visual identity is a lost opportunity.

The marketing exploration (4 variants on 2026-05-13) showed what's possible: V1 Aurora (premium SaaS, off-system), V2 Neo Brutalism (anti-design rebellion), V3 Award Editorial (magazine-portfolio), V4 Blok Elevated (premium-grade composition *within* the design system). V4 was identified as the right answer — premium feel without leaving Blok, theme-aware, brand-consistent, ships entirely from existing semantic tokens.

PRD-002 takes V4 from speculative marketing exploration to actual shipped product. The non-trivial work is twofold: (1) faithfully translating V4's visual decisions across every PRD-000 surface, (2) reconciling V4's speculative content with PRD-000 reality (the V4 POC was built freehand and contains aspirational elements that don't map to real data).

## 3. Target Users

Same persona set as PRD-000; same workflows; same expectations of utility. The change is **perception**:

- **Marketers** authoring redirects from Pages — get a more polished workspace and a marketing-grade welcome on the IntroPage that signals quality.
- **Site administrators** managing redirect maps — same task flows, but the visual elevation makes daily-driver usage feel premium rather than purely utilitarian.
- **Migration operators** importing redirect inventories — wizard UX unchanged; visual chrome elevates.
- **Stakeholders / decision-makers** evaluating the product — the IntroPage hero + Dashboard Widget become marketing surfaces that pitch the product even before authoring starts.

## 4. Proposed Solution

A surface-level redesign of every PRD-000 visual surface using V4 patterns. Specifically:

- **IntroPage at `/`** — V4 marketing hero (gradient-text headline via `linear-gradient(135deg, var(--primary), color-mix(in oklch, var(--primary) 55%, var(--info)))` clipped to text; drifting `--primary`/`--info` plume backdrop on 28s loop; frosted-glass topnav; three feature cards routing to the three extension points; HahnSoloFooter inherited from the existing intro commit).
- **Full Page** — full V4 motion + composition treatment. Workspace toolbar with frosted-glass styling, hero headline on the workspace section, hover-lift cards with primary-tinted glow, animated background plumes, gradient-clip text on key headlines. All current CRUD orchestration (CollectionPicker, SitePicker, RedirectMapList, RedirectMapDetail, NewRedirectMapModal, DeleteMapConfirmModal, ImportRedirectMapModal) preserved 1:1 functionally; re-skinned in V4 style.
- **Context Panel** — quieter motion budget (hover lifts only; no plumes, no kinetic letter reveals). Frosted card surfaces; small marketing-grade headline at the top of the panel; existing exact-match Context Panel banner styling refined per V4. Workflow density preserved.
- **Dashboard Widget** — quieter motion budget. V4 hero stat presentation (large tabular-num Geist Sans 700 number) but **rendering real PRD-000 data** where it exists, **hardcoded mocks** where V4 added speculative metrics. Frosted tile surfaces. Mini-bar / sparkline as a mock.
- **Voice & copy** — hybrid: marketing-grade voice on hero / headline zones (IntroPage, Full Page workspace hero, Dashboard Widget headline). Utility-tool direct voice in dense workflow areas (table rows, modals, Context Panel body, inline forms).
- **Mock-data labeling** — single "Preview data" banner at the top of any surface containing hardcoded values. One statement per surface; doesn't repeat per element.
- **Theme** — both light + dark continue to work end-to-end. V4 validated this; PRD-002 carries forward.
- **Accessibility** — WCAG 2.1 AA from PRD-000 NFRs inherited; `prefers-reduced-motion` respect everywhere (V4 already implements; structural test should enforce).
- **HahnSoloFooter** carries forward across every route per recent intro commit.

## 5. Value Proposition

- **Marketing-grade impression** — operators and stakeholders see the product as more carefully-built than its utility-tool predecessor.
- **Brand consistency** — by composing from Blok semantic tokens (no invented hex, no off-system typefaces), the redesign is brand-coherent with the rest of the Sitecore Marketplace surface area.
- **Future-proofs the analytics story** — by adopting V4 patterns that show hero stats / sparklines / top-hit lists with hardcoded mocks, the visual chassis is ready when the data-plumbing PRD lands. The follow-on PRD becomes "wire real data into pre-existing visual slots" rather than "design + build + wire from scratch."
- **Risk-bounded scope** — pure visual re-skin means no functional regressions; smoke gates carry from PRD-000; ship is fast because the heavy plumbing (SDK wrappers, schema, CRUD logic) stays untouched.

## 7. Decisions

Captured from operator answers during this interrogation.

- **D1 — Pure visual / surface re-skin.** Functional surface stays 1:1 with PRD-000. UX improvements discovered along the way → follow-on PRDs, not in-scope creep.
- **D2 — V4 redesigned in FULL visual fidelity** including currently-speculative elements (hero stat, sparkline, top-hit list, 4th stat tile). Where V4 elements have no real data source in PRD-000, **hardcoded mock values render in those slots**. Each mock is tracked; the **follow-on "data plumbing" PRD** wires real data into the pre-built visual chassis.
- **D3 — Mixed motion budget.** Full V4 motion (drifting plumes, gradient text, kinetic letter reveals, hover lifts, count-ups) on IntroPage `/` + Full Page. Quieter on Context Panel + Dashboard Widget (hover lifts only; no plumes, no kinetic reveals).
- **D4 — Mock-data labeling via single per-surface "Preview" banner.** One statement at the top of any surface that contains hardcoded values; doesn't repeat per element. Operator-visible; honest; bounded clutter.
- **D5 — Hybrid voice.** Marketing-grade voice on hero / headline zones (IntroPage, Full Page workspace hero, Dashboard Widget headline). Utility-tool direct voice in dense workflow areas (table rows, modals, Context Panel body, inline forms).
- **D-Blok-Discipline** — Strict adherence to Blok Nova token system. Zero invented hex outside the existing `theme.css` block. All gradients composed via `color-mix(in oklch, var(--primary), var(--info))` or equivalent token expressions. Geist Sans + Geist Mono are the only typefaces. Component anatomy stays Blok.
- **D-Theme-Modes** — Both light + dark mode must work end-to-end. V4 POC already validates this.
- **D-Reduced-Motion** — `prefers-reduced-motion: reduce` respected everywhere. Disables plumes, kinetic reveals, count-ups. Static fallbacks remain visually pleasant.
- **D-Footer** — HahnSoloFooter from intro commit `76a6507` carries forward across every route.

## 9. High-Level Technical Considerations

Concept-level technical concerns surfaced during exploration. Detailed architecture happens in `/architect`.

- **Blok token discipline.** Every color / gradient / shadow / surface effect must compose from existing semantic tokens. `color-mix(in oklch, var(--primary), var(--info))` is the canonical pattern for gradient composition. A new structural test should assert no `#` hex literals outside `theme.css`.
- **`color-mix(in oklch, ...)` browser support.** Excellent in evergreen browsers; modest fallback story for older browsers (use a solid Blok token as fallback). To confirm at `/architect`.
- **Motion budget per surface.** IntroPage + Full Page get full V4 treatment. Context Panel + Dashboard Widget get hover lifts only. The boundary should be enforced by CSS scoping (motion utilities only loaded where appropriate) or by JS gating.
- **`prefers-reduced-motion` respect.** Existing PRD-000 components may have motion that wasn't audited for reduced-motion. PRD-002 should pass a structural guard that every `@keyframes` block has a `@media (prefers-reduced-motion: reduce)` override OR is in a class explicitly suppressed by the reduced-motion query.
- **Theme-aware gradients.** Gradients composed from tokens (`var(--primary)`, `var(--info)`) automatically theme-switch. No separate light/dark gradient definitions needed.
- **Hardcoded mock-data architecture.** Each mocked value lives in a single constants file (e.g. `site/lib/mocks/preview-data.ts`) so the follow-on data-plumbing PRD has one canonical location to swap. The "Preview data" banner reads from a per-surface flag indicating "this surface uses mocks" — when the data-plumbing PRD lands, the flags flip and the banner disappears.
- **Marketing voice copy.** The hero copy lives in a single content file (e.g. `site/content/marketing.ts` or inline in component files) — easy to iterate on without touching component logic.
- **Visual smoke gate (host_frame_smoke).** V4 POC at `pocs/poc-marketing-v4-blok-elevated/` is the new canonical visual reference. At `/test`, the 5-axis Playwright comparison (layout, typography, color, component anatomy, state fidelity) compares the live host-frame render against V4. PRD-002 ships a new POC under `pocs/poc-v1-prd002/` that refines V4 with verified feature mapping (decided at `/architect`).
- **Iframe constraints.** Mode A continues; same Marketplace SDK envelope; no changes to extension-point registration.
- **HahnSoloFooter rendering across routes.** Already in place from intro commit. PRD-002 verifies the footer renders correctly under V4 surface treatments (no z-index conflicts with drifting plumes; not occluded by frosted topnav scroll).
- **Backdrop-filter performance on iframe.** Frosted-glass effects via `backdrop-filter: blur(...)` are slightly expensive. Inside an iframe with parent rendering, this could compound. Worth measuring at `/test`.
- **Test fixture continuity.** Existing fixtures in `site/tests/fixtures/graphql/` (PRD-000) stay valid because no SDK changes are made.
- **Bundle size impact.** Additional CSS for V4 styling + a marketing-content file may increase bundle ~5-10 KB. Acceptable; Mode A apps don't ship with strict bundle budgets.

## 10. Risks

- **R-Mock-Data-Mistrust** — Operators see hardcoded numbers and think they're real, then complain when they don't update. Mitigation: D4 "Preview data" banner per surface. Risk reduces to: did we banner every surface that contains mocks? Structural test could assert pairing.
- **R-Scope-Creep-Functional-Improvements** — During implementation, developer notices a UX gap V4 reveals (e.g. a better empty state, a cleaner empty-version flow) and is tempted to fix it. Mitigation: D1 strict re-skin rule; any new affordance becomes a follow-on PRD entry; structural test or code-review gate flags new affordances introduced in PRD-002 commits.
- **R-Performance-Inside-Iframe** — Drifting plumes + frosted-glass + count-ups + kinetic reveals all run inside the iframe. Real-tenant smoke could surface performance issues invisible in the standalone POC. Mitigation: profile at `/test` smoke; reduce motion intensity if needed.
- **R-Reduced-Motion-Audit-Cost** — Auditing every PRD-000 component for reduced-motion respect may surface that some existing motion wasn't gated. Mitigation: structural guard for `@keyframes`; explicit audit task in `/task-breakdown`.
- **R-V4-Feature-Reconciliation-Discovery** — At `/create-prd` critical review, PM may find that more V4 elements are speculative than expected, expanding the future "data plumbing" PRD scope. Mitigation: classify explicitly during /create-prd; the bigger the future PRD becomes, the more rigorously D2's "hardcoded mocks + future wiring" tradeoff is interrogated.
- **R-Theme-Token-Drift** — Implementation may introduce `#` hex values silently. Mitigation: new structural test asserting no hex literals outside `theme.css`. Required in `/task-breakdown` § 4c.
- **R-Browser-Backdrop-Fallback-Ugliness** — Browsers without `backdrop-filter` show solid backgrounds, which break the frosted layered look. Mitigation: explicit fallback styling (solid `color-mix(in oklch, var(--background) 95%, var(--primary) 5%)` or equivalent) tested with `@supports not (backdrop-filter: blur(1px))`.
- **R-Visual-Regressions-In-PRD-000-Functionality** — Re-skinning every component could subtly break an existing modal / form / state. Mitigation: every existing Vitest UI test re-runs as part of PRD-002 ship; if a test fails because the new V4 styling broke an assertion, fix the assertion to the new design intent (not the other way around — keep tests aligned to design contract).
- **R-Mock-To-Real-Migration-Friction** — When data-plumbing PRD wires real data into the V4 visual slots, the slot designs must accommodate real-range values (e.g. hero stat showing 9-digit numbers when V4 mock is 5 digits). Mitigation: PRD-002 mock values use realistic-range numbers; data-plumbing PRD verifies range assumptions in advance.
