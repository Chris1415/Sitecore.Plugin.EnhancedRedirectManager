## 1. Architecture Overview

**This is a feature-scoped architecture delta, not a redesign.** PRD-000 shipped 2026-05-12 and is live; PRD-001 was cancelled (ADR-0023). PRD-002 is a **pure presentation-layer redesign of three already-shipped extension-point routes** (`/full-page`, `/context-panel`, `/dashboard-widget`) plus **one client-side interaction-pattern change** (Context Panel modal → inline quick-add per ADR-0026 — US-2 evolves into US-R5).

**Zero new SDK surfaces. Zero new GraphQL mutations. Zero new GraphQL fields. Zero backend work. Zero Cloud Portal registration changes. Zero Sitecore data-model changes.** Everything that crosses the iframe → portal → XM Cloud boundary in PRD-002 is bytes already shipped in PRD-000. The architectural delta is therefore narrow and load-bearing only at the presentation layer:

1. **A new CSS module** (`site/styles/elevated.css`) carrying V4 Blok Elevated tunable values (plumes, glass, hover-glow, hero clamp) as **one CSS variable set** — single source so R-13 token drift cannot happen.
2. **A new motion CSS module** (`site/styles/elevated-plumes.css`) holding the drifting `--primary`/`--info` plume utilities and kinetic-letter-reveal keyframes — imported **only** by Full Page route source files. Structural guard enforces (ADR-0027 § Enforcement).
3. **A new constants module** (`site/lib/mocks/preview-data.ts`) holding `PREVIEW_DATA` and `PREVIEW_DATA_ACTIVE` — the single swap point for the follow-on data-plumbing PRD (ADR-0025).
4. **A new presentation component** (`site/components/ui/preview-data-banner.tsx`) — single banner mounted at the top of Full Page and Dashboard Widget when their `PREVIEW_DATA_ACTIVE` flag is `true` (ADR-0025).
5. **The Context Panel `AddRedirectModal` component is removed** (ADR-0026 Option A — see § 11 Architectural Decision 1). The new inline quick-add form lives at the top of `ContextPanel.tsx` and calls the existing `createRedirectMap` / `updateRedirectMap` wrappers without parameter change.
6. **Five new structural guards extend `tests/structural/structural-guards.test.ts`** in place (no-hex, no-Active/Draft, @keyframes-reduced-motion pairing, banner ↔ `data-preview-mock` pairing, no-language-count, plume-CSS-import-boundary).

Everything else — `MarketplaceProvider`, the 7 wrappers in `lib/sdk/*`, the 3 extension-point routes, the root `notFound()` policy, the GraphQL envelope contract (`xmc.authoring.graphql` mutate / body INSIDE `params` / DOUBLE `.data.data` unwrap, verified 2026-05-11), the RedirectType enum (3 values, verified 2026-05-11), the parser/serializer modules, the import diff, the structural test harness — **carries from PRD-000 unchanged**.

**What this enables:** PRD-002 ships through CSS + one interaction-pattern change. No SDK re-capture, no real-tenant fixture session, no Auth0 work, no infrastructure work. The smoke surface narrows: m1 (registration), m4 (import/export) and most of m3 (CRUD round-trip) are continuity verifications against an already-shipped contract; only m2 (host-frame visual smoke against the refined POC) and m5 (live walkthrough of the new inline quick-add) are genuinely new gates.

**What this constrains:** the relaxed redesign rule (ADR-0024) is bounded. V4 UX evolutions that V4 already shows are in scope; "while we're in here, let's also …" is out of scope and becomes a follow-on PRD. The R-12 test-rework cascade is real and quantifiable (§ 9) — but it stays inside `tests/`, never bleeding into the SDK layer.

**Webfont loading strategy at hero scale — architect's review:**

PRD-000 ships `next/font/google` with `Geist` + `Geist_Mono`, default `display: 'swap'` (see `site/app/layout.tsx` per `next/font` convention). At hero clamp (48-96px Geist Sans 700), `display: 'swap'` does produce a brief FOUT (system font → Geist swap) on cold load. Three options:

- **(A) Leave it.** `swap` is a single-frame flash on hero text; modern hardware renders the swap in <100ms; iframe operators see this once per cold load.
- **(B) Switch to `display: 'optional'`.** Removes the swap by rejecting Geist if it hasn't loaded in ~100ms — but then a slow-network operator sees the entire hero in fallback. Worse outcome on the target operator profile (corporate networks behind portals).

**Architect's decision: option (A) — leave it.** PRD-000's existing strategy is sufficient because `next/font` already preloads the layout-referenced families. The hero clamp does not change the loading characteristic; only the *rendered size* changes. Smoke gate m5 verifies (operator confirms perceived polish on cold load). **No webfont strategy change in PRD-002.** Captured as OQ-A4 in § 10 in case operator subjective experience disagrees.

### 9.1 Tradeoffs explicitly accepted

- **Single `elevated.css` + `elevated-plumes.css` split, not per-route CSS modules.** A per-route CSS-Modules approach would isolate V4 styling but force class-name churn (`.fpHeroText` vs `.cpHeroText` etc.) without gain. Two files are enough: one site-wide variable + utility surface, one Full-Page-only motion surface. The motion split is structurally enforceable; finer granularity is not load-bearing.
- **Vanilla CSS + JS for animations, not a motion library.** NFR-R5 bundle budget + R-3 iframe perf concern: a 5kb gzipped library to render ~30 LOC of count-up + ~20 LOC of letter-split is the wrong trade. Vanilla wins.
- **`AddRedirectModal` removed entirely, not retained as fallback** (§ 11 Decision 1 below). Cleaner component graph; "create new map with full flag control" moves to the Full Page workspace where it belongs.
- **No new analytics, no new telemetry, no new operational surface.** PRD-000's stance (no Sentry, no LogRocket, no custom telemetry) carries. Mock data + smoke gates remain the only operational signals.

### 11.1 Architectural decisions made at this /architect stage

**Architecture Decision 1 — ADR-0026 Option A is the chosen path: remove `AddRedirectModal` entirely.**

PRD-002 / ADR-0026 left this open with a recommendation toward Option A. Architect adopts the recommendation and locks it.

- **What this enables:** one fewer surface to test (R-12 partial mitigation); one fewer component file (`AddRedirectModal.tsx`) — cleaner component graph; Context Panel renders without a button → modal indirection; the "Create new map with full flag control" path lives at the Full Page workspace where the operator has room to think (modal there is `NewRedirectMapModal`, unchanged); inline form is faster (2 clicks vs 4).
- **What this costs:** `AddRedirectModal.tsx` is deleted (file removal in T6 per R-12 tranche plan). Existing Vitest tests against `AddRedirectModal` get re-pointed at `QuickRedirectForm` (T7 test-rework tranche). The "Create new map with full flag control on the Context Panel" path is removed — operators wanting that path open the Full Page workspace, click "Add new map", and use `NewRedirectMapModal` there (this is the unusual case, not the common one).
- **What this makes hard:** if a future PRD discovers operators actually wanted the full-flag-control path on the Context Panel after all, restoring it requires resurrecting `AddRedirectModal` (or a minimal new modal). Acceptable: the empirical signal that "operators wanted full flag control on the Context Panel" has not appeared from PRD-000 smoke.

**ADR candidate:** **ADR-0028 — Context Panel inline quick-add removes `AddRedirectModal` (Option A locked).** Amends ADR-0026 (which had Options A and B both open). The new ADR records the final commitment: Option A, with rationale + the "Create new map with full flag control via Full Page workspace" fallback path.

**Architecture Decision 2 — Context Panel inline form map-selection model: "most-recently-updated matching map by default" + RedirectType select is disabled-with-display-only when adding to an existing map.**

Resolves PRD OQ-9. The decision is two parts:

- **Default match selection:** when `matchedGroups.length >= 1`, the inline form adds to the **most-recently-updated** matched map by default (highest `updatedAt`). When `matchedGroups.length > 1`, an affordance lets the operator switch — chip cluster vs. dropdown is a UI Designer decision (architectural commitment: *some* affordance exists; visual choice is the next stage).
- **RedirectType select semantics:** in the create-new path (`matchedGroups.length === 0`), the form's select is enabled and the value is the map's `redirectType` per the schema. In the add-to-existing path, the select is **disabled and pre-set to the chosen map's `redirectType`** with a small "uses existing map's redirect type" hint — adding a mapping does not change the parent map's `redirectType` (the field is per-Item, not per-mapping). This eliminates the AC-R2.4 ambiguity ("the form's RedirectType select then governs only the 'create new map' path").

**ADR candidate:** **ADR-0029 — Context Panel `QuickRedirectForm` map-selection + RedirectType semantics.** Records the most-recent-updated default + the disabled-select-on-existing-map rule + the auto-name pattern (`${pageSlug}-redirects` lower-kebab-case from `pageInfo.url`).

**Architecture Decision 3 — Full Page hero banner CTAs (View activity / Publish all) are DECORATIVE in PRD-002.**

Resolves PRD OQ-10. Neither "View activity" nor "Publish all" maps to existing functionality in PRD-000 (no activity log; no bulk-publish surface). Wiring them in PRD-002 would expand scope. Three options were considered:

- (a) **Wire them.** Requires designing an activity log (mock or real) and a bulk-publish flow. Out of PRD-002 scope.
- (b) **Drop them.** Loses V4's marketing chrome.
- (c) **Ship as decorative.** Buttons render with V4 chrome but `onClick` is a no-op (or shows a "Coming in a follow-on release" toast).

**Architect's call: (c) — ship as decorative.** Both buttons are part of the V4 marketing chrome; removing them weakens the hero. A "coming soon"-style toast is cheap, honest, and gives the operator feedback. The follow-on data-plumbing PRD (PRD-003 candidate) is the natural place to wire them.

**ADR candidate:** **ADR-0030 — Full Page hero CTAs are decorative in PRD-002 (toast-only).** Records the deferral so the next PRD picks them up; documents the toast copy (UI Designer iterates).

**Architecture Decision 4 — z-index hierarchy commitment.**

PRD R-10 + OQ-A3 needed an explicit hierarchy. Architect locks:

| Layer | z-index | Stacking note |
|---|---|---|
| `.fp-plume-backdrop` (Full Page plumes) | `-1` (per V4 POC line 36) | Sits behind every content layer including default `z-index: 0`. |
| Page content (default) | `0` (default) | Default flow. |
| Frosted-glass topbar (Full Page sticky topbar) | `40` (per V4 POC line 83) | Sits above page content, below the footer. |
| **HahnSoloFooter** (carry from PRD-000) | **`50`** | Right-bottom corner. Always visible. |
| Modal overlay (`.blok-overlay` / `NewRedirectMapModal`, `DeleteMapConfirmModal`, `ImportRedirectMapModal`, `EditMapSettingsModal`) | `100` (per theme.css line 731) | Sits above the footer (correct — modal interactions should obscure the footer briefly). |

Plume layer creates a new stacking context (`position: fixed; z-index: -1`) — important because nested transforms inside content do not break the stacking promise. Frosted topbar's `backdrop-filter` creates its own stacking context regardless of z-index; operator-visible result is fine because `z-index: 40` keeps it below the footer.

**Not a separate ADR** — this is a tactical commitment that lives in this architecture file + flows into the elevated.css / elevated-plumes.css implementation. The plume-CSS-import-boundary structural guard plus the footer z-index assertion (already in PRD-000 structural test) suffice.
