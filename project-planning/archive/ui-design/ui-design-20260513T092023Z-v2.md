> **Direction name:** **Coverage Atlas**
>
> **One-sentence thesis:** The map list is the multilingual radar — every row exposes a horizontal **language-coverage chip strip** so the operator scans the list and immediately sees which maps have filled / empty / missing versions across every site language; the picker is demoted to a secondary tool because the list itself answers "where is content?" before any switch is needed.

---

## 1. Design context & assumptions

### Audience

- **Marketers** authoring redirects in a non-`en` working language. Coverage Atlas gives them an at-a-glance view of which maps are still in `en`-only and need a localized version — they decide what to translate next *from the list*, not from a hunt-and-peck switch.
- **Site administrators** managing multilingual Sitecore sites (e.g. `/de/produkte` ↔ `/en/products`). They need to spot coverage gaps across the whole site without serially flipping the picker through every language.
- **Migration operators** importing multi-language redirect inventories. They need a preview that shows per-map / per-language status (create / overwrite / skip) at high information density without becoming a wall of text — the expandable-per-map import pattern is the answer.

### Goals and success criteria

- **G1** — One look at the map list answers "which maps still need translation in language X?" without changing the picker.
- **G2** — Switching language never feels like a tax: the secondary topbar picker is one click; the per-map inline mini-picker is half a click; the **most common case (peek at one map's de version)** is zero picker clicks because the multi-chip strip already telegraphed it.
- **G3** — Dashboard tiles answer two questions at once: "what's the state of my **current** language?" and "what's the state of the **whole site**?" without forcing a toggle.
- **G4** — Delete confirmation never relies on operator literacy of two-line radio labels. The card pattern *shows* what survives (and what doesn't) via the same chip vocabulary used everywhere else.
- **G5** — Create-version flow is visibly progressive: stepper signals "you're not committing yet" up to step 3, so operators with ambivalent intent ("maybe I want copy-from, maybe empty") can navigate without dead-ends.
- **G6** — WCAG 2.1 AA holds for every new affordance, including the chip strip (which is information-dense and at risk of becoming a color-only signal — explicit a11y patterns are part of the spec).

### Constraints

- **Blok is the UI layer** (mandatory — Marketplace platform contract). Every named component below is a real Blok primitive (`@blok/<name>`) or a composition of Blok primitives. No invented components.
- **Real Blok semantic tokens only**: `--primary`, `--muted`, `--destructive`, `--success`, `--warning`, `--info`, `--border`, `--ring`, `--background`, `--foreground`, `--card`, `--card-foreground`, etc. (Per PRD-000's `theme.css` token set; carries forward identical.) No invented hex.
- **WCAG 2.1 AA** on every new surface (NFR-A1).
- **Iframe-bound viewport** (Marketplace extension points): Full Page 1024-1920w × 600-1080h; Dashboard 300-800w × 200-400h; Context Panel 320-400w. Layout must hold at the lower bounds.
- **PRD-000 visual continuity** — same Blok foundation, same theme, same overall layout grammar. The new visual rhythm (per-row multi-chip strip) is achieved by **composing existing Blok primitives** (`@blok/badge` strip + `@blok/tooltip`) — not by introducing new components.
- **No color-only state encoding.** Coverage Atlas leans heavily on color chips, so every chip carries a shape/glyph signal too. Monochrome glyphs (`✕` `✓` `–`) per the Blok theming skill's color-emoji-poison rule; never bare emoji codepoints.
- **Geist Sans + Geist Mono** (continuity with PRD-000 `theme.css`).

### Reference / inspiration

- **Linear's issue list "labels strip"** — colored chips on every row, scan-friendly, never obscures the row's primary identity. Adapted here to language-coverage chips.
- **GitHub's repo "Languages" bar** — horizontal bar showing language proportions. Coverage Atlas inverts this (per-row, not per-repo) and uses discrete chips instead of a continuous bar.
- **Stripe Dashboard's twin-stat pattern** — "current period" + "lifetime" tiles side-by-side, never forcing a toggle. Twin-tile Dashboard derives from this.
- **macOS / Notion-style segmented stepper** for the create-version modal — progress visible, can navigate back.
- **Linear's destructive-action card pattern** (two cards, each visually showing the consequence) for the delete modal.

### Answers to clarifying questions

### Assumptions made

- **A-V2-1** — The chip strip's typical max width on a list row is ~280px (after the row title + meta + redirect-type badge). At 7 site languages the strip is comfortable; at 12+ the design degrades to a "summary chip" with a count + tooltip, per § 4 Component Spec below.
- **A-V2-2** — Most real sites have 2-5 languages (operator-confirmed). The strip is designed for that range; 8+ is a graceful-degradation case, not the primary target.
- **A-V2-3** — Site languages enumeration returns a stable ordering (site-default first, then ISO-alphabetical). The chip strip uses that order; flipping the picker does not re-order chips.
- **A-V2-4** — The Dashboard widget's "Maps in `[current language]`" tile and "Maps total" tile can sit side-by-side at 400px+ width. At <400px (rare for Dashboard) they stack; at 600px+ they sit side-by-side comfortably.
- **A-V2-5** — The detail-pane inline mini-language-switcher (chip pills, not a select) is bi-directional with the topbar picker — clicking a chip updates both immediately. Per ADR-0017 shared state, this is a single source of truth, not two synchronized stores.
- **A-V2-6** — Stepper modal is 3 steps for copy-from; 2 steps for empty (skip the source-language step). The progress bar shows the right denominator on each path.

## 2. Design direction

### Visual direction

**Coverage Atlas** treats the **list** as the dashboard. The thesis: in a multilingual app the answer to "what should I work on?" is *spatial* — operators want to see coverage holes, not navigate to find them. So every map row shows its language coverage as a horizontal chip strip — a tiny per-row multilingual radar.

The chip strip uses three states per language, distinguishable by **both color and shape**:

- **Filled chip** — solid `--primary-background` fill, `--primary` border, language code text in `--primary` color, no glyph → *"has content"*.
- **Dashed-outline chip** — transparent fill, dashed `--border-a11y` border, language code in `--muted-foreground`, no glyph → *"empty version exists"* (the count-≥-1-but-`UrlMapping`-empty state from US-5).
- **Missing-glyph chip** — transparent fill, dotted `--border` border, language code in `--placeholder` color, monochrome `–` glyph (U+2013) prefix → *"no version exists"* (US-4).

The chip vocabulary is the **visual language of V2** — it appears in the map list, in the delete confirmation cards (showing what survives vs what doesn't), in the import preview rows (showing tenant-vs-bundle coverage), in the Dashboard's "Languages with content" tile, in the create-version source picker (showing which languages are filterable), and in the detail-pane inline switcher. **One pattern, eight surfaces** — operators learn it once.

The **picker** is demoted: it sits in the topbar at smaller visual weight than V1 would treat it (`@blok/select` height `1.75rem` not `2rem`; subdued `--muted-foreground` label; tucked right of the breadcrumb, not center-stage). The picker is a tool. The list is the radar.

**Why this works for Sitecore Marketplace iframes:** the iframe is width-constrained but height-generous. A chip strip per row is dense vertically (no chip strip needed = saved space) but readable horizontally because rows are already wide. Twin-tile Dashboard sits in 600-800px Dashboard widgets; below that, the tiles stack and remain readable because each tile is self-contained.

### Layout and structure

**Full Page (`/full-page`, viewport 1024-1920w × 600-1080h):**

- **Topbar (`@blok/topbar` shell pattern)** — breadcrumb left; secondary-weight `@blok/select` language picker right of breadcrumb, labelled "Lang:" inline at `--text-xs`; right cluster `Import` / `Export` / `New map` buttons (carries from PRD-000). The picker is intentionally not a focal point — operators rarely need to touch it because the multi-chip strip already tells them where content is.
- **Left rail** — Collection + Site selects (carry from PRD-000), then **map list with per-row multi-chip strip**. Each row: title + display name + meta line + redirect-type badge + chip strip. Chip strip uses `@blok/badge` primitives composed inline. Row height grows from ~64px (PRD-000) to ~72px to accommodate the chip strip below the title.
- **Main pane** — when a map is selected: header (display name + discoverable "Edit `[language]` display name" button + **inline mini language-switcher chip pills**) + map detail (mappings list, flags, etc.). The inline mini-switcher is the V2 differentiator: chip pills horizontal, current selected with `--primary-background` fill, others dashed/dotted matching list semantics. Click to switch language without touching the topbar picker — both stay in sync per ADR-0017.

**Dashboard Widget (`/dashboard-widget`, viewport 300-800w × 200-400h):**

- **Twin tiles per stat**:
  - Top row: `Maps in [current language]` + `Maps total` (two `@blok/card` primitives side-by-side at ≥600w; stacked at <600w).
  - Second row: `Mappings in [current language]` + `Mappings total`.
  - Third row (single, full-width): **`Languages with content` tile** — a horizontal chip strip showing every site language with its state (filled / empty / missing) aggregated across all maps in the site. This tile is the "everywhere else's chip strip" zoomed out to site level.
- No picker UI on the Dashboard widget itself — it inherits the picker state read-only from `localStorage` via the shared `use-language-picker` hook. The current-language tile labels update reactively.

**Context Panel (`/context-panel`, viewport 320-400w, variable height):**

- **No structural change** beyond the banner. The banner becomes a Pages-language-aware **info card** (PagesAwareBanner, `@blok/alert--info` styling with `--border` emphasis): copy varies per AC-13.2.
- The Context Panel does NOT show coverage chips — it's iframe-narrow, mapper-focused, and the chip vocabulary belongs to authoring surfaces. Carry from PRD-000 Context Panel design.

**Surfaces unchanged in layout grammar from PRD-000:** topbar shell, modal overlay, toast positioning, focus-ring contract, theme-token bindings, dark/light flip. V2 inherits all of these.

## 3. Key screens and flows

### S1 — Full Page home (map list view with multi-chip strip) `[V2 HALLMARK]`

**Purpose:** the landing surface. Operator sees the list and immediately knows the coverage state of every map across every language.

**Layout:**

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│ [Breadcrumb: SneakerCo / MainSite / Marketing campaigns]  Lang: [de ▾]  [Import][Export][New map]│
├────────────────────────────┬─────────────────────────────────────────────────────────┤
│ Collection: SneakerCo  ▾   │  Marketing campaigns                                    │
│ Site:       MainSite    ▾  │  ────────────────────────────────────────────────────   │
│ ────────────────────────── │  Display name (de): SEO-Weiterleitungen                 │
│ Redirect maps    12        │  [↗ Edit de display name]                               │
│ ────────────────────────── │                                                          │
│ ▸ Marketing campaigns      │  Languages: [en●][de●][fr◌][es–]                        │
│   12 mappings · 3 days ago │                                                          │
│   [en●][de●][fr◌][es–]     │  Mappings (12):                                          │
│   [301]                    │   /summer-2025 → /sale-summer                            │
│                            │   /sneakers-2024 → /catalog/2024  …                     │
│   Legacy product URLs      │                                                          │
│   28 mappings · 2 wk ago   │                                                          │
│   [en●][de◌][fr–][es–]     │                                                          │
│   [302]                    │                                                          │
│                            │                                                          │
│   A/B test variants        │                                                          │
│   6 mappings · 1 mo ago    │                                                          │
│   [en●][de–][fr–][es–]     │                                                          │
│   [Server]                 │                                                          │
└────────────────────────────┴─────────────────────────────────────────────────────────┘

Chip legend (rendered as a one-line caption above the list, dismissible):
  [en●] filled (has content)  [de◌] empty version  [es–] no version
```

**Chip strip mechanics:**

- Chip width: ~32px each, gap 4px. With ISO codes (`en`, `de`, `fr-CH`) chips are size-comfortable.
- Order: site-default language first; rest in `Sites.Site.languages` order (ADR-0020) — stable across picker changes.
- Color states (light mode):
  - **Filled:** `background: var(--primary-background)`, `border: 1px solid var(--primary)`, `color: var(--primary)`. Solid dot glyph (`●` U+25CF, monochrome) optional after the lang code.
  - **Empty version:** `background: transparent`, `border: 1px dashed var(--border-a11y)`, `color: var(--muted-foreground)`. Hollow circle glyph (`◌` U+25CC) after lang code.
  - **No version:** `background: transparent`, `border: 1px dotted var(--border)`, `color: var(--placeholder)`. En-dash glyph (`–` U+2013) after lang code.
- Hover on chip: `@blok/tooltip` shows full state description ("`de` version exists, 12 mappings" / "`fr` version exists, empty" / "`es` no version yet — click row to create").
- Click on chip: **does not switch picker** (rows are still the unit of selection). Click on the **row** does the selection; the chip is informational. *(Alternative considered: click chip to switch picker; rejected because it would conflict with row-click selection semantics on mobile/touch and add ambiguity.)*
- Graceful degradation at 8+ languages: chip strip becomes a summary `[4/8 ●]` chip with a tooltip listing all language states, OR a `@blok/popover` revealing the full strip. Decision: prefer popover (preserves the strip's pattern).

**Top-right picker:** smaller `@blok/select` (height `1.75rem`, font `--text-xs`); label "Lang:" `--muted-foreground`. Default selection: persisted via ADR-0017 default-resolution-chain.

**Empty state (no maps):** carries PRD-000 empty state; copy unchanged. Chip strip is not rendered in empty state.

### S2 — Map detail with inline mini-picker `[V2 HALLMARK]`

**Purpose:** detail view of a selected map; operator can switch language without touching the topbar picker.

**Layout (right pane only):**

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│ Marketing campaigns                                                                   │
│ ─────────────────────────────────────                                                │
│ Display name (de): SEO-Weiterleitungen                                                │
│ [↗ Edit de display name]                                                              │
│                                                                                       │
│ Language version:  ┌────┬────┬────┬────┐                                              │
│                    │ en●│ de●│ fr◌│ es–│  ← chip-pill mini-switcher (radiogroup)     │
│                    └────┴────┴────┴────┘                                              │
│                          ↑ current                                                     │
│                                                                                       │
│ Mappings (12):                                                                        │
│   ┌───────────────────────────────────────────────────────────────────┐               │
│   │ /summer-2025          →  /sale-summer            [Edit] [Delete] │               │
│   │ /sneakers-2024        →  /catalog/2024           [Edit] [Delete] │               │
│   │ ...                                                              │               │
│   └───────────────────────────────────────────────────────────────────┘               │
│                                                                                       │
│ [Add mapping]                                                                         │
│                                                                                       │
│ Shared flags (apply to all languages):                                                │
│   [✓] Include virtual folder                                                          │
│   [ ] Preserve query string                                                           │
│   [ ] Preserve language                                                               │
│   Redirect type: [301 ▾]                                                              │
│                                                                                       │
│                              [Delete map ▾]   [Save changes]                          │
└──────────────────────────────────────────────────────────────────────────────────────┘
```

**Mini-switcher (chip pills as radiogroup):**

- Same chip vocabulary as the list strip — three states, same colors, same glyphs.
- Wrapped in `<div role="radiogroup" aria-label="Language version">` with each chip `<button role="radio" aria-checked="..." aria-label="...">`.
- Current selection has `--primary-background` fill + `--primary` border + `--primary` text (filled chip pattern reused as "selected" pattern).
- Selecting a chip:
  - **If language has a version (filled or empty):** updates picker state immediately (writes to `localStorage`; topbar picker reactively updates); re-fetches detail data; transient loading state per AC-1.3.
  - **If language has NO version (missing-glyph chip):** clicking opens the create-version stepper modal (S3) pre-seeded with that language as target. The picker does NOT switch yet — only switches after the create completes.
- AC-3.5 dirty-edit protection: when any field is dirty, all chips except current become `aria-disabled="true"` with tooltip "Save or discard your changes before switching languages." Visual treatment: chips dim to `opacity: 0.5` and cursor becomes `not-allowed`.
- Keyboard nav: Arrow Left / Right move focus across chips; Space / Enter selects; matches `radio-group` aria pattern.

**Display-name edit affordance** (below the title):

- A visible `@blok/button` variant `outline` size `sm` with text "Edit `[language]` display name" + a Lucide `Pencil` icon. **Always visible**; not hover-revealed.
- Click → inline editor replaces the display-name line: `@blok/input` (text) prefilled with current value (or placeholder `_(no display name for [language] — add one)_` per FR-11) + `@blok/button` "Save" + `@blok/button` variant `ghost` "Cancel".
- Enter on input commits (AC-9.4); Escape cancels; blur does NOT commit.
- On save: `@blok/sonner` toast "Saved display name for `[language]`."

### S3 — Create-version stepper modal `[V2 HALLMARK]`

**Purpose:** US-6 + US-7 — create empty version OR copy from another language. V2 uses a stepper to make the path choice and source-language pick feel deliberate.

**Stepper layout:**

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│ Create de version of "Marketing campaigns"                              [×]           │
│ ─────────────────────────────────────────                                            │
│ ●───────○───────○                                                                     │
│ 1. Path     2. Source     3. Confirm                                                  │
│                                                                                       │
│ Step 1 of 3 — Choose how to start                                                     │
│                                                                                       │
│   ○ Empty version                                                                     │
│     Start with no mappings. Add them yourself after creation.                         │
│                                                                                       │
│   ○ Copy from another language                                                        │
│     Pre-fill mappings from an existing language version.                              │
│                                                                                       │
│ ─────────────────────────────────────────                                            │
│                                                       [Cancel]       [Next →]         │
└──────────────────────────────────────────────────────────────────────────────────────┘
```

**Flow:**

- **Step 1 — Path** (always present): radio choice between Empty / Copy-from. No default selection (consistency with delete modal's no-default principle, even though scope is less destructive here — the consistent pattern reduces operator surprise).
- **Step 2 — Source** (only present if Copy-from chosen; **skipped entirely if Empty**, jumping to Step 2/2 Confirm):
  - `@blok/combobox` listing **only languages with existing non-empty `UrlMapping` content** (ADR-0016 + AC-7.2). Each option includes a coverage chip showing the source language and the target language's current state (`[en● → de–]`).
  - UI hint (`@blok/alert` variant `info`): "Mappings will be copied from `[source]`. Flags and redirect type are shared across all languages and apply automatically." Wording adjusts per OQ-2 capture (PRD AC-7.4).
- **Step 3 — Confirm** (always present; numbered 2/2 if Empty, 3/3 if Copy-from):
  - Plain-language summary: "Create empty `de` version of *Marketing campaigns*." OR "Create `de` version of *Marketing campaigns* by copying 12 mappings from `en`."
  - Final action button: "Create" → executes; toast on success.

**Progress bar:** `@blok/progress` at the top of the modal, width animated between steps; reduced-motion respected.

**Stepper navigation:**

- "Next" button enabled only when the current step's required choice is made (strict no-default principle).
- "Back" appears on Step 2+; closes the modal at Step 1.
- Keyboard: Tab cycles through interactive elements; Enter on focused "Next" advances; Escape closes (with confirm-discard if any choice was made).

**Empty-version state (S3 truncated):** if operator picks "Empty version" at Step 1, the stepper indicator collapses from 3 dots to 2 dots, and Step 2 of 2 is Confirm. This is the key UX win of the stepper pattern: it makes the path's length **visible**, so operators don't feel they signed up for "a long flow" when they really only need two clicks.

**Partial-failure UX (Path B copy-from):** per ADR-0021, the modal's "Create" action shows progress text "Creating version… / Copying mappings… / Done". On failure-and-rollback-success: toast "Could not copy mappings; new version removed" + modal stays open at Step 3 with retry. On failure-and-rollback-failure: toast "Partially populated version detected" + modal closes + a banner appears in the detail view offering "Clean up partial version" (manual delete affordance) per AC-7.6.

### S4 — Delete card-pattern modal `[V2 HALLMARK]`

**Purpose:** ADR-0018 delete scope safety — present the two delete options as **visually-distinct cards** that show the consequence, not as plain radio labels.

**Layout:**

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│ Delete "Marketing campaigns"?                                           [×]           │
│ ─────────────────────────────────────────                                            │
│ This map has 3 language versions: en, de, fr.                                         │
│                                                                                       │
│ Choose what to delete:                                                                │
│                                                                                       │
│ ┌────────────────────────────────────┐  ┌────────────────────────────────────┐       │
│ │ ○                                  │  │ ○                                  │       │
│ │ Delete de version only             │  │ Delete entire map                  │       │
```

**Card mechanics:**

- Each card is a `@blok/card` wrapped in a clickable region (`<label>` containing a hidden `@blok/radio-group` input for semantic correctness — the visible card is the affordance, the radio is the a11y/state underlay).
- **Strict no-default** per ADR-0018: both cards start with the radio unselected; the "Delete" footer button is disabled until one is clicked.
- Selected card: `--primary` border (2px, replacing the default `--border` 1px), `--primary-background` fill at 50% mix, radio dot fills.
- Inside each card: the **post-deletion coverage chip strip** rendered with a new chip state — **deleted** chips use `--destructive-background` fill + `--destructive` border + `✕` monochrome glyph (U+2715, never `❌` U+274C per Blok theming color-poison rule).
- The coverage preview *is* the consequence — operator literally sees what survives.

**Single-version edge case (AC-8.6):** still two cards, both still unselected by default. Below the cards: informational `@blok/alert` variant `info` with text "This map has only one language version; both options remove all redirect data. Choose explicitly to confirm intent." The visual underscores: even though the cards depict the same outcome (`[en✕]` in both), the operator must pick deliberately.

**Hard-fail variant (`deleteItemVersion` absent — R-1 cut per ADR-0018):**

- Single card only ("Delete entire map and all language versions") with the post-delete chip strip showing `[en✕][de✕][fr✕]`.
- Modal title and helper copy adjust: "Single-version-delete is unavailable in this Authoring schema. Delete the entire map across all languages?"
- Strict no-default still applies: the single card starts unselected; Delete button enables on selection.

**Footer:** "Cancel" (`@blok/button` variant `outline`) + "Delete" (`@blok/button` variant `destructive`, disabled until a card is selected). Per AC-8.2.

**Toast on success:** named action — "Deleted `de` version of Marketing campaigns" or "Deleted Marketing campaigns and all language versions" (AC-8 + ADR-0018).

## 4. Component and pattern spec

### Component inventory

#### 4.1 Coverage chip strip `[V2 HALLMARK]`

- **Implementation:** N × `@blok/badge` composed inline, wrapped in `<div role="group" aria-label="Language coverage">`.
- **Per-chip element:** `@blok/badge` + monochrome glyph + ISO code text. Hovered: `@blok/tooltip` reveals detail.
- **Variants (4 states):**
  - **Filled** — has version with content. Styling: `bg-primary-background`, `border-primary`, `text-primary`, glyph `●` (U+25CF).
  - **Empty version** — has version, `UrlMapping` empty. Styling: `bg-transparent`, `border-dashed border-a11y`, `text-muted-foreground`, glyph `◌` (U+25CC).
  - **No version** — version count = 0. Styling: `bg-transparent`, `border-dotted border-border`, `text-placeholder`, glyph `–` (U+2013).
  - **Deleted preview** (S4 card modal only) — Styling: `bg-destructive-background`, `border-destructive`, `text-destructive-foreground`, glyph `✕` (U+2715).
- **Props (conceptual):** `languages: Array<{ iso: string; state: 'filled' | 'empty' | 'missing' | 'deleted' }>`, `compact?: boolean` (for narrow contexts).
- **States:** default, focus-visible (on chip, when keyboard-focused), disabled (during loading or AC-3.5 dirty-edit-protection in mini-switcher mode).
- **Accessibility:**
  - Each chip has `aria-label="<language name> — has content"` / `"empty version"` / `"no version"`. State is **conveyed via aria-label**, not color alone.
  - Glyphs (`●` `◌` `–` `✕`) are monochrome unicode → respect theme color. **Never `❌` U+274C / `✅` U+2705** per Blok color-emoji-poison rule.
  - Contrast: filled chip background-on-foreground is `--primary-background` on `--primary` text — passes AA at the type size used.
  - Tooltip is keyboard-reachable (focus on chip → tooltip via `:focus-visible` styling); not motion-required.
- **Reuse:** map list rows, detail-pane mini-switcher (with selected state added), import preview rows (tenant + bundle strips), delete card modal (post-delete preview), Dashboard "Languages with content" tile.

#### 4.2 Mini language-switcher (chip pills as radiogroup) `[V2 HALLMARK]`

- **Implementation:** wrap the coverage chip strip in `<div role="radiogroup" aria-label="Language version">`; each chip becomes `<button role="radio" aria-checked="...">`. Adds a fifth "selected" visual state (`--primary-background` fill + `--primary` border, regardless of underlying coverage state).
- **Variants:** selected, not-selected (filled / empty / missing variant from § 4.1).
- **Selection behavior:** click on "filled" or "empty" chip → switch picker. Click on "missing" chip → open S3 create-version stepper. Click on already-selected chip → no-op.
- **States:** default, hover, focus-visible, selected, disabled (AC-3.5 dirty-edit-protection).
- **Accessibility:**
  - Arrow Left / Right navigates between chips; Space / Enter selects.
  - Disabled state announces "Save or discard your changes before switching languages" via `aria-describedby` pointing to a visually-hidden helper text element.
  - Selected chip `aria-checked="true"`; others `aria-checked="false"`.

#### 4.3 Topbar language picker

- **Implementation:** `@blok/select` (option list of site-scoped languages; tenant-wide list appears as a separate option group with header "Other tenant languages" when fallback applies per ADR-0020).
- **Variants:** default, error (AC-1.6 — disabled appearance + Retry button beside it).
- **States:** default, hover, focus-visible, disabled (AC-3.5 dirty-edit-protection), error.
- **Visual weight:** smaller than V1 (height `1.75rem` not `2rem`; label `--text-xs --muted-foreground`).
- **Accessibility:**
  - `aria-label="Language"` per NFR-A5.
  - Disabled state has `aria-describedby` pointing to dirty-edit hint.
  - Error state has visible "Retry" button (separate component) for keyboard recovery.

#### 4.4 Display-name edit affordance

- **Implementation:** `@blok/button` variant `outline` size `sm` + Lucide `Pencil` icon + text "Edit `[language]` display name".
- **States:** default, hover, focus-visible, pressed (editor open).
- **Inline editor (when active):** `@blok/input` + `@blok/button` "Save" + `@blok/button` variant `ghost` "Cancel". Per AC-9.4: Enter commits; Escape cancels; blur does NOT commit.
- **Accessibility:**
  - Button has clear text label ("Edit `de` display name") — not icon-only.
  - Input has `aria-label="Display name in [language]"`.
  - Placeholder text per FR-11 ("_(no display name for de — add one)_") rendered as native `placeholder` attribute (semantically distinct per NFR-A5).

#### 4.5 Create-version stepper modal

- **Implementation:** `@blok/dialog` wrapper + `@blok/progress` for step indicator + step bodies. (Blok ships `@blok/stepper`; if available, prefer it; otherwise compose with `@blok/progress` + step-number badges.)
- **Variants:** Empty path (2 steps) / Copy-from path (3 steps).
- **States:**
  - Step 1: default, choice-made (Next enables).
  - Step 2 (Copy-from only): default, source-chosen (Next enables).
  - Step 3: default; "Create" button executes.
  - During execution: progress text replaces footer buttons; on success → close modal; on failure → revert to step 3 with error message + retry.
- **Accessibility:**
  - Modal is focus-trapped (`@blok/dialog` ships this by default).
  - Step indicator has `aria-label="Step N of M"` updated reactively.
  - Path radio group has standard `role="radiogroup"` semantics.
  - Source-language combobox is keyboard-navigable (Blok's `@blok/combobox` ships this).

#### 4.6 Delete card-pattern modal

- **Implementation:** `@blok/dialog` + two `@blok/card` instances + hidden `@blok/radio-group` for semantic state.
- **Variants:** normal (two cards), single-version-edge (two cards + info alert), hard-fail-cut (one card per R-1).
- **States:** default (both cards unselected), card A selected, card B selected, action in-flight.
- **Selection feedback:** selected card gets `--primary` 2px border + `--primary-background` 50%-mix fill; radio dot fills.
- **Accessibility:**
  - Each card wrapped in `<label>` containing the radio input — clicking anywhere on the card selects.
  - `role="radiogroup"` on the cards container; `aria-label="Delete scope"`.
  - Coverage chip strip inside each card has same a11y as § 4.1 (aria-labels per chip).
  - "Delete" button has `aria-disabled="true"` until a card is selected, with explanatory `aria-describedby`.

#### 4.7 Import preview row (expandable)

- **Implementation:** `@blok/collapsible` + collapsed-view inline layout (tenant strip + bundle strip + action picker) + expanded-view per-language diff table.
- **States:** collapsed, expanded, action-selected, disabled (during apply).
- **Accessibility:**
  - Chevron is `@blok/button` variant `ghost` size `icon` with `aria-expanded` + `aria-controls`.
  - Per-language diff table is a real `<table>` with column headers — screen-reader can announce row context.
  - Action radio group has standard `role="radiogroup"` semantics.

#### 4.8 Dashboard twin tiles

- **Implementation:** two `@blok/card` instances side-by-side (CSS grid `grid-template-columns: 1fr 1fr` at ≥600w; `1fr` at <600w).
- **Per-tile content:** label (`--text-xs --muted-foreground`), big number (`--font-mono --text-3xl`), sub-line (`--text-xs --muted-foreground`).
- **States:** default, loading (skeleton replaces number + sub-line), error (small error icon + retry).
- **Accessibility:**
  - Each tile has an `<h3>` heading for its label so screen-readers announce structure.
  - Numbers use `--font-mono` `font-variant-numeric: tabular-nums` for legibility (carries from PRD-000 `.mono` rule).

#### 4.9 Dashboard "Languages with content" tile

- **Implementation:** `@blok/card` with horizontal coverage chip strip (§ 4.1 component) + label + helper text.
- **Per-chip variant:** chips here use a slightly different visual — they include a fraction (`12/12`, `8/12`) to communicate aggregate coverage. State is determined by aggregate: all-12-filled → filled state; any-empty → empty state (with fraction making the partial-ness explicit); all-missing → missing state.
- **States:** default, loading (4 chip skeletons), error.
- **Cross-surface affordance:** chip click → navigate to Full Page with `?lang=<iso>` query param (or via shared `localStorage` write to picker key) → Full Page reads picker → renders filtered.

#### 4.10 Pages-aware Context Panel banner

- **Implementation:** `@blok/alert` variant `info` with elevated `--info` border (2px instead of default 1px).
- **Copy variants:** two strings per AC-13.2, conditioned on `pageInfo.language`.
- **Behavior:** non-dismissible. Icon `Info` from Lucide (currentColor).
- **States:** default (en or unavailable language), pages-language-aware (non-en).
- **Accessibility:** `role="status"` (live region; informational, not error → not `alert`); copy is real text, not icon-only.

#### 4.11 Carries from PRD-000 unchanged

- Topbar shell (`@blok/topbar`), Collection/Site picker selects, map detail mapping rows, Add Mapping / Delete Mapping buttons, Save Changes button, breadcrumb, redirect-type badge, Sonner toast positioning, focus-ring contract, theme tokens, dark/light flip, modal overlay treatment.

### Accessibility notes per component

- **All interactive elements** carry `:focus-visible` ring (`2px solid var(--ring)` + `outline-offset: 2px`) — never `outline: none` without paired `:focus-visible` rule (structural guard from PRD-000 carries).
- **Color is never the sole signal.** Every chip state combines color + border style (solid / dashed / dotted) + monochrome glyph + aria-label. WCAG 1.4.1 holds.
- **All operator-typed text** (`__Display name`, mappings) rendered via Blok primitives that auto-escape (React default + Blok composition). No `dangerouslySetInnerHTML` outside `components/ui/*` (NFR-Sec3 carries).
- **Modal focus traps:** `@blok/dialog` provides them; verified per Blok primitive contract.
- **Live region for toasts:** `@blok/sonner` registers as `role="status"` (carries from PRD-000).
- **Stepper announces step changes:** "Step 2 of 3 — Choose source language" announced via `aria-live="polite"` on step heading.

## 5. Responsiveness and accessibility

### Breakpoints

| Range | Surface | Behavior |
|---|---|---|
| 1024-1920w (Full Page nominal) | Full Page | Left rail 320-360px; main pane fills remainder. Chip strip shows up to 7 chips inline; `+N` overflow popover at 8+. |
| 800-1024w (Full Page minimum) | Full Page | Left rail 280px; chip strip up to 5 chips inline; `+N` overflow popover at 6+. |
| 600-800w (Dashboard nominal+) | Dashboard | Twin tiles side-by-side; "Languages with content" tile full-width. |
| 400-600w (Dashboard low) | Dashboard | Twin tiles stack vertically; "Languages with content" tile stays full-width; chip strip wraps. |
| 320-400w (Context Panel) | Context Panel | Banner full-width; carries PRD-000 layout. No coverage chips rendered. |
| 300w (Dashboard absolute floor) | Dashboard | All tiles stack; chip strip wraps to multiple lines as needed. |

**Note:** Full Page at <800w is not a practical Marketplace iframe size — the design's hard floor is 800w. Below that, layout wraps gracefully but is not the primary target.

### WCAG 2.1 AA requirements

- **Contrast:** every text-on-background combination passes 4.5:1 (normal text) / 3:1 (large text) in both light and dark themes. Coverage chips' filled state (`--primary` on `--primary-background`) verified at the `--text-2xs` (11px) and `--text-xs` (12px) sizes used in the strip. Dashed/dotted chips' `--muted-foreground` / `--placeholder` text on `--background` passes AA.
- **Non-text contrast (UI components):** every chip border (solid / dashed / dotted) has 3:1 contrast against its surrounding background. Verified token-by-token.
- **Focus visibility:** every interactive element shows a `2px solid var(--ring)` focus indicator with `outline-offset: 2px` (carries from PRD-000 `theme.css`).
- **Keyboard nav:** every flow (F1–F9) completable via keyboard. Specifically:
  - Map list rows: Tab to focus row; Enter / Space to select; Arrow Up/Down to move between rows (Blok's `role="listbox"` pattern).
  - Coverage chips in mini-switcher: Arrow Left/Right (radiogroup pattern).
  - Stepper modal: Tab cycles; Enter on Next advances; Escape closes.
  - Delete card modal: Tab between cards; Space selects; Enter on "Delete" confirms.
  - Import preview rows: Tab to chevron, Enter to expand; Tab into action radios.
- **Color-blind safety:** every chip combines color + border-style + glyph. Operators with deuteranopia / protanopia / tritanopia still distinguish all four states.
- **Reduced-motion:** chip hover delay and stepper progress animation respect `prefers-reduced-motion: reduce`. Skeleton shimmer disabled.
- **Screen reader:**
  - Each chip's `aria-label` announces full state (e.g. "German has content").
  - Mini-switcher group has `aria-label="Language version"`.
  - Stepper modal step heading updates announce via `aria-live="polite"`.
  - Delete card group has `aria-label="Delete scope"`; cards have descriptive `aria-label="Delete de version only — en and fr survive"`.
  - Twin-tile numbers have `aria-label="8 maps in de"` (numeric + context).
- **Form labels:** picker `aria-label="Language"`; display-name input `aria-label="Display name in [language]"`; source-language combobox `aria-label="Source language"`. Per NFR-A5.

### A11y open questions

- **Chip strip horizontal scrolling vs popover overflow:** at 8+ languages, V2 prefers a popover (`@blok/popover`) revealing the full strip on demand, rather than horizontal scroll inside the row. Verify with screen-reader users at QA — popover may be harder to navigate than a horizontal scroll. Decision deferred to QA.
- **Stepper progress text under reduced-motion:** the animated progress bar collapses to a static "Step 2 of 3" text label. Verify operators find the static label sufficient.

## 6. Open questions and handoff

### Open questions for the operator

**OQ-V2-1 — Chip-strip overflow at 8+ languages.** V2 proposes a `+N` overflow chip that reveals the full strip in a `@blok/popover` on click. Alternative: horizontal-scroll inside the row. **Recommendation:** popover (cleaner; row stays compact). Decision deferred until a real tenant with 8+ site languages surfaces.

**OQ-V2-2 — Chip click semantics in the list (vs in the mini-switcher).** V2 currently makes list-row chips informational (click selects the row, not the chip) and mini-switcher chips actionable (click switches picker / opens create modal). Alternative: list-row chip click also switches the picker (no row navigation needed). **Recommendation:** keep current split — it preserves row-selection clarity and the mini-switcher is one click away. Verify in the clickdummy.

**OQ-V2-3 — Dashboard "Languages with content" tile cross-surface affordance.** Clicking a chip on the tile navigates to Full Page with that language pre-set. Does Marketplace allow cross-surface navigation from a Dashboard widget? **Recommendation:** verify with `sitecore:marketplace-sdk-client` skill at `/task-breakdown`; if not, fall back to the chip being informational-only on the Dashboard.

**OQ-V2-4 — Stepper component availability.** Blok's `@blok/stepper` primitive may not ship in the quickstart registry (per `blok-components` skill — `stepper` is in the catalog but not always in the quickstart install). If absent: compose with `@blok/progress` + step-number badges (no functional loss; slightly less polished). Verify at `/task-breakdown` and add explicit `npx shadcn add @blok/stepper` to the install list if needed.

**OQ-V2-5 — Mini-switcher disabled-state communication under AC-3.5 dirty-edit-protection.** Currently chips dim + tooltip + cursor change. Alternative: hide chips entirely except the current one. **Recommendation:** keep current (dim + tooltip) — hides information without explaining why. Verify with a test operator.

**OQ-V2-6 — Coverage chip vocabulary in the import preview.** The preview uses the same 4-state chip vocabulary plus a "WILL CREATE / WILL OVERWRITE / WILL SKIP" annotation. Consider: is the chip vocabulary sufficient on its own (the diff `[en–]→[en●]` says "create") and the textual annotation redundant? **Recommendation:** keep both — operators in a 50-map import benefit from text scanning, not just chip scanning.
