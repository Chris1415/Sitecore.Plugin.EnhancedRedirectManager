## 1. Idea Summary

Add **multilingual support to Redirect Manager** so a Sitecore operator can author and manage redirect mappings in any language version of a Redirect Map item, not just `en`. PRD-001 covers the **app-internal multilingual surface**: language picker, language-version CRUD, create-version flow (empty + copy-from), Dashboard tile language scoping, and multilingual-aware JSON import/export. Context Panel multilingual matching and the head-app resolver contract change are deferred to **PRD-002** because they require a parallel head-app deliverable.

This is the second of an extended phase plan: PRD-001 (multilingual CRUD) → PRD-002 (Context Panel multilingual + head-app contract) → PRD-003 (analytics — original PRD-001 split) → PRD-004 (sync-back). ADR-0004's three-phase plan is being reframed to a four-phase plan; new ADR captures the split rationale.

## 2. Problem Statement

PRD-000 shipped en-only by design (ADR-0010) so the MVP could install with zero external dependencies and a single SDK surface family (Authoring GraphQL). The language seam was intentionally preserved — every Authoring read and write is already parametric on `language` (passes literal `"en"` today). Today's app is honest about the limitation via a non-dismissible banner, but Sitecore is intrinsically multilingual: virtually every Sitecore customer has multiple languages defined (operator stated this directly during Q2: "every customer literally has that potentially"). Without multilingual CRUD, the app is unusable for the majority of real Sitecore tenants — even a single-language tenant's site collection may contain other sites in other languages and the operator has no way to manage their redirects from this app.

## 3. Target Users

- **Marketers** authoring redirects in their primary working language (often not `en`) — today they cannot.
- **Site administrators** managing multilingual Sitecore sites with localized URL structures (e.g. `/de/produkte` ↔ `/en/products`) — today the app shows only `en` mappings even when their site has `de` and `fr` versions.
- **Migration operators** importing redirect inventories from legacy systems where mappings already cover multiple languages — today they must convert everything to `en` before import.

Same persona set as PRD-000; the change is what the personas *can* do, not who they are.

## 4. Proposed Solution

A bounded multilingual layer added to the Full Page and Dashboard Widget surfaces of Redirect Manager. The Context Panel stays en-only with an updated banner (its multilingual story lands in PRD-002 with the head-app contract).

Specifically:

- **Language picker** in the Full Page surface — driven by an SDK-enumerated language list, ideally **site-scoped** (only languages the current site supports) with a **tenant-wide fallback** if no site-scoped helper exists. Sub-question: confirm which SDK surface returns site-scoped languages (open question OQ-A).
- **Language-version CRUD** — read, edit, delete the current language version's mappings. All existing Authoring queries/mutations already pass `language` as a parameter; PRD-001 changes the value from literal `"en"` to the picker's current value.
- **Create-version flow** — two paths, no name input:
  - "Create empty [language] version" → creates the version → lands in the standard edit flow.
  - "Copy from [language]" → operator picks source language → pre-fills the new version with the source's mappings → standard edit flow.
- **Two-state UX for missing/empty versions** — distinguish "no version exists" (modal CTA "Create [language] version") from "version exists but empty" (inline "Add first mapping" affordance). This is the honest Sitecore reality, not a UX collapse.
- **Display name per language** — map list and detail show `__Display name` for the current language; fall back to item name when display name is empty. Operator can **edit `__Display name` per language** inside map detail.
- **Delete scope safety** — delete action opens a confirmation modal with explicit radio choice: "Delete [current language] version only" vs "Delete entire map (all languages)." Two-click safety; no ambiguous default.
- **Dashboard tile language scoping** — exact tile flavor decided via clickdummies in `/architect` (UI Designer produces 3 variants per `scope_dial.ui_variants: 3`). Plausible directions: scope-to-picked-language with extra "languages with content" tile; aggregate across all languages with language list tile; toggle between the two.
- **Multilingual-aware import/export** — forward-only schema bump to `redirect-manager/v2`. Single JSON bundle contains all language versions per map (`{ guid, languages: { en: {...}, de: {...} } }`). v1 imports are rejected with a clear message (no auto-migration — confirmed during final question pass).

**Scope boundary:** Context Panel matching stays en-only; its banner is updated to: *"Exact-match, en-only. Regex + multilingual matching ship in follow-ons."*

## 5. Value Proposition

- **Unlocks the majority of real Sitecore tenants** — multilingual is the norm, not the edge case (Q2 answer: "every customer literally has that potentially"). Without this, PRD-000 has a narrow real-world ceiling.
- **Honest UX over collapsed UX** — distinguishing "no version" from "empty version" matches what Sitecore actually exposes and prevents downstream confusion when the operator's mental model meets the data model.
- **Bounded scope keeps ship velocity high** — splitting Context Panel multilingual + head-app contract to PRD-002 means PRD-001 can ship purely via app + SDK changes, smoke-gated end-to-end without any head-app coordination. Matches the operator's working principle: phase by SDK surface and by external-dependency boundary.
- **Forward-only schema** — no v1 backward-compat burden ("no one is using it" — operator). Cleaner code, no legacy branching.

## 6. Core Features

- Site-scoped language picker (with tenant-wide fallback) in Full Page.
- Language-aware Redirect Map detail view — reads/writes scoped to picked language.
- Two-path create-version flow: empty / copy-from-language.
- Distinct UX for "no version" vs "empty version" states.
- Per-language `__Display name` editing in map detail.
- Map list shows current-language `__Display name`, falls back to item name.
- Delete-scope confirmation modal (version-only vs entire-item).
- Dashboard tiles scoped to the picked language (exact tile inventory decided via clickdummies).
- Import/export bundle schema `redirect-manager/v2` (all languages, single file).
- Context Panel banner updated to acknowledge multilingual lands in PRD-002.

## 7. Decisions

Captured live during the brainstorming session.

- **D-Framing — Sitecore content-versioning model.** Multilingual = Redirect Map items have per-language versions; mappings in `de` may differ from `en`. Not per-visitor target-language; not a runtime resolver pattern.
- **D-Audience — Proactive parity, no named customer.** Multilingual is standard Sitecore behavior; every customer has it.
- **D-MVP-Size — Large** ("highly needed"), then split for delivery.
- **D-Languages-Source — Site-scoped preferred, tenant-wide fallback.** Capture the actual SDK surface during `/architect`.
- **D-Create-Version-Flow — Two paths, no name input:** (1) create empty → edit; (2) copy from another language → edit. No display-name input at create time.
- **D-Phasing — Option C split (2 PRDs):**
  - **PRD-001 (this brief)** — app-internal multilingual: Full Page CRUD + create-version + Dashboard scoping + import/export schema bump.
  - **PRD-002 (deferred)** — Context Panel multilingual matching + head-app resolver contract (parallel head-app deliverable required).
- **D-Picker-UX — Resolve via clickdummies during `/architect`.** Three plausible homes (global topbar, per-map, hybrid). Not locked at concept level.
- **D-Display-Name — `__Display name` per current language, fall back to item name when empty.** Operator can edit `__Display name` per language inside map detail.
- **D-Two-States — Distinguish "no version" from "empty version."** Modal CTA for the former; inline affordance for the latter.
- **D-Delete-Scope — Confirmation modal with explicit radio:** "Delete current language version only" vs "Delete entire map (all languages)." Two-click safety; no ambiguous default.
- **D-Dashboard-Scope — Resolve via clickdummies.** Plausible directions: scope-to-picked-language + "languages with content" tile, aggregate-all + language list tile, or toggle.
- **D-Import-Export — Single multi-language JSON bundle.** Schema bump to `redirect-manager/v2`. Per-map `languages` map of language code → mappings.
- **D-Backward-Compat — No v1 backward-compat.** v1 imports rejected with clear message. Forward-only.
- **D-Context-Panel-Banner-Update —** "Exact-match, en-only. Regex + multilingual matching ship in follow-ons."

## 8. Assumptions

Items where the operator did not explicitly answer but the brief assumes a default. None should land in the PRD without explicit confirmation during `/create-prd` convergent discovery.

- The Marketplace SDK exposes a working language-enumeration call (operator: "helper functionality like `listLanguages` available"). The **exact name and call shape** are assumed — possibly `xmc.sites.languages`, possibly elsewhere. To verify against `node_modules/@sitecore-marketplace-sdk/xmc/*.d.ts` per rule 40-sdk-contracts.
- The Authoring GraphQL surface offers a **per-language version-delete mutation** (e.g. `deleteItemVersion`) distinct from the whole-item `deleteItem` already captured. If it doesn't, the "delete current language version only" path of D-Delete-Scope becomes more complex (e.g. clear `UrlMapping` field instead of true version delete). Verify during `/architect`.
- The Authoring GraphQL surface offers a **language-version-create mutation** (e.g. `addItemVersion` or similar) that creates a new version of an existing item in a target language. Operator's mental model assumes this is possible; capture during `/architect`.
- The Redirect Map template's `UrlMapping` field is **versioned** (different content per language). The 3 flags (regex / include-virtual-folder / strip-querystring) and `RedirectType` are **assumed shared** — capture confirms during `/architect`. If any flag is versioned, the picker switches that too, which is a larger surface than expected.
- The "Redirect Map Grouping" template behavior (captured in memory `reference_sitecore_redirect_map_grouping_template`) is **language-agnostic** — groupings don't have per-language versions in any meaningful way. Verify.
- `__Display name` is a versioned field in Sitecore's standard template. Treated as fact.
- Operator persona for PRD-001 is the same as PRD-000 — marketer, site admin, migration operator. No new persona introduced.

## 9. High-Level Technical Considerations

Concept-level technical concerns surfaced during exploration. Detailed architecture happens in `/architect`.

- **Language enumeration source.** Site-scoped via Sitecore's `supportedLanguages` setting is the ideal; SDK helper return shape unknown at concept level. Decision point during `/architect` whether to use `xmc.sites.*`, `xmc.authoring.graphql` with a site-languages query, or a fallback path that reads `__Languages` from the site root item.
- **Authoring GraphQL mutation gaps.** PRD-000 captured `createItem`, `updateItem`, `renameItem`, `deleteItem`. PRD-001 likely needs **`createItemVersion`** (or equivalent verb) and possibly **`deleteItemVersion`**. Capture during `/architect`; fixture capture and ADR per the workflow from ADR-0013 (real-tenant fixture capture, progressive).
- **Field-level versioning awareness.** All Authoring writes today pass `language: "en"`. PRD-001 changes this to a variable. But: writes against shared fields ignore `language`. The wrapper layer in `lib/sdk/redirects-write.ts` must encode field-vs-language coupling correctly — overwriting a shared field "in `de`" doesn't make sense in Sitecore. Capture the shared-vs-versioned matrix during `/architect`.
- **Display name editing surface.** `__Display name` is a standard Sitecore field; editing it via Authoring GraphQL likely uses `updateItem` with a `fields` payload. Confirm the field's standard ID / name. New affordance in map detail.
- **Schema bump for import/export.** `redirect-manager/v2`. Zod schema in `site/lib/schema` adds a `languages` map per item. Import wizard's diff logic (`buildItemDiffs`) gets a language dimension — per-language create/overwrite/skip actions OR a parent-level action that fans out across languages (decide at `/architect`). The import preview UI gets meaningfully more complex; this is a non-trivial UX increment.
- **Performance.** PRD-000's virtualized list (`react-virtuoso`) handles 30 maps / 500 mappings comfortably. Multiplied across 5 languages this becomes 30 × 5 = 150 logical views, still well within budget for the picker model (operator views one language at a time). Aggregation queries for Dashboard tiles may need batching.
- **Real-tenant fixture capture.** ADR-0013 progressive workflow applies. Capture at least one map with multiple language versions, one map with `en` only (no `de`), one map with `de` only (no `en`), and one map with versioned-but-empty content during fixture pass. Operator-tenant has `CHAH DevEx Journey / PROD` per PRD-000; check if that tenant has multilingual maps or if a fresh tenant is needed.
- **Smoke gate plan.** Five gates anticipated (mirroring PRD-000's pattern): `registration` (likely unchanged), `host_frame_smoke` (POC clickdummy comparison — new variants), `crud_round_trip` (now with language switching), `import_export_round_trip` (now with multi-language bundle), `live_walkthrough`. Operator-driven, per `feedback_working_style`.
- **Backward shipping of PRD-000 install.** PRD-001 is an additive change on top of an installed PRD-000. Operators who already installed PRD-000 should be able to upgrade without losing en-only mappings — schema migration is conceptual (no on-disk data to migrate; the Redirect Map items already exist in Sitecore in whatever language versions they have).

## 10. Risks

- **R-Mutation-Gap** — Authoring GraphQL does not expose a version-delete or version-create mutation in the form we expect. Mitigation: capture during `/architect`; if the surface is missing, fall back to "clear field content + leave version" semantics for delete and "create-via-updateItem-with-language" for create. Possible ADR.
- **R-Field-Versioning-Surprise** — A field assumed shared turns out to be versioned (or vice versa). Mitigation: explicit capture pass during `/architect`; the assumption matrix in this brief flags every uncertain field.
- **R-Site-Languages-Surface-Missing** — No SDK call returns site-scoped languages, only tenant-wide. Mitigation: accept tenant-wide fallback (operator pre-approved during Q4). Cosmetic regression only.
- **R-Import-UX-Complexity-Creep** — Multi-language bundle import wizard is meaningfully harder than v1's single-language. Risk that the per-row action picker (create/overwrite/skip) needs a language dimension which explodes the UI. Mitigation: prototype the import preview in the `/architect` UI design pass; consider deferring "per-language override" to a follow-on if the UI becomes overwhelming.
- **R-Context-Panel-Trust** — Operator sees `de` content in Full Page, switches to Pages on a `de` page, opens Context Panel — and the panel still matches `en`-only. Mitigation: PRD-001 banner change makes this explicit; PRD-002 closes the gap. Surface this in the live walkthrough script.
- **R-Display-Name-Authoring-Surface** — Editing `__Display name` may behave differently than editing custom fields (it's a standard field, not a template field). Mitigation: capture during `/architect`; smoke-test in the real-tenant fixture pass.
- **R-Phase-Overrun-Into-PRD-002** — Discovery uncovers that Context Panel multilingual is too tightly coupled to Full Page language picker state to defer cleanly. Mitigation: keep the picker fully Full-Page-scoped; Context Panel reads Pages-host language only (already its current behavior). The phase boundary is clean if no shared state crosses.

## 12. Suggested MVP Scope

The full Large-scope multilingual, scoped to **app-internal only**. Excludes Context Panel multilingual + head-app contract (PRD-002).

**In scope (PRD-001):**

- Language picker (Full Page, site-scoped with tenant-wide fallback).
- Language-aware map list (current-language `__Display name` with item-name fallback).
- Language-version CRUD on existing versions (read, edit, delete-version-only).
- Create-version flow (empty + copy-from-language).
- Per-language `__Display name` editing.
- Delete-scope confirmation modal (version-only vs entire-item).
- Dashboard tile language scoping (exact tile inventory via clickdummies).
- Import/export schema bump to `redirect-manager/v2` (single multi-language bundle).
- v1 import rejection with clear message.
- Context Panel banner update only (no behavior change).

**Out of scope (PRD-001) — deferred to PRD-002:**

- Context Panel multilingual matching.
- Head-app resolver contract change for per-language redirect resolution.
- Pages-host language to Context Panel scoping.

**Out of scope (PRD-001) — deferred to PRD-003 and later (original ADR-0004 phasing, reframed):**

- Usage analytics (Upstash + Vercel Marketplace + head-app instrumentation).
- Dashboard tiles for used/unused, broken/healthy, top-5 lists.
- Per-mapping analytics column.
- Sync-back to Sitecore template (`UsageCount`, `UsageLastSyncedAt`).
- Regex-aware Context Panel matching.
- Concurrent-edit detection.
- Bulk operations (multi-row delete, bulk type change).
- Audit log.
- App UI localization (separate from content localization).
- Public Marketplace App submission.
