## Major Findings

### M1 — Idempotency race window: Publish Site button briefly re-enables between dialog-close and first polling tick

- **File:** `site/components/full-page/WorkspaceHero.tsx` lines 220–289 (pre-fix)
- **Category:** Correctness
- **Severity:** Major
- **Description:** `publishButtonDisabled` only checked `isPolling` (= `trackerState.kind === "polling"`). The tracker state starts as `"idle"` and only transitions to `"polling"` after `setPollingJobId(outcome.jobId)` triggers a re-render AND `usePublishJobTracker`'s first poll interval fires. The gap between the operator clicking "Republish site" (dialog close → POST starts) and that first poll tick (~100ms–3s) is a window where the button is unguarded. A fast double-click could submit two publish jobs.
- **Suggested fix:** Add a dedicated `isSubmitting` state, set to `true` immediately on confirm and cleared in a `finally` block after the `await publish(...)` returns. Include `isSubmitting` in `publishButtonDisabled`.
- **Status:** FIXED (see "Fixes Applied" below).

### m3 — Route validation missing `name` and `options` fields check

- **File:** `site/app/api/publish/route.ts` `validateBody` function lines 29–38
- **Category:** Correctness (defense-in-depth gap)
- **Severity:** Minor
- **Description:** The task breakdown T015 specifies three server-side checks: (1) `body.name` is non-empty string, (2) `body.source === "Redirect Manager"`, (3) `body.options` is an object. Current implementation only checks that `source` is a non-empty string (not that it equals `"Redirect Manager"` specifically). `name` and `options` are unchecked.
