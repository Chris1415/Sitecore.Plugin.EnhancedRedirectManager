## Critical Findings (must fix)

### C1. Hydration mismatch in `useIsTwoPane` (`FullPage.tsx:55-69`)
**Category:** correctness
**Severity:** critical
**Description:** The `useState` initializer at line 56 branched on `typeof window !== "undefined"`, which SSR-renders as `true` (since the default branch evaluates `window.innerWidth`) and CSR may produce a different value. This is the exact pattern flagged in `feedback_hydration_mismatch_pattern` — a SSR/CSR mismatch that is undetectable by Vitest or `npm run build` (only Playwright smoke catches it). The `useState(typeof window !== "undefined" ? window.innerWidth >= 960 : true)` pattern is pre-existing from PRD-000, but PRD-002 touches `FullPage.tsx` (adding `WorkspaceHero`, `StatStrip`, `PreviewDataBanner`) which makes it in-scope for this review.

**Fix applied:** Changed `useState(typeof window !== "undefined" ? ... : true)` to `useState(true)` with a `handler()` immediate call inside `useEffect` to sync the real viewport width on first mount. This is SSR-safe (server always yields `true`; CSR syncs to reality before any visible frame).

## Major Findings (should fix)

### M1. `QuickRedirectForm.handleSubmit` has no catch — unhandled Promise rejection (`QuickRedirectForm.tsx:122-147`)
**Category:** correctness
**Severity:** major
**Description:** `handleSubmit` uses `try { await onSubmit(...) } finally { setSubmitting(false) }` with no `catch` block. `ContextPanel.handleQuickSubmit` correctly shows a toast error AND re-throws the exception. The re-throw means the error propagates out of `handleSubmit` as an unhandled Promise rejection — even though the toast is shown. In Chrome this surfaces as a `UnhandledPromiseRejection` console error and may trigger browser error-tracking alerts. The form correctly preserves user input on error (no reset inside `finally`), but the rejection should be contained.

**Fix applied:** Added `catch (err)` block to `handleSubmit` that stores the error in local `submitError` state (displayed as an inline `role="alert"` `aria-live="assertive"` paragraph below the form). The toast (from parent) still announces the error. The entered values are preserved on error. `submitError` is cleared on the next submit attempt.

### M2. Letter-reveal CSS class mismatch — animation silently broken (`use-letter-reveal.ts:115`, `elevated.css:250`)
**Category:** correctness
**Severity:** major
**Description:** `useLetterReveal` creates character spans with `className = 'fp-hero-letter'`, but `elevated.css` targets `[data-letter-reveal] .letter` (class `letter`). Additionally, `WorkspaceHero.tsx` never adds `data-letter-reveal` to any element — the `headlineRef` points to a plain `<span>` with no container attribute. Both conditions (`data-letter-reveal` attribute AND `.letter` class) must be present for the CSS to apply. The animation silently does nothing in the browser — letters appear flat immediately with no reveal, no `opacity: 0` initial state, no transition. This affects only the Full Page hero headline visual effect; it is not a functional regression.

**Fixes applied:**
1. `use-letter-reveal.ts:115` — Changed `span.className = 'fp-hero-letter'` to `span.className = 'letter'`.
2. `WorkspaceHero.tsx` — Added `data-letter-reveal="true"` attribute to the inner `<span ref={headlineRef}>` element so the CSS selector `[data-letter-reveal] .letter` can match.
