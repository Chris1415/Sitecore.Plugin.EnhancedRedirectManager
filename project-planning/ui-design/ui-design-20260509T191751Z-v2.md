## 1. Design context & assumptions

### Audience (hybrid by surface — applied through the Editor's Studio lens)

- **Context Panel** — content authors and marketers in the Pages editor (PRD § 4 primary 1). They are *not* technical; they want clarity, not density. Editor's Studio leans **soft** here: large map titles, generous whitespace, calm chips. The panel reads like a Notion page sidebar — a list of things-that-matter-for-this-page, never a database table.
- **Dashboard Widget** — site managers (PRD § 4 primary 2). Neutral middle. Three big numbers, breathing room, a calm one-line caveat. Reads like a Linear "insights" card.
- **Full Page** — Sitecore admins, dev leads, implementers (PRD § 4 primary 3). They are technical and they will spend longer here. Editor's Studio leans **denser** on this surface — but the density still buys breathing room compared to a CRM. Soft cards instead of grid lines; type-led hierarchy instead of badges-everywhere; progressive disclosure instead of every-flag-visible. Reads like Linear's project view: scannable, calm, click-to-expand for depth.

The hybrid is achieved entirely through **type scale, density of stacked elements, and how aggressively secondary metadata recedes** — not by switching design systems. All three surfaces share the same direction.

### Goals and success criteria (mapped to PRD)

- **G1** — Eliminate the Pages → Content Editor jump. Editor's Studio answers this by making the Context Panel feel **inviting to read**: marketers won't avoid it because it isn't intimidating. Map names large; flags as gentle chips; the regex-coverage banner soft and friendly.
- **G2** — Per-site overview. The Dashboard Widget is built around three large numerals over generous whitespace. The number is the answer; the rest gets out of the way.
- **G3** — Single management surface. The Full Page replaces Content Editor. Editor's Studio buys *patience* from admins — they'll stay longer, find more, mistake less, because the surface doesn't tax them visually.
- **G4** — Portable redirect sets. The import flow uses a per-item action picker with **one row per item, lots of horizontal room, calm collapsible diff**. The opposite of a 500-row spreadsheet review.

**m2 (host-frame visual smoke):** the POC clickdummy is the canonical visual ground truth. Every spacing, type, and token decision in this spec is reproducible from the clickdummy.

### Constraints (locked, not re-litigated)

- **Brand & system:** Blok (`@blok/*` shadcn registry). Tailwind v4 / Nova preset — tokens live in `app/globals.css` `@theme inline { … }` (per `sitecore:blok-setup`). **No raw hex.** No legacy `@sitecore/blok-theme`.
- **Architecture (locked):** three iframe routes — `/context-panel`, `/dashboard-widget`, `/full-page` — per ADR-0011. Single `MarketplaceProvider`. Every Sitecore call rides the operator's session via the postMessage bridge.
- **Iframe viewports** (per `sitecore:blok-marketplace-integration`):
  - Context Panel: 320–400 px wide × variable height, scrollable.
  - Dashboard Widget: 300–800 px × 200–400 px.
  - Full Page: 1024–1920 px × 600–1080 px (two-pane fits at ≥1024 px; tabbed fallback below).
- **Theme:** Cloud Portal sets light/dark via `.dark` on `<html>`. Editor's Studio is designed dark-first *and* light-first — both are first-class. No mode is "default."
- **A11y:** WCAG 2.1 AA. Keyboard-complete. `:focus-visible` token-styled (never `outline: none`). No color-only state signals. Live regions for save/error toasts.
- **Browser:** Chrome / Edge / Firefox / Safari — latest 2 versions.

### Reference / inspiration (researched, adapted — not copied)

- **Notion** — page sidebars and inline editor blocks. Adapted: the Context Panel's "this page has things attached" reading metaphor; soft cards with rounded corners; section-as-affordance ("+ Add redirect for this page" feels like inserting a block, not opening a form).
- **Linear** — issue cards with progressive disclosure, mono numerals on metadata, calm light/dark parity. Adapted: the Full Page Redirect Map cards (header → expand → mappings list); the import preview rows; the dashboard tile typography.
- **Stripe Dashboard 2024+** — generous whitespace around a dense table, thin separators instead of full grid lines, type-led KPI numbers. Adapted: the Full Page two-pane proportions; the dashboard widget number scale.
- **Vercel project pages** — light-touch chips for tags/states, collapsible details, the "metadata recedes" aesthetic. Adapted: how `RedirectType` and the three boolean flags appear (chips that *don't* shout).
- **Editor's Studio is NOT:** glass-morphism, neon, "AI dashboard" aesthetic, gradient-everything, dense compliance tables, Material 3 elevation cascades. We use **one** elevation level (`shadow-sm`) and rely on whitespace for hierarchy.

### Answers to clarifying questions

- **Audience tone hybrid** — applied per surface as described above. One direction, three weights.
- **Density** — Blok default token spacing. Editor's Studio's *felt* lightness comes from larger type, less per-row metadata, and progressive disclosure — not from larger pixel values.
- **Three extension routes** — locked at `/context-panel`, `/dashboard-widget`, `/full-page` (architecture § 2.2 + ADR-0011).
- **Long-list relationship visibility (agent identity § 92)** — the Full Page redirect list stacks Redirect Maps + their parsed mappings. **At 20+ Redirect Maps each containing 20+ mappings, would two parallel sections fail?** Yes — that is precisely why this variant uses a **nested expandable card** pattern (parent card discloses child mapping rows on expand). Mappings are scoped *inside* their parent map; never displayed as a separate list. ADR-aligned with the Component Usage Atlas lesson. See § 3 + § 4 for the mechanic.

### Assumptions made

- **A-V2-1** — Marketers will use the Context Panel because it does *not* feel like Content Editor. Validated post-launch via live walkthrough (PRD m5).
- **A-V2-2** — Admins on the Full Page tolerate progressive disclosure better than wall-of-fields once they trust the disclosure is fast. Mitigation: expand/collapse is keyboard-accessible (Space / Enter on the card header) and the chevron rotates < 200 ms.
- **A-V2-3** — Mono-tabular numerals on counts (Dashboard tiles, mapping counts on cards, import summary) feel "earned" — Blok's default sans supports `font-variant-numeric: tabular-nums`. We use it, no separate font.
- **A-V2-4** — Light and dark are equal-weight design targets. The clickdummy ships both, side-by-side, on the same screen so QA can spot drift early.
- **A-V2-5** — The Full Page two-pane (sidebar 280 px / content fluid) is the right choice at iframe widths ≥ 1024 px. Below that, fallback to a tab bar (`Sites | Redirects`) on top of a single content pane. Verified at 1024 / 1280 / 1440 px in the POC.

## 2. Design direction

### Visual direction — "Editor's Studio"

**One-line north star:** *the redirect set is a manuscript; the operator is the editor; the page is calm enough to read for ten minutes without fatigue.*

Type-led hierarchy. Generous whitespace. Soft rounded cards (Blok default radius — no custom radii). Secondary metadata recedes into `text-muted-foreground`. One elevation level. No grid lines on tables — separators only. Flags are not banners; they are gentle chips. The whitespace **is** the brand decision — operators feel the difference before they can name it.

### Color system — Blok semantic tokens only, light + dark first-class

| Role | Blok semantic token (Tailwind utility) | Used for |
|------|----------------------------------------|----------|
| Page surface | `bg-background` | Outermost canvas (every route) |
| Card surface | `bg-card`, `text-card-foreground` | Every redirect map card, every dashboard tile, every modal/drawer |
| Subtle surface | `bg-muted`, `text-muted-foreground` | Recessed metadata (mapping counts, last-updated, "drag to reorder" hints), empty-state illustrations, sidebar inactive items |
| Hairline | `border-border` (default), `border-border/60` (extra-soft inside cards) | Card outlines, expanded section dividers, table-style separators in mapping lists |
| Primary action | `bg-primary`, `text-primary-foreground` | "Save", "Apply import", "Confirm import" — the page's one decisive button per surface |
| Secondary action | `variant="outline"` (Blok button outline = `border-border` + `bg-transparent` + `hover:bg-muted`) | "Cancel", "Add mapping", "Show technical details", "Show diff" |
| Ghost / icon-only | `variant="ghost"` (transparent → `hover:bg-muted`) | Per-row edit / delete, drag handle, expand chevron |
| Type chip — calm informational | `bg-muted text-muted-foreground border-border/60` | `RedirectType` badge (e.g. "301", "ServerTransfer") — calm, not alarming |
| Flag chip — present | `bg-accent/10 text-accent-foreground border-accent/30` | `PreserveQueryString`, `PreserveLanguage`, `IncludeVirtualFolder` chips when `true`. Chips are **only rendered when the flag is true** (PRD US-1 AC). |
| Success | `text-success` (Blok semantic if shipped — verify at install; falls back to `text-foreground` + Lucide `<CheckCircle>`) | Save confirmation toast, import-summary "succeeded" rows |
| Warning | `text-warning` + `bg-warning/10 border-warning/30` | The persistent regex-coverage banner; "modified externally" placeholder copy (deferred to PRD-001+) |
| Destructive | `text-destructive`, `border-destructive` | Delete confirm prompts, error banners, import "skip — conflicting" pill |
| Focus ring | `ring-ring` + `ring-2 ring-offset-2 ring-offset-background` | All `:focus-visible` outlines on interactive elements |

**No invented tokens. No raw hex.** If a Blok semantic isn't shipped (e.g. `success` / `warning` not exposed by `theme.json`), the spec degrades gracefully to `text-foreground` + a Lucide icon carrying the meaning.

**State icon convention (per agent identity § 91):** all state icons use **Lucide-react** glyphs (`<XCircle>`, `<CheckCircle2>`, `<AlertTriangle>`, `<Info>`, `<Trash2>`, `<Pencil>`, `<GripVertical>`, `<ChevronRight>`, `<Plus>`, `<Search>`, `<Upload>`, `<Download>`). They inherit `currentColor` and respect light/dark theme. **No emoji codepoints anywhere.**

### Typography — type-led hierarchy is the variant's signature

Single sans family — **Geist Sans** (Blok ships it via Nova preset; do *not* re-install via `geist` package per `sitecore:blok-setup`). Single mono — **Geist Mono** (used only for: counters, mapping source/target source/target arrow rows, drag-handle row indices, JSON-import preview).

| Role | Token / utility | Where |
|------|-----------------|-------|
| Display (largest) | `text-3xl font-semibold tracking-tight` | Dashboard Widget primary numerals (e.g. "12 maps") |
| Page title | `text-2xl font-semibold` | Full Page header ("Redirects — Northwind Marketing"); modal titles |
| Section / Map name | `text-lg font-medium` | Redirect Map card header name (Context Panel + Full Page) |
| Body | `text-sm font-normal` | Default body, form labels |
| Body emphasis | `text-sm font-medium` | Buttons, tab labels, table column headers |
| Metadata / kicker | `text-xs uppercase tracking-wide text-muted-foreground` | Labels above dashboard numerals; "Last updated" / "3 mappings" / "RedirectType" kickers; section labels in the import preview |
| Code / mono | `font-mono text-sm tabular-nums` | Source/target strings in mapping rows, drag-row index numbers, count numerals on cards |
| Caption | `text-xs text-muted-foreground` | Banner copy, tooltip text, footnote ("Redirect counts only — usage analytics ship in a follow-on release.") |

**Tabular numerals are required** on every count column and mapping-row source/target — this is what makes Editor's Studio feel "clean" rather than "designed". Apply via `tabular-nums` Tailwind utility on the relevant element. No font swap.

### Spacing — Blok default scale; whitespace is the differentiator

Everything snaps to Blok default spacing tokens (`p-1` … `p-12`). Editor's Studio uses the **upper half of the scale** more aggressively than dense variants:

- Card outer padding: `p-6` on Full Page redirect cards (vs `p-4` typical) — this is the single biggest reason the surface feels editorial.
- Dashboard tile padding: `p-8` — generous around the big numeral.
- Context Panel section gap: `gap-4` between Redirect Map groups; `gap-2` between mapping rows inside a group.
- Full Page two-pane gap: `gap-6` between sidebar and content; `p-8` page margin.
- Modal / drawer inset: `p-6` body, `p-4` header/footer.
- Sticky sub-page elements (banner, footer): `py-3 px-4`.

### Layout & structure

**All three surfaces share a 3-zone vertical anatomy** — different proportions, same metaphor:

```
┌──────────────────────────────────────────────────────────────┐
│  Zone 1 — HEADER (compact, type-led)                         │
│  Page title or kicker; secondary actions on the right        │
├──────────────────────────────────────────────────────────────┤
│  Zone 2 — DISCLOSURE / BANNER (sticky, soft warning)         │
│  Context Panel: regex-coverage notice                        │
│  Dashboard:     usage-analytics caveat (footnote position)   │
│  Full Page:     none in default; sticky for unsaved changes  │
├──────────────────────────────────────────────────────────────┤
│  Zone 3 — PRIMARY DATA SURFACE (fills available)             │
│  Context Panel: stacked Redirect Map cards (vertical scroll) │
│  Dashboard:     3 stat tiles in a row (or stacked < 360 px)  │
│  Full Page:     two-pane (sidebar + cards) or tabbed fallback│
└──────────────────────────────────────────────────────────────┘
```

### Key components and patterns

- **Soft Card** — `@blok/card` with `shadow-sm` and `rounded-lg` (Blok default). Used for: Redirect Map groups (Context Panel + Full Page), Dashboard stat tiles, modal/drawer content blocks. **One elevation only** across the variant.
- **Type-Led Card Header** — composed pattern: large `text-lg font-medium` map name on row 1; row 2 is `text-xs uppercase` kickers separated by `·` ("3 mappings  ·  Last updated 2h ago"). `RedirectType` and flag chips trail right-aligned. The map name is a `<button>` (toggles expand on Full Page) with the chevron at the right edge.
- **Inline Mapping Row** — `font-mono text-sm tabular-nums`, source on the left, `→` arrow (Lucide `<ArrowRight>` at `size-4 text-muted-foreground`), target on the right. Drag handle (Lucide `<GripVertical>`) in a `text-muted-foreground` wrapper that *only* shows on row hover or keyboard focus. Edit / delete actions appear in the same hover/focus zone, right-aligned.
- **Empty State** — Blok `@blok/empty-states` primitive **with substantial whitespace around it**. Uses the Blok-shipped illustration (graphic from `blok.sitecore.com/graphics`). Headline: `text-base font-medium`; body: `text-sm text-muted-foreground`; primary CTA below.
- **Soft Banner** — Blok `@blok/alert` with `variant="default"` *not* `variant="destructive"`. Background `bg-warning/10`, border `border-warning/30`, icon Lucide `<Info>` at `text-warning`. Used for the regex-coverage notice — friendly tone: "Heads up — direct-string matches only. Regex pattern matches aren't covered yet."
- **Sticky Action Bar** — at the bottom of forms/drawers when the form has dirty state. Single primary action right-aligned; secondary "Cancel" outline-button to its left; `bg-background/95` with `backdrop-blur` and a top hairline.
- **Progressive-disclosure card** — the load-bearing pattern of this variant. Parent card collapsed shows header + flags + count. Click → expand → mappings list reveals beneath the header on the same card (no separate panel). 200 ms ease-out animation; chevron rotates 90°. Implementation: `@blok/collapsible` (already in quickstart inventory).
- **Searchable list (modal)** — Blok `@blok/command` (post-quickstart install). Used for "pick which Redirect Map to add to" in the Context Panel + the searchable site picker in the Full Page sidebar.

```
│  └────────────────────────────────────┘  │
│                                          │

```

**On the Context Panel, Redirect Map cards are *always* expanded** — narrow viewport + vertical scroll means progressive disclosure adds no value here. Mappings are inline. Hover/focus reveals per-row edit/delete actions. Long source/target strings truncate with `text-ellipsis` mid-string + tooltip on focus/hover.

**"Add redirect for this page" flow (US-2):**

1. Operator clicks the `+ Add redirect for this page` row.
2. Blok `@blok/dialog` (modal) sized for the panel width opens. Title: "Add a redirect for /spring-campaign". Body has a Blok `@blok/command` searchable list.
3. The first row in the list is **"+ Create new Redirect Map"** (highlighted with a soft `bg-accent/10` background and a Lucide `<Plus>` icon — visually distinct from existing-map rows).
4. Existing maps below show: map name (left, `text-sm font-medium`) + last-updated kicker (right, `text-xs text-muted-foreground`).
5. Picking an existing map → modal swaps to a **single inline form**: source pre-populated (read-only), target field empty (focused), "Add" + "Cancel" sticky bottom bar.
6. Picking "+ Create new Redirect Map" → modal swaps to a **fuller form**: map name input, `RedirectType` `@blok/select` (initial value empty, placeholder "Choose a redirect type"), three boolean toggles (`@blok/switch` × 3), and the first mapping row (source pre-populated, target focused). Save validation blocks empty target and unset `RedirectType`.

**Per-row inline edit/delete (US-3):**

- Hover or keyboard focus on a mapping row reveals two ghost icon buttons on the right edge: `<Pencil>` (edit), `<Trash2>` (delete).
- Edit → row swaps in place to two inline `@blok/input` fields. Save → persist + collapse. Cancel → discard prompt only if dirty.
- Delete → small confirm popover (`@blok/popover`) anchored to the trash icon: "Remove this mapping?" + "Remove" / "Cancel". On confirm, the row animates out (220 ms slide-up + fade).

**States — Context Panel:**

| State | Visual |
|---|---|
| Default | Banner + "+ Add redirect" + N map cards. |
| Loading (initial) | Banner static; below it, 2 `@blok/skeleton` map cards (`rounded-lg h-32 bg-muted`). No spinner. |
| Empty | Banner static; below it, Blok `@blok/empty-states` with illustration, headline "No redirects affect this page", body "Direct-string matches only — there may be regex-based redirects not shown here." Primary CTA: "+ Add redirect for this page". |
| Error | Friendly Blok `@blok/alert` `variant="destructive"`: "Couldn't load redirects." Body includes a "Try again" Blok `@blok/button` `variant="outline"` and a "Show technical details" `<button>` that toggles the verbatim GraphQL error in a `font-mono text-xs` block. |
| Focus | All focusables show `ring-2 ring-ring ring-offset-2 ring-offset-background`. |
| Success toast | Blok `@blok/sonner` toast bottom-center: "Redirect saved" + Lucide `<CheckCircle2>`. Auto-dismisses 3 s. |

**Import flow (US-10) — full-screen takeover, three steps:**

```
Step 1 — Upload  ───────►  Step 2 — Review  ───────►  Step 3 — Apply / Summary
```

The takeover replaces the content pane; the sidebar dims (`opacity-50 pointer-events-none`). A breadcrumb-style stepper at the top shows the three steps with the current active.

- **Step 1 — Upload.** Centered drop-zone (`@blok/card` with dashed `border-border/60 border-dashed`). Lucide `<Upload>` icon at `size-8 text-muted-foreground` above an "Choose a JSON file" Blok button + drag-and-drop affordance. Caption: "Schema redirect-manager/v1 · Items matched by Sitecore item GUID." Below the dropzone: a soft-warning callout reminding the operator to "Export current redirects first (one-click) — that's your rollback if something goes wrong" with a one-click "Export current site" outline button.
- **Step 2 — Review.** Stacked rows, one per incoming item. Each row is a `@blok/card p-4` with:
  - Top-left: classification chip — "NEW" (`bg-accent/10 text-accent`) or "CONFLICT" (`bg-warning/10 text-warning`).
  - Map name `text-lg font-medium`.
  - Kicker: "5 mappings · ServerTransfer · GUID …836DE".
  - Right edge: action `@blok/radio-group` with three options — `Create` / `Overwrite` / `Skip`. **Default for "NEW" rows: "Create" pre-selected. Default for "CONFLICT" rows: nothing selected (operator must pick).**
  - Below the row, a `@blok/collapsible` "Show diff" toggle. When expanded, a hairline-separated two-column layout: "Current" / "Incoming" with field-by-field rows. Mappings diff uses a Linear-style line-by-line block: removed lines `bg-destructive/10 text-destructive`, added lines `bg-success/10 text-success`, unchanged lines `text-muted-foreground`.
  - Sticky top: "Apply same action to all conflicts: [Create | Overwrite | Skip]" `@blok/toggle-group` (post-quickstart install). Selecting one fills every CONFLICT row.
  - Sticky bottom: "Confirm import (12 actions)" primary button — **disabled until every CONFLICT row has an action**. To its left: "Back" outline button.
- **Step 3 — Apply / Summary.** A progress bar (`@blok/progress`) animates as the per-item mutations execute. After completion, a summary card lists per-item outcomes:
  - Succeeded rows: `text-success` + `<CheckCircle2>` + "Created" / "Overwrote" verb.
  - Failed rows: `text-destructive` + `<XCircle>` + verbatim error in a `font-mono text-xs` collapsible.
  - Skipped rows: `text-muted-foreground` + "—" + "Skipped".
  - Bottom: "Done" primary button (closes takeover) + "Retry failed (N)" outline button (re-runs only the failed rows).

**Friendly error UX (FR-13):** every save failure shows a Blok `@blok/alert variant="destructive"`:
- Headline: "Couldn't save Spring Launch Map" (uses the friendly map name, not "GraphQL error").
- Body: short reassurance: "Your changes are still here — try again, or copy the technical details below to share with your developer."
- Two actions: "Try again" outline button + "Show technical details" `<button>` that expands a `font-mono text-xs` block with the verbatim GraphQL error and a "Copy" icon button.

**Cancel / discard (FR-12):** on close/cancel of any form/drawer with unsaved changes → `@blok/alert-dialog`: "Discard your changes?" with "Keep editing" (outline) and "Discard" (destructive) buttons. On clean form → silent close.

| Flow | Surface | Key steps |
|---|---|---|

**Long-list relationship-visibility check (agent identity § 92) — answered:** the Full Page card pattern keeps each map's mappings *inside* its parent card (progressive disclosure on expand). Mappings are never displayed in a separate parallel section. At 30+ maps with 30+ mappings each, virtualization (`react-virtuoso`) keeps cards performant; expanded mappings live within their parent card's bounds; the parent-child link is structural, not visual-affinity-dependent. The user can never lose track of which mapping belongs to which map because they're nested. ✓ Pattern verified safe at scale.

### 4c. Accessibility notes (WCAG 2.1 AA)

- **Keyboard:**
  - Tab order follows reading order on every surface.
  - Card expand/collapse on Full Page: `Space` and `Enter` toggle when the header button is focused.
  - Mapping reorder: `Space` (pickup) → arrow keys (move) → `Space` (drop). `Esc` cancels. ARIA live region announces "Row N of M, picked up" / "moved to position K" / "dropped".
  - Modal / dialog / drawer: focus trapped on open; `Esc` triggers cancel/discard logic; first focusable receives focus on open; focus returns to the trigger on close (Blok primitives handle this).
- **Screen readers:**
  - Every card header is a `<button>` with `aria-expanded` reflecting state and `aria-controls` pointing at the body.
  - Mapping list is a `<ul>` with `aria-label="Redirect mappings"`; rows are `<li>`.
  - Status changes (save success, save error, import progress, drag-drop reorder) emit to a polite `aria-live` region (`role="status"`).
  - Import progress also uses `aria-valuenow` / `aria-valuemax` on the progress bar.
  - The persistent regex-coverage banner has `role="note"` (not `alert` — it is informational, not urgent).
  - The Dashboard tile is a `<button aria-label="12 redirect maps · open in Redirects">` so the screen reader gets the full context, not just "12".
- **Visible focus:** `:focus-visible` always shows `ring-2 ring-ring ring-offset-2 ring-offset-background`. Never `outline: none` without replacement.
- **Color contrast:** all text/background pairs use Blok semantic tokens that ship at 4.5:1 (normal text) or 3:1 (large text + UI). Verified via Blok's theme defaults; no overrides that risk contrast.
- **No color-only signals:** every classification chip ("NEW" / "CONFLICT") has an icon AND a text label. Save / error toasts have icon + text. Failed import rows have `<XCircle>` + "Failed" verb + the actual error.
- **Reduced motion:** `@media (prefers-reduced-motion: reduce)` removes the 200 ms expand animation, the 300 ms success flash, and the 200 ms button-shake. Replace with instant state change.
- **Touch targets:** ≥ 44 × 44 px on every interactive element. Inline icon buttons in the mapping row use Blok `size="icon"` which defaults to 36 px — we override to `h-11 w-11` (44 px) inside the panel context to comply.
- **Labels:** every input has a visible `<Label>` (Blok `@blok/label`); placeholders never replace labels. Toggle switches have a secondary `text-xs text-muted-foreground` helper line directly beneath the label.

### 4d. Motion language

Editor's Studio uses motion sparingly — motion is for confirmation, not entertainment.

| Pattern | Duration | Easing | Where |
|---|---|---|---|
| Card expand/collapse | 200 ms | `ease-out` | Full Page Redirect Map cards |
| Modal / drawer enter | 240 ms | `ease-out` | All modals + drawers |
| Modal / drawer exit | 180 ms | `ease-in` | All modals + drawers |
| Toast enter/exit | 220 ms | `ease-out` | Sonner toasts |
| Button shake (error) | 200 ms (3 cycles, 4 px amplitude) | `ease-in-out` | Form save error |
| Success flash | 300 ms | `ease-out` | Row / button after successful save |
| Drag pickup lift | 120 ms | `ease-out` | Mapping row drag start (subtle scale `1.02` + `shadow-md`) |
| Skeleton shimmer | 1.4 s loop | `ease-in-out` infinite | All skeletons (Blok default) |
| Chevron rotate | 200 ms | `ease-out` | Card expand chevron |

All durations and animations respect `prefers-reduced-motion: reduce` and degrade to instant state changes.

## 5. Responsiveness and accessibility

### 5a. Breakpoints (per Blok defaults — refined for iframe constraints)

Blok's Tailwind v4 ships standard `sm` / `md` / `lg` / `xl` / `2xl`. Editor's Studio uses them as follows:

| Token | Width | Target surface | Behavior |
|---|---|---|---|
| (base, < `sm`) | < 640 px | Context Panel only | Single column; cards `p-4`; numerals reduce to `text-2xl` |
| `sm` | 640 px | Dashboard | Tiles grid 2-up |
| `md` | 768 px | Dashboard | Tiles grid 3-up |
| `lg` | 1024 px | Full Page two-pane minimum | Sidebar 280 px appears; content pane fluid |
| `xl` | 1280 px | Full Page comfort | Cards `max-w-3xl` to keep readable line length |
| `2xl` | 1536 px | Full Page wide | Sidebar can grow to 320 px; right margin absorbs extra width |

**Iframe-specific rule** (per `sitecore:blok-marketplace-integration`): the panel (`/context-panel`) treats every breakpoint as mobile-first — never apply `md:`/`lg:` styles, since those widths are unreachable inside the panel iframe. The Full Page design uses `lg:` and `xl:` aggressively because that route is delivered to a wider iframe.

### 5b. A11y requirements (WCAG 2.1 AA — full list)

- **NFR-A1 (WCAG 2.1 AA):** every text/background pair meets 4.5:1; UI components meet 3:1. Verified via Blok semantic tokens.
- **NFR-A2 (Full keyboard):** every action and reorder is keyboard-driven. Skip-to-content link at the top of the Full Page.
- **NFR-A3 (Screen reader labels):** every interactive element has an accessible name. Status changes announced via `aria-live="polite"` region.
- **NFR-A4 (Visible focus):** `ring-2 ring-ring ring-offset-2 ring-offset-background` on every focusable element via Tailwind `:focus-visible:`.
- **NFR-A5 (Color contrast at runtime):** Blok defaults; the Editor's Studio variant adds **no** overrides that change contrast. Validated at QA in both light and dark.
- **No emoji codepoints** for status icons (per agent identity § 91); Lucide-react glyphs throughout.
- **`prefers-reduced-motion`:** all motion respected; no parallax / no auto-play anywhere in this variant.
- **`prefers-color-scheme`:** the host (Cloud Portal) sets the theme; the iframe respects the parent. We do not override.

## 6. Open questions and handoff

### 6a. Open questions (v2-specific — design)

- **OQ-V2-1 — Light/dark first paint timing.** Cloud Portal pushes the theme via the SDK; on initial iframe handshake there is a ~100–500 ms window where the iframe might render light when the parent is dark. Recommendation: a one-frame `bg-background` mask + `@blok/skeleton` placeholders during the handshake, so the operator never sees a flash of the wrong theme. Confirm at smoke against the real portal.
- **OQ-V2-2 — Sidebar width on Full Page at < 1280 px.** 280 px is comfortable at 1024 px but feels heavy at 1280 px (sidebar takes ~22% of width). Should the sidebar be **collapsible** (an icon-only rail on `xl:` and below)? Recommendation: ship at fixed 280 px in MVP; gather operator feedback during the live walkthrough; add a collapse toggle in PRD-001+ if asked.
- **OQ-V2-3 — Drag-drop on touch devices.** `@dnd-kit/core` supports touch but the keyboard-reorder path is the certified accessible flow. At MVP, the design assumes operators are on desktop (Cloud Portal). If touch becomes a real surface, revisit drag affordance sizing.
- **OQ-V2-4 — Diff renderer for mappings on import preview.** A Linear-style line-by-line diff is specified; for very large mapping diffs (>50 rows) the collapsible can become tall. Recommendation: cap visible diff at 20 rows with a "Show 30 more rows" affordance. Confirm at architecture-of-import-renderer if the row count threshold is right.
