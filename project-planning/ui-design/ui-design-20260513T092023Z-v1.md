## 1. Design context & assumptions

### Audience

Per PRD § 4 — three personas, unchanged from PRD-000:

1. **Marketers** authoring redirects in their primary working language (often non-`en`). They spend the most time in one language at a time and benefit most from the picker being unmistakable.
2. **Site administrators** managing multilingual sites with localized URL structures (`/de/produkte` ↔ `/en/products`). They switch languages occasionally but live in one language per work session.
3. **Migration operators** importing multilingual JSON bundles via the wizard. They need the language axis visible during preview but do not "live" in the picker — for them, picker context matters at create-version time only.

The shared property: **most operator time is spent in one language**. Switching is occasional but high-consequence (US-3 dirty-edit protection in AC-3.5 exists precisely because a mid-edit language switch would corrupt intent). V1 leans into this — make the current language a focal-point so the operator never wonders what scope they are editing in.

### Device context and viewport constraints

Per `blok-marketplace-integration`:

- **Full Page (Marketplace `xmc:fullscreen`):** 1024–1920px wide, 600–1080px tall. Primary surface — picker lives here.
- **Dashboard Widget (`xmc:dashboardblocks`):** 300–800px wide, 200–400px tall. Compact. Picker is read-only here (reads the same `localStorage` state per ADR-0017).
- **Context Panel (`xmc:pages:contextpanel`):** 320–400px wide, variable height. Narrow. No picker visible — Context Panel reads `pageInfo.language` from Pages, not the picker state (architecture § 5.6 + § 2.2). The Pages-aware banner is the only Context Panel change.

### Goals and success criteria

1. The current working language is **never ambiguous**. An operator who walks away mid-edit, returns, and looks at the screen for two seconds knows what language they are editing in without scrolling, hovering, or clicking.
2. Switching language is a **two-click commitment**, not a side-effect. Operator clicks the picker (1) and chooses a language (2). The picker's prominence makes this feel intentional, not incidental.
3. The map list communicates per-row language status with **minimal visual weight** so the language signal does not fight the picker for attention. One dot per row, that is all.
4. The Dashboard surfaces are **strictly scoped to the current picker language** by default — no toggle, no "view all" mode. The one tile that breaks scope ("Languages with content") shows the codes so the operator has tenant-wide context without a mode switch.
5. The import preview surfaces **every language column at once** in a horizontal-scrolling table so the migration operator never has to expand a row to see what they are importing.
6. The Context Panel banner reads `pageInfo.language` and explains the en-only matching limitation in the operator's current Pages-host language context.

### Constraints

- **Blok mandatory** per `blok-marketplace-integration`. Every interactive primitive is a Blok `@blok/*` component or composes from Blok primitives. No raw HTML buttons. No shadcn defaults — Blok theme tokens drive every color.
- **WCAG 2.1 AA** per NFR-A1. Color contrast verified light + dark. Focus-visible outlines on every interactive element (structural guard from PRD-000 extends — `outline:none` without paired `:focus-visible` rule is banned).
- **Iframe-bound viewport** — no top-level navigation, all SDK postMessage. Designs respect the iframe widths above.
- **No new third-party libraries** per architecture § 7 — picker composes from already-installed Blok primitives.
- **No color-emoji codepoints** for state icons (per ui-designer signature behavior + `blok-theming` "Color-emoji codepoints" note). Use `lucide-react` icons or monochrome glyphs.
- **No `dangerouslySetInnerHTML` outside `components/ui/*`** per NFR-Sec3.
- **Visual continuity with PRD-000** — same Blok Nova theme tokens, same Geist Sans/Mono, same `--radius-lg` card chrome, same `--shadow-sm` elevation. New components plug into the established visual frame.

### Reference and inspiration

- **PRD-000 POC v1** at `pocs/poc-v1/` — the established visual rhythm. V1's spec extends this; it does not introduce a new style. The topbar treatment is new but the typography scale, color tokens, card chrome, and modal patterns are inherited verbatim.
- **Linear command-bar / topbar pattern** — Linear's app shell promotes the current workspace context (project, view) into a topbar slot that is always visible regardless of which view the operator is in. V1 borrows that pattern intent: the picker is workspace-level, not view-level.
- **Vercel project-switcher dropdown** — when an active context is the privileged scope, a single large dropdown with `code + name` formatting (e.g. "EN — English") is the clearest visual idiom. V1 uses the same chip-with-name pattern in the picker.
- **Sitecore Pages / XM Cloud language dropdown** — Pages places the language picker in the topbar with an ISO code chip. V1 mirrors this familiarity — operators coming from Pages see the same idiom.

### Assumptions made (explicit — for traceability)

- **A1** — The Cloud Portal's iframe sets `prefers-color-scheme` and/or `.dark` class. V1 honors host preference (Blok marketplace integration § "Theming inside iframes"). No theme toggle inside the iframe.
- **A2** — Geist Sans + Geist Mono load reliably via `next/font/google` per Blok setup § "Fonts — Geist already ships with Nova preset". V1 inherits this — does not specify alternative fonts.
- **A3** — `lucide-react` is already in the Blok dependency tree (per `blok-theming` § "Color-emoji codepoints"). V1 uses Lucide icons throughout — no need for a new icon library.
- **A4** — The Sites.Language wire shape exposes both `iso` (e.g. `"en"`) and `nativeName` / `displayName` per architecture § 5.1. V1's picker UI uses the pattern `{iso.toUpperCase()} — {nativeName}` for every entry. Falls back gracefully when `nativeName` is null (shows only the ISO).
- **A5** — The `versionsByLanguage` query (architecture § 5.4) returns a list of language codes for each map. V1's per-row dot indicator reads from this — filled dot when current-language is in the list, outlined dot when it is not.
- **A6** — Cross-tab `storage` event behavior (architecture § 2.1, `use-language-picker.ts`) means that if the operator opens Full Page in two tabs and changes the picker in tab A, tab B's picker reflects the change after the next render. V1 does not visualize this transition (no animation, no notification) — the change just happens. Cross-tab edge case in ADR-0017 is accepted as documented behavior.
- **A7** — The "Pages-aware banner" reads `pageInfo.language` once on Context Panel mount and re-reads on subscription update. V1 does not animate the banner copy change — it swaps in-place.

## 2. Design direction

### Visual direction — "Topbar Pilot"

The defining metaphor: **the operator sits in the cockpit of a single-language working session, and the picker is the throttle in front of them.** It is large, persistent, color-toned to the brand-primary (`--primary` = Blok purple `#6e3fff`), and never scrolls off-screen. Switching language feels like reaching for the throttle — deliberate, not accidental.

Visual hierarchy from top to bottom on Full Page:

1. **Topbar (sticky, ~64px tall)** — Logo wordmark left, **language picker dominant-center** with a `@blok/badge` chip showing the ISO code in `--primary-background` fill and the native language name beside it, secondary actions (Import / Export / New Map) collapsed into a `@blok/dropdown-menu` action button on the right.
2. **Content area** — Two-pane layout: list-pane left (map list with row dots), detail-pane right (map detail, mappings table). Same layout as PRD-000 — PRD-001 inherits and does not reorganize.
3. **No bottom chrome** — Status / footer area kept blank; surface area is dedicated to operator's working content.

Color tone: **inherit PRD-000 directly.** The topbar is `--card` with a `--border` bottom rule and `--shadow-sm` elevation. The picker is `--background` with `--border` outline; its dropdown body is `--popover`. The current-language chip inside the picker trigger uses `--primary-background` fill (light purple wash, `#eae7ff`) with `--primary` text (`#6e3fff`) — this is the only spot of brand-primary in the resting state of the surface, which gives the picker its weight without competing with the operator's content.

Typography: **inherit PRD-000.**

- **`--font-sans` (Geist Sans)** for everything except code-style content.
- **`--font-mono` (Geist Mono)** for the ISO code chip inside the picker trigger (e.g. `EN`, `DE`) — the monospace treatment makes 2-letter codes a recognized visual element.
- **`--text-2xl` (1.5rem) `font-weight: 600`** for the picker trigger label container — picker is the heaviest text element in the topbar after the logo wordmark.
- **`--text-base` (1rem)** for body copy, list rows, modal body text.
- **`--text-sm` (0.875rem)** for secondary copy, captions, footnote text.
- **`--text-xs` (0.75rem)** for badges, breadcrumbs, language-code chips in row dots (when shown).

Motion (inherit PRD-000 + minimal additions):

- **Picker open/close** — `@blok/dropdown-menu` default Radix motion (slide + fade, 120ms, `ease-out`). No custom override.
- **Picker language change → list refetch** — Existing `@blok/skeleton` rows replace map list during fetch. No bespoke animation. The list re-renders with the new language data in <1.5s per NFR-P2.
- **Dot indicator** — No transition. Filled / outlined is a static state.
- **Toast (Sonner)** — Inherits PRD-000 Sonner config. Saves toast: *"Saved to [language] version."* with default Sonner enter/exit motion.
- **Reduced-motion** — `@media (prefers-reduced-motion: reduce)` shortens all transitions to 0ms (Blok primitives already respect this via Radix; V1 does not override).

**Sticky topbar.** The picker stays visible during list scrolling. Topbar uses `position: sticky; top: 0` with `z-50` to clear modals / popovers.

**Dot indicator placement.** Left-most cell of each row in the map list, vertically centered. Filled = `bg-primary` (purple). Outlined = `border-2 border-input bg-transparent`. Dot is 8px diameter (`size-2`) — small enough not to compete with the picker for attention, large enough to scan a list of 30+ maps.

**Iframe responsiveness inside Full Page.**

- At ≥1280px (typical Cloud Portal Full Page): two-pane layout, list-pane fixed `w-80` (320px), detail-pane fluid.
- At 1024–1279px: two-pane layout, list-pane narrows to `w-72` (288px).
- At <1024px (atypical, but iframe could be resized): collapse to single-pane stacked. List shows; clicking a map navigates to detail; back-affordance returns to list. Topbar always sticky.

**Dashboard scoping model — strict current.** Per V1's anchor decision: the first two tiles ("Maps in EN", "Mappings in EN") are explicitly scoped to the picker language. The label even includes the ISO code. The third tile ("Languages with content") lists every language code that has at least one map version on the tenant — this gives the operator tenant-wide context **without** a toggle. No "view all" mode exists.

**Why no toggle?** A toggle introduces a transient view-mode state that the operator might leave in an unexpected state. V1's bet: operators want **either** "what's in my language" (the picker context) **or** "what's available on the tenant" (a one-shot question answered by reading the third tile). No middle ground needed.

**Footnote** carries forward PRD-000's pattern: tile content + small-print explanation of scope. The PRD-001 footnote also reminds operators where to change language ("Switch language in the Full Page surface").

### Critical user flows

#### Flow 1 — Switch language (US-1)

1. Operator clicks the picker trigger in the topbar.
2. `@blok/dropdown-menu` opens. Site-scoped languages listed first; if fallback applied, a `Separator` and "All tenant languages" group below.
3. Operator clicks a language. Dropdown closes; topbar picker label updates immediately.
4. List-pane shows `@blok/skeleton` rows for 200–1500ms while re-fetch completes.
5. List re-renders with new language data; per-row dots reflect the new language.
6. Detail-pane (if a map was selected) re-fetches in parallel — if the map has no version in the new language, transitions to the no-version state (F3).
7. `localStorage` writes the new picker state. Dashboard Widget tab (if open) sees the `storage` event and re-renders.

**Disabled state.** If any field-edit is dirty in the detail-pane (per AC-3.5), picker trigger is disabled (`aria-disabled="true"`) with tooltip "Save or discard your changes before switching languages." Clicking does nothing.

#### Flow 2 — Create new language version, empty path (US-6)

1. Operator selects a language with no version → lands on F3 (no-version state).
2. F3 shows two CTAs: "Create empty [language] version" + "Copy from another language". Also lists "Languages with versions: en, de" so operator can switch instead of create.
3. Operator clicks "Create empty [language] version".
4. Modal F5 opens — but with empty-start side highlighted / focused.
5. Operator confirms (clicking the empty-start CTA again, or pressing Enter on the focused CTA).
6. `addRedirectMapVersion` mutation runs. `@blok/skeleton` overlay on modal during the call.
7. On success: modal closes, detail-pane transitions to F4 (empty version, inline "Add first mapping").
8. Toast: *"Created empty `[language]` version."*

#### Flow 3 — Create new language version, copy-from path (US-7)

1. Operator on F3 clicks "Copy from another language".
2. Modal F5 transitions to step 2 = F6 (source picker).
3. F6 shows a `@blok/dropdown-menu` (or inline list) of only **non-empty-version languages** per AC-7.2. If only one such language exists, it's pre-selected.
4. Operator picks source language → clicks "Create".
5. `addRedirectMapVersion` runs → success → `updateItem` populates `UrlMapping` from source.
6. Per ADR-0021 state machine: success path → modal closes, detail F2 with mappings pre-filled. Toast: *"Created `[language]` version from `[source]`."*
7. Populate-failure → rollback `deleteItemVersion` → on rollback success, toast *"Could not copy mappings — new version removed"* + collapsible technical details.
8. Rollback failure → state `partial-version-detected` → F9 warning banner persists at Full Page top until operator clicks one of the CTAs.

**UI hint copy** per AC-7.4 displayed in F6: *"Mappings will be copied from [source language]. Flags and redirect type are shared across all languages and apply automatically."*

#### Flow 4 — Delete with explicit scope (US-8)

1. Operator clicks "Delete" on a map in the detail header overflow menu.
2. Modal F7 opens — `@blok/dialog` with `@blok/radio-group`.
3. Modal title: *"Delete `[map display name]`?"*
4. Body: *"This map has 3 language versions: en, de, fr."*
5. Two radios, vertically stacked, **no default selected**:
   - "Delete `[current language]` version only — other languages remain."
   - "Delete entire map and all language versions."
6. Delete button is disabled (`disabled` attribute + `aria-disabled="true"`) until a radio is checked.
7. Single-version edge case (AC-8.6): both radios still shown; below the radios, informational note: *"This map has only one language version; both options remove all redirect data. Choose explicitly to confirm intent."*
8. Hard-fail variant (AC-8.4, R-1): if `deleteItemVersion` is absent, only option (b) is shown with adjusted copy.
9. Operator picks a radio → Delete enables → clicks Delete → mutation runs → modal closes.
10. Toast: *"Deleted `[language]` version of `[map name]`."* OR *"Deleted `[map name]` and all language versions."*

#### Flow 5 — Edit display name (US-9)

1. Operator hovers the map title in F2 detail header.
2. Pencil affordance (Lucide `Pencil` icon, `size-4`, `text-muted-foreground`) appears beside the title with a 100ms fade.
3. Operator clicks pencil → title swaps to `@blok/input` with current display-name value (or empty + placeholder `_(no display name for [language] — add one)_` per AC-9.2).
4. Operator types. Two affordances visible: "Save" (`@blok/button` variant=primary, size=sm) and "Cancel" (`@blok/button` variant=ghost, size=sm).
5. Enter → Save. Click Save → Save. Escape → Cancel. Click Cancel → Cancel. **Blur does NOT commit** (AC-9.4).
6. On Save: `updateItem` runs with `language: <current>` and `fields: [{ name: '__Display name', value: <new> }]`. Skeleton on the input during the call.
7. Success: input swaps back to static title with new value. Toast: *"Saved display name to `[language]` version."*
8. Failure: input stays in editing state; Sonner toast with friendly summary + collapsible technical details.

**Mobile / touch interaction (AC-9.1 derivative):** On touch devices, pencil is always visible (no hover). Touch the title or the pencil — either invokes edit mode.

#### Flow 6 — Import multilingual bundle (US-10, US-11)

1. Operator clicks "Import" in topbar actions → F10 wizard step 1.
2. Drop file or pick. v1 detection runs immediately — if `schema === "redirect-manager/v1"`, show clear error per FR-13 inline in step 1 (no advance).
3. Validate (F11): Zod parse. If errors, list them with file-line context. If pass, advance.
4. Preview (F12): table renders.
   - Rows = maps (ordered as in bundle).
   - Columns = `bundleSiteLanguages` ∪ `tenantSiteLanguages` (union — operator sees every language present anywhere). Horizontal scroll via `@blok/scroll-area` when column count > viewport.
   - First column (sticky-left): map name + GUID + per-map action picker (`@blok/select` with 3 options: create / overwrite / skip — wired to map-level action per FR-16).
   - Cell content per (map, language) intersection:
     - `[+] create` — language in bundle, no version on tenant. Status badge: `--success-background` + `--success-foreground` text, label "Create".
     - `↻ overwrite` — language in bundle AND on tenant. Status badge: `--warning-background` + `--warning-foreground`, label "Overwrite".
     - `· skip` — map's action is "skip" OR (cell has no bundle entry). Status badge: `--muted` + `--muted-foreground`, label "—".
     - `· unchanged` — language on tenant, NOT in bundle, map's action is "overwrite". Status badge: `--muted`, label "Unchanged".
5. Footer of preview step: counts ("12 creates, 3 overwrites, 8 skips, 5 unchanged"). "Apply" button.
6. Apply (F13): progress bar (`@blok/progress`) + streaming summary list as each map completes. Per-map outcome with per-language detail per AC-11.5: *"Map X: en updated, de created, fr unchanged."*

Blok ships standard breakpoints but Marketplace iframe widths matter more than device categories:

| Breakpoint | Width | Behavior in V1 |
|---|---|---|
| `mobile` (theoretical) | <640px | Topbar collapses logo to mark-only; picker stays full-size. Two-pane collapses to single-pane. Almost never seen in Marketplace context — included for completeness. |

- At <480px wide: tiles stack vertically.
- At 480–800px wide: tiles in 1 × 3 horizontal row.
- At >800px wide (theoretical, Dashboard host caps width): same — no benefit to wider layout.

**Context Panel** (always narrow per `blok-marketplace-integration`): single-column layout, banner full-width inside panel, no breakpoint changes.

#### Touch targets

- Picker trigger: 48px tall × ≥160px wide (exceeds 44×44 minimum per WCAG 2.5.5 AAA).
- Map list row: 48px tall, full-width clickable (the dot + map name + chevron all advance into the same click target).

- Radio in delete-scope modal: full-row clickable (label clickable, not just the radio dot).
- Buttons in modals: 40px tall × variable width with min 96px.

| **Apply complete** | Each row's action cell replaced with `@blok/badge` variant=success or error showing final outcome. |

**Long-list relationship-visibility check:** The import preview is a single table — rows-to-columns binding is preserved by the table structure itself. At 50+ maps × 10+ languages, sticky-left first column anchors map identity while horizontal scroll moves through languages. Operator never loses track of which row binds to which language column because the first column stays put. ✓ Passes the check. **Bonus protection:** if column count exceeds 8 (some tenants have 15+ languages), V1 adds a "language filter" `@blok/dropdown-menu` above the table to hide/show columns. Filter state is session-local (not persisted).

**Accessibility:**

- Table has `<caption>` describing the preview.
- Header cells have `scope="col"`.
- Per-map action picker has `aria-label="Action for [map name]"`.
- Cells with conflict status have `aria-label="[Language] [status]: [explanation]"` so screen-reader users get the full context per cell.

### 4.8 Topbar — actions overflow

**Composes from:** `@blok/dropdown-menu` (Trigger + Content with grouped items).

Actions (right side of topbar):

- New Map → opens existing `NewRedirectMapModal` from PRD-000.
- Import → opens import wizard F10.
- Export → triggers export with 3 sub-actions (open in new tab / copy to clipboard / download) per AC-10.5.
- Refresh → re-fetches map list in current language.

**Visual:** At `lg`+ breakpoints, "New Map" is a standalone `@blok/button` (variant=primary, size=md). Import / Export / Refresh collapse into a single overflow `@blok/dropdown-menu` triggered by Lucide `MoreHorizontal` icon. At `md` breakpoint, "New Map" also collapses into the overflow menu.

### Iframe-specific breakpoint behavior (per `blok-marketplace-integration`)

- **Dashboard Widget (iframe 300–800px wide)**: V1's dashboard always renders below `lg` breakpoint. Tile layout is custom (auto-fit grid), independent of two-pane logic.
- **Context Panel (iframe 320–400px wide)**: V1's context panel always renders below `sm`. Single-column layout.

### Accessibility — WCAG 2.1 AA conformance plan

#### Perceivable (1.x)

- **Color contrast (1.4.3 AA)** — All text-on-background pairs verified against Blok semantic tokens in both light and dark modes. Spot checks:
  - `--primary` (`#6e3fff`) on `--primary-foreground` (`#ffffff`) → 6.4:1 ✓
  - `--foreground` on `--background` → ≥7:1 ✓
  - `--muted-foreground` on `--background` → 4.6:1 ✓
  - `--destructive` on `--destructive-background` → verified inline per badge style
- **Non-text contrast (1.4.11 AA)** — Picker border, focus ring, dot indicators all use tokens with ≥3:1 contrast against background.
- **Text resize (1.4.4 AA)** — All typography in `rem` units; respects browser zoom up to 200%.
- **Reflow (1.4.10 AA)** — Two-pane collapses to single-pane at <1024px; no horizontal scroll required up to 320px width (per `blok-marketplace-integration` Context Panel minimum). Import preview table is the one exception (intentional horizontal scroll for the language axis); covered by ARIA + sticky-left column for orientation.
- **Color not sole indicator (1.4.1 AA)** — Dot indicator filled/outlined is **shape-coded**, not color-coded (both states use `currentColor` — the difference is fill vs stroke). Status badges in import preview pair color with text label.

#### Operable (2.x)

- **Keyboard accessible (2.1.1 A)** — All interactive elements reachable and operable via keyboard. Picker, modals, table action pickers, inline editors, banners (no interaction needed).
- **No keyboard trap (2.1.2 A)** — Modals trap focus while open (Radix default); Escape always exits.
- **Focus order (2.4.3 A)** — Tab order follows visual order: topbar (logo → picker → actions) → list-pane → detail-pane.
- **Focus visible (2.4.7 AA)** — Every interactive element has a `:focus-visible` style. Blok primitives ship with this; V1 components inherit. **Structural guard from PRD-000 extends** — banning `outline: none` without paired `:focus-visible` rule in the codebase.
- **Target size (2.5.5 AAA — V1 aims for, not required)** — Primary targets ≥48×48. Pencil affordance is the only sub-target; mitigated by row-level clickable area.

#### Understandable (3.x)

- **Language (3.1.1 A)** — `<html lang="en">` on the app shell. The app's UI is en-only per PRD § 5 Out of Scope ("App UI localization" deferred). Content language is per-version, not surface-language.
- **Predictable navigation (3.2.x AA)** — Picker change re-fetches list/detail but does NOT navigate. URL is stable. No surprise context changes.
- **Error prevention (3.3.x AA)** — Delete confirmation modal (ADR-0018) + dirty-edit-protection (AC-3.5) + import preview step before apply.

#### Robust (4.x)

- **Compatible (4.1.x AA)** — Semantic HTML throughout. Blok primitives ship with ARIA per Radix. V1 adds ARIA only where it complements (e.g. fallback-mode `aria-describedby` on picker, dot state in `aria-label` on row).

### Reduced motion

- `@media (prefers-reduced-motion: reduce)` — all 120ms transitions reduce to 0ms. Picker dropdown opens instantly. Pencil affordance appears without fade. Toasts still appear but without slide-in animation.
- Blok primitives respect this via Radix; V1 components inherit.

### Dark mode

- All V1 components use Blok semantic tokens (`--background`, `--foreground`, `--primary`, etc.) and flip automatically under `.dark` class on `<html>`.
- Explicit verification in V1: the dot indicator's filled state uses `currentColor` → inherits row text color → correct in both modes. The outlined state uses `currentColor` for stroke → also correct in both modes.
- Picker chip's `--primary-background` (`#eae7ff` light, dark token in `.dark` mode) verified against `--primary` text in both modes. **Spot-check at /implement**: the Nova preset has shipped with `--primary-foreground` collapsed onto `--primary` in dark mode at least once (per `blok-theming` skill note + memory `project_quickcopy_framework_learnings`). V1 prefers `text-foreground` on the chip rather than `text-primary-foreground` to side-step this risk.

### RTL

- V1 does NOT explicitly target RTL locales — the app UI is en-only per PRD § 5 Out of Scope.
- Layout uses logical properties where Blok primitives use them (most do per `blok-theming` § RTL).
- Custom V1 layout (topbar flex, two-pane grid) is rendered with logical CSS by default (`flex` + `gap` + `grid-template-columns`) so RTL would work as a side benefit if needed later.
