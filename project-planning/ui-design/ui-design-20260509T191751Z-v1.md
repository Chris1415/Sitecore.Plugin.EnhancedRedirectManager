> **Reading note.** This is variant **1 of 3** for the Redirect Manager UI. The named direction is **"Operator Console"** — dense, info-rich, table-first. The aesthetic lineage is the Sitecore **Content Editor** + **Pages** chrome the operator already knows. Power-user energy: monospaced for IDs/paths/redirect counts, compact-but-readable rows, lots of inline metadata, badges as the primary status grammar. All component names below reference real Blok primitives or composite blocks via `@blok/*` registry slugs. All colors and spacing use **Blok semantic tokens** — never raw hex, never invented token names.

## 1. Design context & assumptions

### 1.1 — Audience (hybrid by surface)

The PRD names three personas with three different comfort levels. v1 honors that with **per-surface tone** while staying inside one cohesive direction:

- **Context Panel — softer Operator Console.** The editor is a non-technical marketer mid-task in Pages. The panel is dense but not technical: monospace is reserved for the URL fields (where it earns its keep — paths are the data); chrome around it is sentence-case body copy with friendly labels. Group headers carry the Redirect Map name as a `--text-sm` label, not as a code-style identifier.

- **Dashboard Widget — neutral Operator Console.** Site managers glance at it daily. It is a three-tile counter card, neither dense nor decorative. Numbers are big and tabular; supporting copy is muted body text. No power-user affordances at all in this surface.

- **Full Page — full Operator Console.** Admins and implementers live here. This is where the dense, table-first, monospaced lineage gets its full expression: GUID badges visible on hover, item-tree paths in monospace, mapping rows rendered as a compact two-column table, last-updated timestamps in tabular figures, virtualized scroll for 500+ row sets, drag handles on every reorder-able row.

- **Anti-persona:** the casual editor who wandered into the Full Page hoping for a "fix my links" wizard. v1 does not water down for them — the Context Panel is their surface, and the Full Page is honest about being a tool for people who understand redirect maps.

### 1.2 — Goals and success criteria

| Goal (PRD § 3) | What this design must support |
|---|---|
| **G1** — eliminate Pages → Content Editor jump | Context Panel surfaces matched redirects **grouped by parent Redirect Map** with per-row inline edit/delete. Operator never leaves Pages. |
| **G2** — per-site at-a-glance overview | Dashboard Widget renders three stat tiles in ≤2 s (NFR-P3). Tabular numbers, tabular alignment. |
| **G3** — single management surface | Full Page two-pane layout: collection/site picker on the left, redirect content on the right. Power-user CRUD for items and mappings. |
| **G4** — portable redirect sets across environments | Export button emits `redirect-manager/v1` JSON. Import flow has a **diff-aware preview screen** with per-conflict action picker (create/overwrite/skip — ADR-0006), bulk action, disabled confirm-until-resolved, and a final summary screen. |

Acceptance metrics this v1 must support unchanged:
- **m2 — host-frame visual smoke** — POC clickdummy will be the visual ground truth at `/test`.
- **m3 — CRUD round-trip ≤10 s end-to-end** — the form UI must give immediate feedback (optimistic Blok skeleton on resubmit, success toast on settle).
- **m4 — import/export round-trip with zero rule loss** — the import preview's diff and the action picker carry the entire correctness contract.
- **m5 — ≥5-min editor live walkthrough with zero unrecoverable errors** — friendly error UX (ADR + PRD FR-13) with technical details collapsed.

### 1.3 — Constraints (apply across this variant)

**Brand / design system (mandatory):**
- All UI is **Blok** via shadcn registry — `@blok/*`. Tokens come from `theme.json`. **No** hand-picked hex. **No** invented `--blok-*` token names. **No** raw Google Fonts — Blok ships Geist Sans + Geist Mono via the Nova preset (the `--font-sans` / `--font-mono` tokens published by `next/font/google` per `blok-setup` "Geist already ships").
- **Light + dark mode parity** is automatic via Blok semantic tokens. Honors `.dark` class set by Cloud Portal — never forces a mode (NFR-C2 + Blok marketplace integration § "Theming inside iframes").

**Architecture-locked (cannot revisit at design time):**
- **Three iframe routes.** `app/context-panel/page.tsx` → `xmc:pages:contextpanel`. `app/dashboard-widget/page.tsx` → `xmc:dashboardblocks`. `app/full-page/page.tsx` → `xmc:fullscreen`. Root `/` returns `notFound()` (ADR-0011 / architecture § 6.4).
- **Single Authoring GraphQL backend.** No second store. Loading and error states reflect that — there is exactly one source of truth and one error class (the friendly Blok error banner with collapsed technical details).
- **MVP is `en`-only** (ADR-0010). v1 does not show a language switcher anywhere. The PRD-001 multilingual surface will add it later — leave a stable spot for it (top-right of the Full Page detail header, currently empty).
- **Context Panel matches direct strings only** (ADR-0005). The non-dismissible banner is a load-bearing affordance, not optional.
- **`UrlMapping` round-trip is lossless** (ADR-0008). Operator-defined row order is authoritative. Drag handles on every mapping row in the Full Page detail.
- **Import matches by Sitecore item GUID** (ADR-0009). The diff UI must show the GUID for every row; "create vs overwrite vs skip" attaches to the GUID classification.
- **List virtualization via `react-virtuoso`** (ADR-0012). The redirect-map list and the import preview list both virtualize.

**Iframe constraints (per `blok-marketplace-integration`):**
- Context Panel: 320–400 px wide, scrollable. **No topbar** — too tall.
- Dashboard Widget: 300–800 px × 200–400 px. Single widget, no internal navigation.
- Full Page: 1024–1920 px × 600–1080 px. Two-pane layout fits at ≥1024 px; tabbed fallback at <960 px.
- Shadows clip at iframe edges — prefer `shadow-sm` on cards and panels. **No** large drop shadows.

**Accessibility (NFR-A1 → NFR-A5):**
- WCAG **2.1 AA** across both modes (Blok tokens already meet AA at standard combinations).
- Full keyboard operability. `Tab` order follows reading order. `Enter` activates row edit. `Esc` closes inline forms (with the unsaved-changes confirm prompt per FR-12). Arrow keys navigate within the redirect-map list.
- `aria-live="polite"` on the success toast + the loading-row count announce. `aria-live="assertive"` on the error banner.
- Visible `:focus-visible` everywhere — Blok default ring (`ring-ring`).
- Color is **never** the only signal. `RedirectType` badges carry the type label (`301`, `302`, `307`, `Server Transfer`); flag chips (`Pres. QS`, `Pres. Lang`, `Virt. Folder`) carry the flag text; success/error states pair color with monochrome glyphs (`✓`, `✕`) per `blok-theming` "Color-emoji codepoints".

### 1.4 — Reference / inspiration

The "Operator Console" direction draws on three real-world idioms — adapted, never copied:

- **Sitecore Content Editor + Pages chrome (the Cloud Portal lineage operators already know).** Sentence-case section labels, hairline separators between groups, badge-style metadata on items, monospace for paths. The Full Page reads like a focused, modernized slice of Content Editor's "Settings/Redirects" tree — but with proper inline editing, drag-reorder, and a real diff UI that Content Editor never had.
- **GitHub diff viewer + PR file tree.** The import-preview diff section borrows the per-conflict expand/collapse + side-by-side field comparison. Adds vs removes are colored with `--destructive`/`--success` semantic accents, never ornamental green/red. The "Apply same action to all conflicts" affordance is a `@blok/select` that updates every row's action picker simultaneously — same idiom as GitHub's "Mark all as viewed".
- **Linear issue list density.** Single-line item rows in the redirect-map list, hairline separators, monospace metadata at right (mapping count + last-updated timestamp), a left-edge accent stripe on the currently-selected map (the only color in the resting state). The eye scrolls a Linear list at high speed by reading the leftmost label and the rightmost number; we map the same affordance to the redirect-map list.

**Web-search synthesis (light, summarized):** modern CMS power tools converging in 2025 (Strapi, Sanity Studio, Payload CMS) consistently use: badge grammar for type/state, monospace for IDs and paths, hairline separators between list rows, drag handles for reorder-able rows, virtualized lists for >100 items, friendly error banners with collapsed technical details, and per-row inline edit (no full-page form for trivial fields). The Operator Console adopts this dialect and grafts it onto Sitecore's Blok primitives.

**No external assets.** No imported icons beyond Lucide (already in Blok dependency tree per `blok-theming`). No screenshots, no copyrighted glyphs.

### 1.5 — Answers to clarifying questions

These were locked by the variant brief — recorded for traceability:

- **Count and divergence:** 3 variants, moderate divergence. This is variant 1 (dense, info-rich, table-first). Variants 2 and 3 are produced by parallel sub-agents (different aesthetic directions) — this spec does not refer to them.
- **Audience tone:** hybrid by surface (above § 1.1).
- **Density:** standard Blok defaults — ~8 rows per visible viewport in the Full Page list, ~6 mapping groups visible in the Context Panel.
- **Theme:** light + dark, automatic via Blok tokens. Respect host preference.
- **Browser support:** Chrome / Edge / Firefox / Safari — latest two majors.

### 1.6 — Assumptions made

- `pageInfo.url` is the matcher key in the Context Panel (architecture OQ-A pending real-tenant verification). The design renders both the panel-current-URL and the matched mapping rows in the same monospace style so the eye can verify "yes, this is the page" at a glance even if the resolution of OQ-A flips to `pageInfo.route` — only the variable name changes, not the visual.
- `RedirectType` enum values beyond `ServerTransfer` are TBV (OQ-8). The dropdown options are sourced from real-tenant introspection at architecture-stage (per ADR-0013); the design **assumes** the values render as `301`, `302`, `307`, `Server Transfer` and leaves room in the badge for any of those four widths. Empty initial state is mandatory (PRD US-7 AC).
- The "Add redirect for this page" affordance in the Context Panel uses a `@blok/dialog` modal sized to the panel viewport (320–400 px). It is **not** a popover — the modal has its own focus trap and `Esc` close, and the panel's narrow height makes a popover too small for the searchable list.
- The Full Page two-pane layout defaults to a 280 px left rail / fluid right pane at iframe widths ≥1024 px. At 800–960 px the right pane gets too narrow to host both the redirect list AND the detail editor, so v1 falls back to a tabbed layout (`Picker | Maps | Detail` tabs at the top — `@blok/tabs`).
- The drag-and-drop reorder primitive is `@dnd-kit/core` (architecture § 7), composed with `@blok/draggable` from the primitive catalog. `@blok/draggable` provides the visible drag-handle affordance; `@dnd-kit` provides the keyboard-accessible reorder behavior (`Space` to grab, arrow keys to move, `Space` to drop).
- The Dashboard Widget tiles are not interactive in MVP. PRD § 11 says "clicking a tile may link out to the Full Page"; v1 leaves the tiles non-interactive (no hover affordance, no cursor:pointer) so the widget has no implicit promise to keep — when PRD-001 enriches the tiles with breakdowns, *then* hover affordance gets added. Documented as an open question (Q4 below).

## 2. Design direction

- **Visual direction — "Operator Console":** a dense, info-rich, table-first dialect of Blok. Reads like a CMS power tool. Monospace for the data that *is* paths and IDs (URLs, GUIDs); body sans-serif for everything else (item names, copy, labels). Badge grammar for `RedirectType` and the three flags. Hairline `@blok/separator` between rows. Compact-but-readable row rhythm (~36 px). One narrow accent (left-edge stripe in `--primary`) on the currently-selected item in the Full Page list — that is the only color in the resting state. Numbers (mapping counts, item counts, timestamps) render in tabular figures via Blok's `--font-mono` token so columns line up. Aesthetic neighbors: Sitecore Content Editor's `Settings/Redirects` tree, Linear's issue list, GitHub's PR file tree.

- **Layout and structure (high-level — three surfaces):**
  - **Context Panel** — single vertical column, no topbar. Persistent banner at top → page-context callout → grouped match list → "Add redirect for this page" button. All inside one outer `@blok/card`.
  - **Dashboard Widget** — outer `@blok/dashboard-widget` frame (composite blok), three stat tiles in a horizontal row at ≥600 px width (stacks vertically below 480 px), single footnote line at the bottom.
  - **Full Page** — two-pane layout. Left rail (280 px fixed): collection picker + site picker + redirect-map list (virtualized). Right pane (fluid): detail editor for the selected map (form fields + mappings table with drag-reorder), or empty state, or import preview, or export confirmation. Top row: a `@blok/breadcrumb` showing `Collection / Site / Map name`, plus action buttons (`Import`, `Export`, `+ New map`).

- **Key components and patterns:**
  - **Blok primitives (installed via `npx shadcn@latest add @blok/<name>`):** `@blok/card`, `@blok/badge`, `@blok/button`, `@blok/separator`, `@blok/skeleton`, `@blok/scroll-area`, `@blok/alert`, `@blok/collapsible`, `@blok/dialog`, `@blok/dropdown`, `@blok/select`, `@blok/input`, `@blok/textarea`, `@blok/checkbox`, `@blok/label`, `@blok/breadcrumb`, `@blok/tabs`, `@blok/tooltip`, `@blok/sonner` (toast), `@blok/draggable`, `@blok/empty-states`, `@blok/error-states`, `@blok/icon`, `@blok/popover`.
  - **Blok composite blocks (bloks):** `@blok/dashboard-widget` for the Dashboard surface frame.
  - **External libs already locked by architecture § 7:** `react-virtuoso` (list virtualization), `@dnd-kit/core` (mapping reorder), `react-hook-form` + `zod` (forms), `lucide-react` (icons — already in Blok tree).
  - **Recurring patterns:**
    - **Badge grammar.** `RedirectType` always renders as a badge: `301` / `302` / `307` use `variant="outline"` with `--foreground` border, `Server Transfer` uses `variant="secondary"`. The visual weight is identical across all four — type is information, not status. Flag chips render only when the flag is `true`, all in `variant="outline"` with `--muted-foreground` border (`Pres. QS`, `Pres. Lang`, `Virt. Folder`).
    - **Monospace metadata column.** In the redirect-map list and the mapping-table rows, numbers and paths align right in monospace so the eye scans columns. Item names and labels stay in the sans-serif left column.
    - **Left-edge accent stripe (2 px in `--primary`)** on the currently-selected redirect map in the Full Page list. The only color in the resting state. Adapts to dark mode automatically.
    - **Inline collapse.** Redirect-map rows expand in place to show their mapping table — no full-row swap, no modal. `@blok/collapsible` with `@blok/draggable` reorder *only on the inner mapping rows*, never on the outer redirect-map list (PRD doesn't ask for map reorder).
    - **Friendly error UX.** Errors render as `@blok/alert` with `variant="destructive"`: friendly summary line + collapsible "Show technical details" containing the verbatim GraphQL error + retry button. The `✕` glyph is monochrome (U+2715) inheriting `--destructive` via `currentColor` — never `❌` U+274C (color-emoji bitmap, ignores theme — see `blok-theming`).
    - **Optimistic UI on writes.** Mapping edit/delete and item create show a brief skeleton state on the affected row(s) for 200–300 ms while the GraphQL mutation is in flight, then settle to the new state with a success toast. No full-pane reload.

- **Token discipline (Blok semantic tokens — every color reference below uses these names; no hex):**
  - Surfaces: `--background`, `--card`, `--popover`.
  - Text: `--foreground`, `--muted-foreground`, `--card-foreground`.
  - Interaction: `--primary` + `--primary-foreground` (verify pair contrast in dark mode per `blok-theming` pitfall — Nova has shipped the pair collapsed once); fall back to `text-background` on `bg-primary` if the pair is collapsed.
  - State: `--destructive` + `--destructive-foreground` (errors), `--success`-equivalent via `--accent` (success toasts — Blok's success channel rides through `--accent`/`--accent-foreground` if the dedicated success token isn't published).
  - Borders / dividers: `--border`, `--input` (inputs), `--ring` (focus ring).
  - Typography scale: `--text-xs`, `--text-sm`, `--text-base`, `--text-lg`. Item names use `--text-sm` weight 500; URLs use `--text-xs` `--font-mono`; large widget numbers use `--text-2xl` `--font-mono` tabular figures.
  - Radius: `--radius-sm` (badges, buttons), `--radius-md` (cards, panels). No `--radius-lg` — too soft for an operator tool.
  - Shadows: `shadow-sm` only. **No** larger shadows — they clip at iframe edges per Marketplace integration constraints.

## 3. Key screens and flows

This variant covers **three extension-point surfaces × N states each**. Every state below gets a clickdummy frame in the POC.

### 3.1 — Context Panel (`/context-panel`) — 320–400 px wide

**Purpose** (PRD US-1, US-2, US-3): for the page being edited in Pages, list redirects that affect it (page is source OR target of a direct-string match), grouped by parent Redirect Map. Inline create/edit/delete.

**Layout (vertical stack, top → bottom inside a single outer `@blok/card`):**

1. **Persistent banner** (`@blok/alert`, `variant="default"`, non-dismissible) — `--text-xs`, friendly copy: "Direct-string matches only — regex pattern matches are not yet covered." Full-width inside the panel gutter (12 px). This is the load-bearing affordance for ADR-0005.
2. **Page-context callout** — small block: monochrome location indicator icon (Lucide `MapPin` at `--muted-foreground`) + the page's published URL in `--font-mono` `--text-sm` `--foreground`. Reads "On `/foo/bar`". One hairline `@blok/separator` below.
3. **Match list** — the body. Grouped by parent Redirect Map. Group header + matched mapping rows beneath (see § 3.1.1).
4. **"+ Add redirect for this page" button** — pinned at the bottom of the scroll-area as a low-emphasis `@blok/button` (`variant="outline"`, full-width). Opens the Add modal (§ 3.1.2). A small hairline separator above it so it never blends into the last group.

**Outer surface:** **no topbar** (Context Panel is too narrow per `blok-marketplace-integration`). Outer `@blok/card` provides the frame; inside it, a vertical `@blok/scroll-area` handles overflow.

#### 3.1.1 — Group anatomy (per Redirect Map)

A group is rendered as a stacked block with no per-group card chrome (groups are visually separated by hairline `@blok/separator`, not by individual cards):

- **Header row** (single line, ~32 px):
  - Left: Redirect Map name in `--text-sm` weight 500.
  - Right (tabular, monospace): `RedirectType` badge + flag chips (`Pres. QS`, `Pres. Lang`, `Virt. Folder` — only chips for flags currently `true`). Chips wrap to a second line at ≤320 px viewport.
- **Matched mapping rows** (one per match, ~28 px each):
  - Left: source URL → target URL, in `--font-mono` `--text-xs`. The arrow glyph is `→` (U+2192) at `--muted-foreground`. The matched side (whichever side equals the current page URL) has `--foreground` weight 500; the other side has `--muted-foreground`.
  - Right: two icon-buttons — `@blok/button` `size="icon" variant="ghost"`: pencil (Lucide `Pencil`) for edit, trash (Lucide `Trash2`) for delete. Visible on row hover and always visible on keyboard focus (`:focus-visible`).
- Hairline `@blok/separator` between groups.

#### 3.1.2 — "Add redirect for this page" modal

`@blok/dialog`, sized to the panel viewport (full-width inside the iframe; ~320–400 px wide × auto height). Two-step flow:

- **Step 1 — pick a map.** Title: "Add a redirect for `/foo/bar`". Body: a search-filterable list rendered with `@blok/command` (the searchable command-palette primitive — gives keyboard navigation for free). First option in the list is always **"+ Create new Redirect Map"** with a Lucide `Plus` icon. Below it: existing Redirect Maps in the current site, each row showing name + last-updated timestamp (right-aligned, monospace tabular figures, `--muted-foreground`).
- **Step 2a — picked an existing map.** Body switches to an inline form: source field (pre-populated to current page URL, read-only with a subtle hint "Source = current page" below), target field (`@blok/input`, required, monospace), a recap line at the top ("Adding to: My Redirect Map · 301 · Pres. QS"). `RedirectType` and flags are inherited from the chosen map — not editable in this path. Footer: `Cancel` (ghost) + `Save mapping` (primary).
- **Step 2b — picked "Create new Redirect Map".** Body switches to a fuller form: new map name (`@blok/input`), `RedirectType` (`@blok/select`, **initial state empty** — first option is a disabled placeholder "Pick a type…"), three flag checkboxes (`@blok/checkbox`), then the first mapping (source pre-populated, target required). Inline validation per FR-11 (block save on empty source/target/RedirectType). Footer: `Cancel` + `Create map`.

#### 3.1.3 — Inline edit on a mapping row

Click pencil → the row inflates inline: source becomes editable, target becomes editable, two compact buttons appear at the right end of the row (`Save`, `Cancel`). No modal. Inline validation (block save on empty source/target). Pristine-cancel closes silently; dirty-cancel triggers a confirm prompt (FR-12).

#### 3.1.4 — Inline delete on a mapping row

Click trash → small inline confirm replaces the row content for 5 seconds: "Delete this mapping? `Yes` `No`". `Yes` triggers the delete; `No` reverts. (Avoiding a separate modal keeps the panel calm.)

#### 3.1.5 — States

| State | Anatomy |
|---|---|
| **default** | Banner + page-context callout + 1+ match groups + "Add" button at bottom. Hairline separators between groups. |
| **loading** | Banner + page-context callout + **3 skeleton group blocks**: each is a single skeleton header row (`@blok/skeleton` 16 px tall) over 2 skeleton mapping rows (12 px tall each). No "Add" button while loading (avoid an early-action trap). |
| **empty** | Banner + page-context callout + `@blok/empty-states` block: small icon (Lucide `Filter`, `--muted-foreground`), copy "No redirects affect this page", supporting copy "Add the first one with the button below". `+ Add redirect for this page` button below. |
| **error** | Banner + page-context callout + `@blok/alert variant="destructive"` with monochrome `✕` glyph, summary "Couldn't load redirects for this page.", collapsible "Show technical details" (verbatim GraphQL error in monospace), retry button. No add affordance (data isn't loaded — fix it first). |
| **focus** | Any focusable element shows the Blok `--ring` outline; row hover affordances (edit/delete buttons) are also revealed on focus, never solely on hover. |
| **success-toast** | After any successful write (create / edit / delete), a `@blok/sonner` toast slides up from the bottom-right of the iframe: monochrome `✓` glyph + friendly summary "Redirect added", "Mapping updated", or "Mapping deleted". Auto-dismiss in 3 s; click to dismiss earlier. |

### 3.2 — Dashboard Widget (`/dashboard-widget`) — 300–800 px × 200–400 px

**Purpose** (PRD US-4): per-site at-a-glance counters.

**Layout:**

The outer surface is `@blok/dashboard-widget` (composite blok). Inside it:

- **Title row** — "Redirects" in `--text-base` weight 600. Right-aligned subdued caption: site name in `--text-xs` `--muted-foreground` (e.g. "MySite · MyCollection").
- **Stat tile row** — three tiles, equal-width, separated by hairline `@blok/separator` (vertical, half-height). At widths <480 px the row stacks vertically (mobile-fallback grid).
  - **Tile 1 — Redirect Maps.** Big number in `--text-3xl` `--font-mono` tabular `--foreground`. Label below in `--text-xs` `--muted-foreground`: "Redirect maps".
  - **Tile 2 — Mappings.** Same anatomy. Label: "Total mappings".
  - **Tile 3 — Last updated.** Big text in `--text-base` `--font-mono` tabular `--foreground` ("3 days ago" or "13:42 today" — relative for recent, absolute for older). Label below: "Last updated".
- **Footnote line** — at the bottom of the widget, `--text-xs` `--muted-foreground`: "Redirect counts only — usage analytics ship in a follow-on release." (PRD US-4 AC.)

**v1 decision:** tiles are **non-interactive** in MVP. Hover does nothing. Keyboard focus skips them. PRD-001 will enrich them with breakdowns (used vs unused, top-5 lists) — at that point hover affordance and click-through to the Full Page get added. Captured as Q4 below.

**States:**

| State | Anatomy |
|---|---|
| **default** | Title + 3 populated tiles + footnote. |
| **loading** | Title + 3 skeleton tiles (each: tall skeleton block where the number goes + thin skeleton line for the label) + footnote. |
| **empty** | Title + `@blok/empty-states` block (icon: Lucide `RouteOff`, `--muted-foreground`), copy: "No redirects configured for this site." Supporting copy: "Add some in the Redirects app to see counts here." Footnote stays. |
| **error** | Title + `@blok/alert variant="destructive"` (friendly + collapsible technical details + retry) replacing the tile row. Footnote stays. |
| **focus** | The retry button in the error state is the only focusable element by default; tiles get focus only when PRD-001 makes them interactive. |
| **success-toast** | Not applicable — the widget is read-only in MVP. |

### 3.3 — Full Page (`/full-page`) — 1024–1920 px × 600–1080 px

**Purpose** (PRD US-5 → US-10): the power-user surface. Site collection + site picker, full CRUD, JSON import/export.

**Layout (≥1024 px width — primary):**

A two-pane layout with a top action row:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  Breadcrumb: Collection / Site / Map name        [Import] [Export] [+ New]  │  ← top action row (~52 px)
├──────────────────┬──────────────────────────────────────────────────────────┤
│                  │                                                          │
│  Left rail       │  Right pane (fluid)                                      │
│  280 px fixed    │                                                          │
│                  │   Either:                                                │
```

**Layout (800–960 px fallback — tabbed):**

Three top-level tabs: `Picker` / `Maps` / `Detail`. Picker tab holds the collection/site dropdowns. Maps tab holds the virtualized list. Detail tab holds whatever the right pane would show. A "Back to maps" affordance at the top of the Detail tab. Tabs use `@blok/tabs`.

#### 3.3.1 — Top action row

- **Breadcrumb** (`@blok/breadcrumb`). "Collection / Site / Map name". Each segment is clickable when later segments are visible; the last segment is non-link `--foreground` weight 500. When no map is selected: "Collection / Site". When no site: "Collection". When nothing: empty (the rail's empty state takes over).

  - `Export` — `@blok/button variant="outline"`. Triggers JSON download for the current site. Disabled until a site is selected. On click: a small `@blok/sonner` toast confirms "Exported 24 redirect maps for MySite".

#### 3.3.2 — Left rail (280 px fixed) — picker + map list

- **Collection picker** (`@blok/select`) — full-width inside rail. Placeholder "Pick a collection". Shows all collections from `xmc.sites.listCollections`.
- **Site picker** (`@blok/select`) — full-width, disabled until a collection is picked. Shows sites filtered by the selected collection (`xmc.sites.listSites`).
- Hairline `@blok/separator`.
- **Redirect-map list** — virtualized via `react-virtuoso`. Each row is a single line (~36 px):

#### 3.3.3 — Right pane: detail editor (selected map)

This is the densest screen in v1. The detail screen is rendered as a vertical stack of three sections inside the right pane (no `@blok/card` wrapping — Full Page is already the card). Sections separated by hairline `@blok/separator` with `--text-xs --muted-foreground` section labels.

**Section 1 — Map attributes (form):**

- Item name — `@blok/input`, full-width, label "Name". Required.
- `RedirectType` — `@blok/select`, label "Type". **Initial state empty** for create (placeholder "Pick a type…"); pre-populated for edit. Options sourced from real-tenant introspection (architecture OQ-8). Required.
- Flags — three `@blok/checkbox` controls in a horizontal row (or stacked at <600 px right-pane width): `Preserve query string`, `Preserve language`, `Include virtual folder`. Labels in `--text-sm`.
- (Right side of the section, top-aligned) **Item GUID badge** — small monospace `@blok/badge variant="outline"` showing the Sitecore item GUID truncated to last 12 chars. Click → copies full GUID to clipboard, fires a toast "GUID copied". `@blok/tooltip` shows the full GUID on hover. This is a power-user affordance — operators shipping JSON imports between environments need the GUID visible.

**Section 2 — Mappings table:**

A compact two-column table with drag-reorder. Each row has:

- **Drag handle** (Lucide `GripVertical`, `--muted-foreground`, 16 px) at the far left. Keyboard-accessible via `@dnd-kit/core`: `Space` to grab, arrow keys to move, `Space` to drop. Tooltip "Reorder".
- **Source** — `@blok/input`, monospace, full-width-ish (40% column). Inline validation (red ring + `--destructive` hint text below: "Required") on empty.
- **`→`** glyph (U+2192) — `--muted-foreground`, decorative.
- **Target** — `@blok/input`, monospace, full-width-ish (40% column). Same validation.
- **Row actions** (icon buttons) — `Save`, `Cancel`, `Delete`. Visible on row hover/focus. `Save` is hidden when the row is pristine.

Below the last row: a `+ Add mapping` button (`@blok/button variant="ghost"`), full-width, `--text-sm`. Adds a new empty row in edit mode.

Table is virtualized via `react-virtuoso` once the row count exceeds 50 (PRD-S1 says ~500 mappings is the scale ceiling — we virtualize defensively).

**Section 3 — Bottom action row:**

- Left: `Delete map` button (`@blok/button variant="destructive-outline"`, opens the delete-confirm modal — see § 3.3.4). Hidden on create.

On success: toast "Redirect map saved" + refresh the rail. On failure: friendly error banner above the form with collapsible technical details + retry.

#### 3.3.4 — Delete confirmation modal

`@blok/alert-dialog`. Title: "Delete `<map name>`?" Body: "This will remove the map and all `<N>` mappings inside it. This cannot be undone." Footer: `Cancel` (`variant="outline"`) + `Delete` (`variant="destructive"`). On confirm: mutation fires; on success a toast "Map deleted" + the rail clears the selection.

#### 3.3.5 — Import flow

The import flow occupies the right pane (replaces the detail editor). Four-step linear flow with a sticky "Cancel" affordance at the top-right:

**Step 1 — Upload.** Title "Import redirects". A `@blok/dialog`-like surface with a single `@blok/input type="file"` accepting `.json`. Below it, a `@blok/alert variant="default"` reminder: "Tip: export the current site first as a backup before importing." On file selection → step 2.

**Step 2 — Schema validation.** If JSON parse or schema validation fails, an `@blok/alert variant="destructive"` shows the validation error in friendly copy + collapsible technical detail. `Cancel` (returns to detail) + `Choose another file`. No way to proceed past validation errors.

**Step 3 — Preview screen.** This is the load-bearing UX of the entire import flow. Layout:

- **Summary header:** "Importing 24 redirect maps into MySite (MyCollection)." Sub-line: "8 new · 14 conflicting · 2 unchanged" (counts by classification). Tabular monospace numbers. (Unchanged items are items where every field matches between source and target — they default to `skip` since no action is needed.)
- **Bulk action row:** `@blok/select` "Apply same action to all conflicts" with options `(no bulk action)` / `Create` / `Overwrite` / `Skip`. When set to a value, every conflicting row's action picker syncs. Initial value: `(no bulk action)` so the operator must look at each row.
- **Item list** (virtualized via `react-virtuoso`):
  - Each row is a single line (~44 px) with: classification badge (`New` / `Conflict` / `Unchanged` — variant accents per state), item name in `--text-sm` weight 500, GUID truncated last 12 chars in `--font-mono --text-xs --muted-foreground`, `Show diff` collapsible toggle (icon: `ChevronRight` rotates to `ChevronDown`), action picker on the right.
  - Action picker is **per row**: a `@blok/select` (small) with options `Create` / `Overwrite` / `Skip`. For `New` items: pre-set to `Create`. For `Unchanged` items: pre-set to `Skip`. For `Conflict` items: **initial value empty** (placeholder "Pick action…") — every conflict row shows a subtle red dot indicator (Lucide `AlertCircle` `--destructive` 12 px) until an action is picked, mirroring the disabled-confirm contract.
  - Click `Show diff` → row expands to reveal a compact field-by-field diff table:
    - Two columns: `Current` / `Incoming`. Rows: `Name`, `RedirectType`, `Pres. QS`, `Pres. Lang`, `Virt. Folder`, then a sub-table for `Mappings` (showing rows that exist in current but not incoming as "removed", rows new in incoming as "added", rows changed as "changed source/target", with monospace URLs and `--destructive`/`--success` accent on the changed sides).
- **Bottom action row:**
  - Left: `Cancel` (returns to detail).
  - Right: `Confirm import` (`@blok/button variant="primary"`). **Disabled until every conflict row has an explicit action picked** (matches PRD US-10 AC). When disabled, hovering shows a `@blok/tooltip` "Resolve all conflicts to enable import" with a count of unresolved conflicts.

**Step 4 — Apply + summary.** On confirm: a progress bar (`@blok/progress`) shows `n/N items applied` as the import runs. On settle: a final summary screen replaces the preview. Layout:

- Title: "Import complete" (or "Import completed with issues" if any failures).
- Three counter pills: `Created: N`, `Overwritten: N`, `Skipped: N`, plus `Failed: N` (only shown if non-zero, using `--destructive` accent).
- Detail list: only failed items are expanded (success items collapsed under a "View all" toggle to keep the failures eye-line). Each failed row shows: item name + GUID + collapsible technical error.
- Footer: `Done` (primary, returns to detail) + `Retry failed items only` (visible only if there are failures — re-runs the apply step targeting only the failed GUIDs).

### 3.4 — Critical user flows (cross-surface)

**Flow A — Marketer fixes a redirect from inside Pages.**
1. Open Pages, edit a page. Context Panel (this app) is in the sidebar. State settles to `default` showing 1 group (1 redirect map matched on this page's URL, source side).
2. Marketer sees the redirect points the wrong way. Clicks pencil on the matched mapping row.
3. Row inflates; target field becomes editable. Marketer types the new target. Inline validation passes. Clicks Save.
4. Brief skeleton on the row, then settles. Toast "Mapping updated". Authoring GraphQL mutation completes; `pages.reloadCanvas` fires (architecture § 5.6) so the canvas reflects the new redirect.

**Flow B — Site manager glances at the dashboard.**
1. Site manager opens the Cloud Portal dashboard. The Dashboard Widget tile shows "12 redirect maps · 84 total mappings · 2 days ago" + footnote.
2. They notice the count grew unexpectedly. They click out of the dashboard and navigate to the Full Page (separate Cloud Portal entry — the widget is read-only in MVP).

**Flow C — Admin promotes redirects from staging to prod.**
1. Admin opens Full Page on staging. Picks `MyCollection / MySite`. Clicks `Export`. JSON download fires. Toast confirms.
2. Switches to production tenant. Opens Full Page. Picks `MyCollection / MySite`. Clicks `Import`. Picks the JSON file.
3. Schema validates. Preview screen renders: "Importing 24 redirect maps · 8 new · 14 conflicting · 2 unchanged". Admin scrolls the list, expands a few diffs to verify changes. Picks "Overwrite" as the bulk action for conflicts (since this is a promotion, they want staging to win). Confirm enables.
4. Clicks `Confirm import`. Progress bar runs. Settles to "Import complete" with 22 / 24 successful, 2 failed.
5. Admin clicks `Retry failed items only`. Settles to "Import complete" with all 24 successful. Clicks `Done`. Returns to detail view of the most recently overwritten map.

**Flow D — Operator gets an Authoring GraphQL error mid-edit.**
1. Operator edits a mapping in the Context Panel. Tenant has a transient 500 from Authoring.
2. Save mutation fails. Friendly banner above the row replaces the inline form: monochrome `✕`, copy "Couldn't save this mapping. Try again, or check Cloud Portal access.", "Show technical details" toggle (verbatim GraphQL error), `Retry` button.
3. Operator clicks `Retry`. Mutation fires again; settles to success. Toast "Mapping updated".

**Flow E — Operator cancels an edit with unsaved changes.**
1. Operator opens an inline edit on a mapping row. Types in a new target. Clicks `Cancel`.
2. `@blok/alert-dialog` appears: "Discard your changes?" `Keep editing` (default focus, primary) + `Discard` (`variant="destructive-outline"`). Operator picks `Discard`. Row collapses to its original content. No mutation fires.

### 3.5 — Long-list relationship-visibility check (per agent identity § 92)

**Target check:** the Full Page right pane stacks **Redirect Maps** (left rail list) and **mappings inside the selected map** (mapping table in the right pane). Question: at 20+ items in either layer, can the operator still trace map → mappings?

**Answer for v1: yes, by construction.** The key is that the two layers are **not** parallel side-by-side lists — they are a **selection-driven single context**:
- The left rail is a virtualized list of redirect maps. Only one is selected at a time. The selected row carries the 2 px `--primary` left-edge stripe.
- The right pane shows **only the mappings that belong to that one selected map**. There is no second map's mappings on screen at the same time. The breadcrumb at the top of the page anchors the selection ("Collection / Site / Map name") so the operator can visually verify "I'm looking at MyMap's mappings" at a glance even when scrolled deep into the mapping table.
- At 30 maps in the rail and 500 mappings in the right pane, the relationship is still 1:N — one map (visible in breadcrumb + stripe) to its mappings (visible in the table). No ambiguity.

**The Component Usage Atlas trap was different:** Atlas had two parallel sections (renderings + datasources) where every item in either could relate to N items in the other. v1 of Redirect Manager has a strict parent → children hierarchy with single selection, so the trap doesn't apply.

**Where the trap *could* recreate itself:** the **import preview list** has a similar shape (24 items each with their own diff). v1 mitigates this by:
- Putting the diff *inline* (collapsible inside the row) rather than in a parallel side-pane. The row's classification badge + GUID + action picker stay co-located with their diff.
- Using `react-virtuoso` so even 100+ rows scroll smoothly and the operator can scroll to verify.
- Anchoring the row label (item name + GUID) at the top of each expanded diff so it stays in context.

**Verdict:** the design is robust at 20+ items in either layer. No hover-link or row-affinity correlations across columns. No re-design needed.

| Full Page left rail | empty-no-maps | Empty state + New map button | `<nav aria-label="No redirect maps">`. Focus on New map. | `Enter` on New map opens create form in right pane. |
| Full Page left rail | error-list | Small destructive alert + retry | `aria-live="assertive"`. | `Enter` on Retry. |

### 4.3 — Accessibility notes (cross-cutting)

- **Color is never the only signal** (NFR-A5). RedirectType: badge has the type label. Flag chips: text label inside the chip. Success: `✓` glyph + green-equivalent token. Error: `✕` glyph + `--destructive` token. Classification badges in import preview: text label in the badge, not just a colored dot.
- **Focus management on modal open.** When a `@blok/dialog` or `@blok/alert-dialog` opens, focus moves to the first interactive element (or to the cancel button for destructive dialogs — safer default). On close, focus returns to the trigger.
- **Live-region announcements** (NFR-A3):
  - Context Panel loading: "Loading redirects."
  - Context Panel data settle: "X redirect maps affect this page" or "No redirects affect this page".
  - Save success: "Mapping saved" / "Redirect map saved" / "Map deleted".
  - Save failure: "Couldn't save. Show technical details for more." (assertive)
  - Import progress: per N rows, announce "X of N items imported." Throttle to once per 10 % progress.
  - Import settle: "Import complete: X of N succeeded." or "Import completed with Y failures."
- **Keyboard shortcuts** (Full Page only — power user surface):
  - `↑` / `↓` in the rail navigates the redirect-map list.
  - `Enter` opens the focused map (sets it as the right pane subject).
  - `n` in the rail (when focused) opens "+ New map" in the right pane (escape hatch: `Esc` cancels with the unsaved-changes confirm).
  - `Esc` anywhere in a form: cancel with confirm-if-dirty.
  - `Cmd/Ctrl + S` in the detail editor: triggers Save (if dirty).
- **Focus-visible everywhere**: the Blok `--ring` token. Never `outline: none` without a replacement ring.
- **Screen-reader copy for power-user metadata.** GUID badge has `aria-label="Item GUID, click to copy"` so the screen reader doesn't read out 36 hex characters by default. The visible text is the truncated GUID; the SR label is descriptive.
- **Drag-and-drop a11y** (mapping reorder):
  - Drag handle has `aria-label="Reorder mapping. Press Space to grab, arrow keys to move, Space to drop."`
  - On grab: `aria-live` announces "Mapping `<source>` grabbed. Position 3 of 12."
  - On move: announces "Position 4 of 12."
  - On drop: announces "Mapping `<source>` moved to position 4."
  - These are the standard `@dnd-kit` announcements; the design adopts them verbatim.

### 5.2 — A11y requirements (formal contract)

- **WCAG 2.1 Level AA** across all three surfaces and both light + dark themes (NFR-A1).
- **Keyboard operability** — every interactive element reachable via Tab, every action triggerable via keyboard (NFR-A2). Drag-reorder uses `@dnd-kit/core` keyboard sensors.
- **Screen-reader labels** on every interactive element. Live-region announcements per § 4.3 above (NFR-A3).
- **Focus styles** — the Blok `--ring` token, `:focus-visible` (mouse clicks don't show the ring) (NFR-A4).
- **Color contrast** — Blok semantic-token defaults meet AA; the design does not override `--primary`, `--destructive`, or any pair globally. `--primary-foreground` pair verified in **both** light and dark before relying on `bg-primary text-primary-foreground` (Nova has shipped the pair collapsed in dark before — `blok-theming` pitfall) — fallback to `text-background` on `bg-primary` if the pair is collapsed in dark.
- **Reduced motion** — every shimmer, slide, and inline-inflate animation honors `prefers-reduced-motion: reduce`. Blok primitives already do.
- **No color-emoji codepoints** for any state icon — `✕` U+2715, `✓` U+2713, monochrome only (per `blok-theming` "Color-emoji codepoints" + agent identity § 91).

## 6. Open questions and handoff

### 6.1 — Open questions for the operator (non-blocking; would sharpen v1 if iterated)

- **Q1 — Dashboard Widget interactivity (when?).** v1 makes the tiles non-interactive in MVP; PRD § 11 says tiles "may link out to the Full Page". Should the tiles be clickable in PRD-000 (so site managers can drill in immediately), or hold for PRD-001 when the breakdown content makes the click destination richer? **Recommendation:** hold for PRD-001 — clicking a "12 redirect maps" tile in MVP just dumps the operator on the Full Page rail with that site preselected, which doesn't add much value. - **Q2 — Default left-rail width.** v1 fixes 280 px. At iframe width 1024 px that leaves ~744 px for the right pane, which is fine for the form but tight for the import preview's diff sub-table (current vs incoming columns). Should the rail be collapsible to ~80 px (icons + map name only) when the import preview is open? **Recommendation:** add this in a fast-follow if real-tenant testing shows the diff sub-table is cramped at 1024 px. Not a v1 blocker.
- **Q3 — Inline mapping edit vs full-form edit.** v1 supports both: per-row inline edit in the mapping table (US-8) AND form-level Save at the bottom of the detail. The Save-at-bottom commits *all* form-level changes (name, type, flags) plus any pending row edits in the mapping table. Should an inline-row Save bypass form-level Save (i.e., commit the row immediately), or batch under Save? **Recommendation (v1):** inline-row Save commits the row immediately as a single GraphQL mutation against `UrlMapping` (re-serialize the entire mapping list with that row's new value). Form-level Save commits all attribute changes. This keeps the round-trip granular. Verify with operators after the POC.
- **Q4 — GUID badge visibility (always vs. on hover).** v1 shows the truncated GUID badge in the detail header always. Some operators may find this noisy ("I don't care about GUIDs"). Alternative: hide behind a "Show metadata" toggle. **Recommendation (v1):** keep visible — admins promoting between environments care about GUIDs and the badge is small. Marketers don't open the Full Page detail (their surface is the Context Panel). Re-evaluate if friction-log captures complaints.

### Post-selection refinements (applied 2026-05-10)

Two stakeholder-flagged issues were resolved in the POC and folded back into this spec:

1. **Dark-mode `--primary-foreground` contrast fix.** The Nova preset (and the sibling `last-edit-trail/site/app/globals.css` it was copied from) sets `--primary-foreground: var(--color-white)` in BOTH light and dark blocks. In dark mode `--primary` becomes `primary-200` (`#d9d4ff`, light lavender) — white-on-lavender is unreadable. The POC's `theme.css` now flips `--primary-foreground` to `var(--color-blackAlpha-900)` in dark mode (both `@media (prefers-color-scheme: dark)` and `.dark` blocks), giving dark-on-lavender (WCAG AA). **Implementation must carry the same override** — do NOT silently revert if Blok bumps the Nova preset; treat the override as an intentional contract. The comment in `theme.css` names the override and the date.
2. **Light-mode availability + per-page Light/Dark/System toggle.** A small `theme-toggle.js` ships with the POC and is injected into every frame; it adds a top-right pill that cycles `System → Light → Dark → System` and persists the choice in `localStorage` so navigating between frames preserves the operator's pick. **Production behavior:** the live app inherits theme from Cloud Portal — the toggle is POC-only convenience and will not ship in the Marketplace app. The POC's `theme.css` continues to honor `prefers-color-scheme: dark` as the default, with the explicit `.light` / `.dark` classes overriding when the toggle is used.
